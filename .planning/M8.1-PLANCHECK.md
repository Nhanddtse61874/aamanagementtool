## Part 2 — What the plan MISSED

### 🔴 CRITICAL-1 — XAML binds a Core-bound type by `clr-namespace` with no `assembly=`. W2-T1 fails to build.

`src/TimesheetApp/Views/Tabs/ReportsTab.xaml:4`:

```xml
xmlns:m="clr-namespace:TimesheetApp.Models"
```

No `;assembly=`. In WPF, `clr-namespace:` **without** `assembly=` resolves **only against the assembly currently being compiled**. The prefix is genuinely used — five times:

```
:165  <HierarchicalDataTemplate DataType="{x:Type m:TeamNode}" …
:173  … {x:Type m:ProjectNode} …
:181  … {x:Type m:BacklogNode} …
:189  … {x:Type m:TaskNode} …
:197  <DataTemplate DataType="{x:Type m:DateEntry}">
```

All five records are `public sealed record` in `Models/ReadModels.cs:74-78` — i.e. **they move to Core in W2-T1, the very first move**. Reproduced in the worktree:

```
error MC3050: Cannot find the type 'm:TeamNode'.  ReportsTab.xaml(165,39)
Build FAILED.
```

**Why the plan cannot self-heal from this.** The failure is uniquely nasty *for this plan* because its guard rails all point the wrong way:

- The banner says **"This plan is PURE FILE MOVEMENT. Zero C# is edited… If a task finds itself editing a `.cs` file, it has gone wrong: stop and report."** The required fix is a **`.xaml`** file — technically not `.cs`, so the tripwire does not fire, but the executor has no sanctioned action either.
- **The word "XAML" appears nowhere in the plan** (only `App.xaml.cs` / `MainWindow.xaml.cs` as filenames). The plan's mental model is "C# files with file-scoped namespaces, therefore namespaces cannot change, therefore no `using` changes" — which is true, and irrelevant: **XAML does not resolve types through `using`. It resolves them through assembly identity**, and assembly identity is precisely what this plan changes.
- W2-T1's `done` demands `git diff --stat` show **"renames only (`R100`), zero content changes"** — which the fix violates.

So the executing subagent is cornered: the build is red, the only fix is forbidden by the acceptance criteria, and the plan's escape hatch ("stop and report") kills Wave 2.

**Verified remedy** — one line, restores green:

```xml
xmlns:m="clr-namespace:TimesheetApp.Models;assembly=TimesheetApp.Core"
```

After it: `Build succeeded. 0 Warning(s) 0 Error(s)` → `Passed: 548`.

**Root cause (STEP 4 gap):** I grepped `.planning/research/*`, `.planning/M8-FEATURE-INVENTORY.md` and the spec for `clr-namespace` / `xmlns` / `MC3050` / `MC3074` — **zero hits in all of them.** No research agent, the spec, nor the plan ever looked at a `.xaml` file. Spec §1's "the extraction is a `git mv`, not a refactor" is a **C#-only survey generalised to the whole project**, and the plan inherited the blind spot verbatim. The other 13 `clr-namespace` declarations in the codebase are safe — they point at `TimesheetApp.Views.*` / `TimesheetApp.ViewModels` (staying in WPF) or carry an explicit `assembly=`. **`ReportsTab.xaml:4` is the only one**, which is exactly why a spot-check missed it.

### 🔴 CRITICAL-2 — OT-4 becomes unsatisfiable, and it is the criterion that guards the whole plan

OT-4 states: *"No `.cs` file is modified. `git diff --stat` on every move commit shows renames only."*

Once CRITICAL-1 is fixed, `git diff --stat` for W2-T1 reads:

```
 src/{TimesheetApp => TimesheetApp.Core}/Models/Entities.cs      | 0
 src/{TimesheetApp => TimesheetApp.Core}/Models/ReadModels.cs    | 0
 src/{TimesheetApp => TimesheetApp.Core}/Models/StandupModels.cs | 0
 src/TimesheetApp/Views/Tabs/ReportsTab.xaml                     | 2 +-   ← content change
```

Sentence 1 of OT-4 stays true. Sentence 2 becomes **false**. An executor that treats OT-4 as a gate will now block on a *correct* commit. Amend to: *"No `.cs` file is modified. The only non-rename hunk in the entire phase is the one-line `xmlns` fix in `ReportsTab.xaml` (W2-T1)."* Keep the spirit — it is a good gate — but make it satisfiable, and make the single exception explicit so a *second* content change still trips it.

### 🟠 IMPORTANT-3 — Wave 2's three tasks are strictly sequential but share one wave label, while `parallelization: true`

`.planning/config.json` has `"parallelization": true`. Per the workflow's own wave algorithm — *"if `files_modified` overlap an earlier task → force later wave"*, *"same-wave plans must have **zero file overlap** (required)"* — and STEP 7's worktree-isolation rule, **same-wave plans get dispatched into parallel worktrees**.

W2-T1 → W2-T2 → W2-T3 are hard-ordered (the plan says so itself: *"Move top-down instead and Core will not compile"* — `Data/` cannot compile in Core until `Models/` is there). Yet all three carry the **same wave number**. The plan mitigates in prose (*"No parallelism is available here, by design"*), but **the structure contradicts the prose**, and STEP 7 reads structure. Fix: renumber to **Wave 2 / Wave 3 / Wave 4** (and hygiene → Wave 5), or add an explicit `depends_on` per task. Cosmetic to a human, load-bearing to a dispatcher.

### 🟠 IMPORTANT-4 — `model_profile: "quality"` silently promotes every `[sonnet]` task to `opus`

Config: `"model_profile": "quality"`. CLAUDE.md STEP 0: *"quality → all up one, clamped."* The plan tags all five tasks `[sonnet]`. Unresolved: does the executor run these as `sonnet` (as written) or `opus` (as the profile dictates)? For five `git mv` invocations, `opus` is pure waste — arguably even `sonnet` is generous. State the intent explicitly, or drop to `haiku`/`mechanical` for W2-T1/T2/T3, which are literally two shell commands each.

### 🟡 SUGGESTION-5 — the plan's own Core-purity gate is narrower than the claim it defends

W2-T3 verifies with `grep -rn "using System.Windows" src/TimesheetApp.Core/`. That is one namespace. OT-3's claim is *"no WPF dependency anywhere"*. Widen it (I ran the wide version — Core is clean, so this costs nothing and locks the property in):

```bash
grep -rnE "using (System\.Windows|System\.Drawing|Microsoft\.Win32)" --include=*.cs src/TimesheetApp.Core/
grep -rn  "clr-namespace" --include=*.xaml src/TimesheetApp/ | grep -v "assembly="   # ← would have caught CRITICAL-1
```

The second line is the regression test for the bug this check found. It belongs in the plan.

### 🟡 SUGGESTION-6 — plan says 47 Services; spec §6 says 48. The plan is right; the spec is self-contradictory.

Spec §6 diagram: *"`Services/` 48 files (50 minus ThemeService + IThemeService)"*, and §6.4 step 5 repeats *"minus `ThemeService.cs` + `IThemeService.cs` (48 files)"*. But §6.4's own **[REV2]** note says *"leave the messenger implementation in WPF"* — i.e. `CurrentTeamService` stays too, which makes it **47**. The plan silently adopts 47. **47 is correct** (verified: 47/3 builds clean, and it is what keeps `CommunityToolkit.Mvvm` out of Core). But the plan should say *"this deviates from the literal 48 in spec §6; the spec's [REV2] note supersedes its own diagram"* rather than leave a reviewer to discover the mismatch. Also worth a line in CLAUDE.md's **Project-Specific Deviations** table.

### 🟡 SUGGESTION-7 — no branch named; repo is on `main`

CLAUDE.md: *"Prefer feature branches for non-trivial work… If on the default branch, branch first."* `git branch --show-current` → **`main`**. The plan specifies five commits and no branch. Add: `feature/m8.1-core-extraction-2026-07-12`.

### ✅ Non-issue, checked and cleared — transitive package flow at runtime

Worth recording because it is the classic way a "green build" ships a broken app: W3-T1 drops `Dapper`, `Microsoft.Data.Sqlite`, `ClosedXML` from WPF and relies on them arriving **transitively** through Core. `Microsoft.Data.Sqlite` carries a **native** payload (`e_sqlite3.dll`), which is exactly the kind of thing that survives compilation and dies on the first DB call. And the test project keeps its **own** explicit `Microsoft.Data.Sqlite` reference, so **548 green would not have proven this.** Checked `src/TimesheetApp/bin/Debug/net8.0-windows/` after the drop:

```
OK  SQLitePCLRaw.batteries_v2.dll · SQLitePCLRaw.core.dll · SQLitePCLRaw.provider.e_sqlite3.dll
OK  runtimes/{win-x64,win-x86,win-arm,win-arm64}/native/e_sqlite3.dll
OK  TimesheetApp.deps.json contains "TimesheetApp.Core/1.0.0"
```

All present. Wave 3 is runtime-safe. OT-2's manual launch check is still the right gate, but it will pass.
