## Why there is anything to test at all

The 548 tests pass and the app launches with real data. So what could still be broken?

Three things, and **none of them is covered by a test**:

1. **XAML type resolution.** `ReportsTab.xaml:4` now says `;assembly=TimesheetApp.Core`. It is the only file whose *content* changed. XAML binding failures are **silent at build time** — WPF logs them and renders an empty or malformed control. Nothing fails, nothing throws.
2. **Transitive package flow at runtime.** `Dapper`, `Microsoft.Data.Sqlite` and `ClosedXML` were dropped from the WPF project and now arrive through Core. The build proves the *reference* resolves; only running the feature proves the *assets* shipped. The test project carries its own copy of these packages, so **"548 green" cannot prove this** — that is precisely why `e_sqlite3.dll` was checked by hand.
3. **A new assembly boundary through a live interface.** `ICurrentTeamService` moved to Core while its implementation `CurrentTeamService` stayed in WPF. That call now crosses assemblies.

Everything else in the app is byte-identical code that simply lives in a different `.dll`.

## The five checks, in order of risk

### 1. Reports → the drill-down tree  ⬅️ **highest risk, do this first**

The right-hand panel on the Reports screen is a `TreeView` driven by five `HierarchicalDataTemplate`s keyed on `m:TeamNode`, `m:ProjectNode`, `m:BacklogNode`, `m:TaskNode`, `m:DateEntry` — **the exact types whose assembly reference we rewrote.**

- **Do:** open **Reports**, pick a week with logged hours, expand the tree on the right.
- **Expect:** nested rows — `Team — 32.0h` → `Project — 20.0h` → `REQ-01 — 12.0h` → `Coding — 8.0h` → `Mon, 2026-07-06 — 4.0h`.
- **Failure looks like:** the tree is **empty**, or rows show a raw type name like `TimesheetApp.Models.TeamNode` instead of the formatted label. That means the templates did not match and the `;assembly=` fix is wrong.

### 2. Reports → Export to Excel

`ClosedXML` was removed from the WPF project and now flows transitively from Core. Same failure mode as the SQLite native binary: builds clean, throws on first use.

- **Do:** click **⬇ Export to Excel**, save the file, open it.
- **Expect:** a "Timesheet" sheet with 7 columns — Team, User, Backlog, Project, Task, Date, Hours.
- **Failure looks like:** `FileNotFoundException` / `Could not load file or assembly 'ClosedXML'`.

### 3. Log Work → type an hour into a cell, then click away

Launching the app proved SQLite **reads** work. It did not prove **writes** do.

- **Do:** open **Log Work**, type `4` into any weekday cell of any task, click elsewhere.
- **Expect:** the status text flicks to `Saving…` then **`✓ Saved`** (green). Row total and Week total update. Press **Prev** then **Next** — the 4 is still there.
- **Failure looks like:** `⚠ <error>` in red, or the value vanishes on reload.

### 4. Settings → Dark mode toggle

`ThemeService` is one of only three files that **stayed** in WPF while everything around it left.

- **Do:** **Settings → Appearance → Dark mode**, toggle on, then off.
- **Expect:** the whole app switches instantly, no restart. Toggle back and it returns.
- **Failure looks like:** nothing happens, or only part of the window changes colour.

### 5. Team switcher (bottom-left)

`ICurrentTeamService` now lives in Core; `CurrentTeamService` stayed in WPF. The call crosses an assembly boundary that did not exist before.

- **Do:** switch the team in the bottom-left dropdown *(skip if you only belong to one team — the switcher is hidden by design in that case)*.
- **Expect:** Log Work and Daily Report re-scope to the new team; the Task List regroups.
- **Failure looks like:** an exception, or the data does not change.
