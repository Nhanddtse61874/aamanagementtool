## Out-of-band findings (not in the claim list — flagged so nobody assumes they were checked and passed)

1. **The multi-team filter and the TEAM column on this tab have no web counterpart.** WPF
   `RequestsViewModel.cs:40-45,49,91,99-102` wires a `TeamFilterViewModel` whose checkbox set scopes
   `SearchAsync(null, teamIds)`, and `RequestsTab.xaml:50-52,80-93` renders the picker plus a TEAM chip column
   shown when >1 team is checked. The web Backlog page has neither: `backlog.component.html` has exactly four
   dropdowns and a seven-column header (`:42`) with no TEAM, and `worklog.service.ts:463-465`
   `getBacklogList()` sends `{}` — no `teamIds` — so `BacklogEndpoints.cs:571-574` always defaults to *every*
   team the user belongs to. A multi-team user loses the ability to narrow this grid to one team. This was
   never claimed as covered, so it is not a refutation — but it is a real gap on this screen and should be
   registered somewhere before `src/TimesheetApp/` is deleted.

2. **The `DEFAULT` backlog is visible and editable on WPF's Backlog tab; it is filtered out on the web.**
   `RequestsViewModel.cs:92` calls `SearchAsync(null, teamIds)` with no code exclusion — `BacklogRepository.cs:28-36`
   returns every row — so DEFAULT appears in the WPF grid with a working Edit button (only
   `TaskListViewModel.cs:18` excludes it, not this tab; the XAML comment at `RequestsTab.xaml:229-230` claiming
   DEFAULT "has no Edit affordance" is wrong). The web drops it client-side at `backlog-list.ts:53`.
   A replacement exists — default-task CRUD lives on the Settings screen
   (`pages/settings/settings.component.html:197-226`) — so this is a relocation, not a loss, but the two grids
   do not show the same rows and the search box therefore cannot find "DEFAULT" on the web.

3. **WPF's `SaveEditAsync` never updates renamed or reordered EXISTING tasks** (`RequestsViewModel.cs:240-245`
   handles only soft-deletes and inserts). The web does (`task-edit.ts:59-89`). Anyone diffing behaviour after
   M10 should expect the web to be *more* correct here, not identical.
