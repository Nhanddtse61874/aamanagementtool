# A3 — Backlog — RequestsTab (class BacklogsTab)

**Section risk:** The one real gap is the per-team visibility control (the `TeamFilter` checkbox toolbar + the TEAM pill column) — without a web equivalent, a multi-team user loses the ability to narrow the Backlog grid to a subset of their teams and loses the visual "which team owns this row" cue, even though the server itself still correctly scopes every backlog to the caller's teams by default.

Source read: `.planning/M8-FEATURE-INVENTORY.md:133-168` (Vietnamese original).

## Summary

- 24 of 27 granular features/rules: **COVERED**, most `[VERIFIED]` against both WPF and web source.
- 2 features: **MISSING** — the `TeamFilter` multi-team checkbox toolbar filter and the TEAM pill grid column. Behavior lived entirely in `src/TimesheetApp/ViewModels/RequestsViewModel.cs` (dies with WPF) and has no counterpart anywhere in `src/timesheet-web/src/app/pages/backlog/`. The underlying data (`teamId` on the list DTO) is already on the wire and the server already scopes the base query to the caller's own teams by default, so this is a UI-affordance gap, not a data-leak risk — but a multi-team user cannot narrow the grid to a subset of her teams the way she can in WPF, and cannot see which team owns a row.
- 1 feature: **PARTIAL** — Progress% is disabled (read-only) on WPF's create form; the web create form's Progress input is fully editable. Not a capability loss, but a documented WPF business rule that is not replicated.
- No feature in this section is CORE-SURVIVES — everything examined lives in `src/TimesheetApp/ViewModels/` (dies with the WPF app), not in `TimesheetApp.Core`.
