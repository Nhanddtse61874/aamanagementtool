# A6 — Reports — ReportsTab

**Section risk:** If anything here were actually lost, it would be the Reports screen's per-target/per-project/per-team drill-down math (weekly total, avg/day, days-logged, the 5-level tree) silently going wrong or disappearing with no running WPF app left to diff against — but every behavior below traces either into `TimesheetApp.Core` (survives the deletion outright) or into an already-built, already-annotated Angular `ReportsComponent` that explicitly cites the WPF source it ported from.

## Method note

`ReportAggregator.cs` (the 4-report engine), `TimeLogRepository.cs` (the report/export SQL), `ExportService.cs` (the .xlsx writer) and `TimeLogService.GetUsersMissingLogsAsync` all already live in `TimesheetApp.Core` — none of them die with `src/TimesheetApp/`. Two items the inventory describes as **current bugs** (the "DAYS LOGGED always N/N" quirk, and the banner's holiday-blind N-day window) turned out, on reading the code, to have already been fixed in Core under an "M8.2" pass, with comments describing exactly the inventory's bug and its fix. So those two rows are marked CORE-SURVIVES rather than MISSING/PARTIAL — the corrected arithmetic is what both the current WPF and the web already consume.
