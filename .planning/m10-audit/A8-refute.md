# A8 — Settings (SettingsTab, 677 lines) — Adversarial Refutation

**Scope reminder.** `src/TimesheetApp/` dies (ViewModels, Views, Services/ThemeService.cs, App.xaml.cs).
`src/TimesheetApp.Core/` survives. Every verdict below traces the behaviour to a file and, where coverage is
claimed, traces the web side route → DTO → service → component → template.

**Result: 24 claims examined, 8 refuted (all PARTIAL, none MISSING), 16 upheld.**
No claim collapsed to MISSING — the web Settings screen is a genuine reimplementation, and in several places a
strict superset (optimistic concurrency, confirm dialogs, activate-as-well-as-deactivate). The refutations
cluster in two places the auditor did not open: **ops status reporting** (failure rendered as success) and
**template-editor validation** (a silent no-op the WPF VM was explicitly changed to fix).

## What actually needs recording before `src/TimesheetApp/` is deleted

Four of the eight refutations are the same shape and are worth stating plainly, because after M10 there is
no running WPF app left to compare against:

1. **Ops actions report failure as success** (Export, Backup). Both Core services already return the
   information — `Task<string?>` for backup, a per-root `ok:`/`failed:` report for export — and both are
   carried intact across the wire in `SettingsOpsResult.Value`. Only the Angular handler discards it
   (`settings.component.ts:550-551,563-564`). This is a ~10-line client fix, not a feature rebuild.
2. **Retention run has no outcome channel** (`SettingsEndpoints.cs:1027-1033`). Architecturally justified —
   a synchronous run would 500 every other writer — but the consequence is that a failed purge is visible
   only in server logs. If that is acceptable, record it as accepted; do not let it pass as parity.
3. **Template editor validation is a silent no-op** (`settings.component.ts:341,345`). The WPF VM carries a
   comment saying this exact behaviour was a bug it fixed. Regressing it knowingly is the worst outcome here.
4. **Dark mode is admin-only on the web** (`app.routes.ts:59` + the toggle's single call site). Every other
   admin gate on this screen protects global shared state and is correct. This one protects a per-user
   cosmetic preference and is almost certainly unintended.

Nothing in section A8 is MISSING. Every behaviour has a web home; four of them have a narrower one.
