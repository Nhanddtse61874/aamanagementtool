# A8 — Settings (`SettingsTab.xaml`, 677 lines, 13 sections + 3 overlays)

**Section risk:** If WPF is deleted today, the single confirmed loss is **backup Restore** — WPF has a working "Restore this backup file over the live .db" flow (`OnRestoreBackup` + `RestoreCommand` → `IBackupService.RestoreAsync`, a Core method that survives) with zero web/API route calling it, deliberately excluded per the API's own comment ("`IBackupService.RestoreAsync` is NOT exposed in M8.3: it overwrites the live .db in place while the API holds open connections, which corrupts live readers") — so once WPF is gone there is **no in-app way to restore a backup at all**, only manual file-copy by whoever has filesystem access to the server. Six other sections (DB path, archive path, both export roots, backup-folder/auto-backup/keep-count, retention-enable/months, SharePoint Verify) are *deliberately* not web features (they are host `appsettings.json` config now, not per-request state) — that is a documented product decision in the Angular code, not an audit gap, so they are graded CORE-SURVIVES/COVERED-by-design rather than MISSING.
