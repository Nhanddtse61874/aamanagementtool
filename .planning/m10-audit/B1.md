# B1 тАФ Smart Input / Smart Fill

**Section risk:** The web Smart Fill flow only reaches a single task, only within the currently-displayed
MonтАУFri week, and only in "distribute evenly" mode тАФ deleting WPF permanently removes the "Full 8h" mode,
multi-task one-shot fill, arbitrary date-range fill, backlog-code search-to-fill, and the user-visible
journal-corruption banner, none of which have a web entry point even though most of the underlying math/logic
still lives in `TimesheetApp.Core` unreachable.

**Important correction to the inventory text (lines 403-444):** the inventory describes a **stale** bug тАФ
"two implementations, only one runs," `SmartInputPanelVm.BuildPlan` duplicating math that `SmartInputService`
never used. That has **already been fixed** in the current tree (see `SmartInputService.cs` comment "M8.2:
moved verbatim from `SmartInputPanelVm.BuildPlan`" and `ISmartInputService.cs` comment on the same line). As
of today, `SmartInputPanelVm.BuildPreviewAsync` (`src/TimesheetApp/ViewModels/SmartInputPanelVm.cs:135-136`)
calls `_smartInput.BuildPlan(...)`, i.e. Core. The duplicate-implementation bug and the holiday-preview
mismatch it describes are moot for the M10 decision тАФ the math is unified and lives in Core. What is **not**
fixed, and what actually matters for "can we delete WPF safely," is documented below: most of that Core
capability (multi-task fill, Full-8h mode, arbitrary range) has no caller on the web side at all.

## Feature-by-feature verdict summary

1. Single-task holiday-aware distribution math (`DistributeEven`/`FillFull8h`) тАФ CORE-SURVIVES
2. Multi-task `BuildPlan` distribution math (tiers 1+2, integer tenths) тАФ CORE-SURVIVES (moved to Core at M8.2, superseding the inventory's "duplicate implementation" bug report)
3. `ValidateSmartFillAsync` business rules тАФ COVERED (shared Core service, same code path both apps)
4. Apply pipeline (backup-before-write, atomic batch, journal-gone check) тАФ COVERED (shared Core service)
5. Reload/broadcast after apply тАФ COVERED (different mechanism: SignalR + merge vs. local VM event, same outcome)
6. `FindBacklogAsync` free-text backlog search feeding a multi-select checklist тАФ MISSING (web Smart Fill can only target tasks already on the visible grid)
7. Arbitrary From/To date range тАФ MISSING (web is hard-capped to the current MonтАУFri week)
8. Multi-task one-shot fill тАФ MISSING (web is single-task per Apply; manual repetition can approximate it but loses the automatic split)
9. "Full 8h" mode тАФ MISSING (no web UI control, no API path reaches the Core method that still implements it)
10. User-visible journal-corruption banner тАФ PARTIAL (detection + server-side logging survive; the banner itself does not)
