# B11 — CurrentUserService — user identification

**Section risk (one line):** The self-service "walk up on any Windows PC → auto-provisioned as a new user → auto-joined to the active team" onboarding flow lives ONLY in WPF's `MainViewModel` and is genuinely absent from the web app (confirmed by reading both sides) — after WPF deletion, any brand-new person can no longer get themselves usable access; an admin must manually create the account, set a username/password, AND manually add them to a team, or they get a silent 401 with no self-recovery path.

## Method notes

- Read `CurrentUserService.cs` / `ICurrentUserService.cs` in Core first — confirmed the core resolution algorithm already lives outside `src/TimesheetApp/` and is reused by the API via `Program.cs:131-137`.
- Traced the auto-provision and auto-join call sites into `MainViewModel.cs` (WPF-only, dies with the project), then searched the API/Angular trees for an equivalent — found none; found explicit doc comments in `ClientContextFilter.cs` and `ApiCurrentTeamService.cs` confirming the auto-join in particular was deliberately not carried over.
- Verified the "dead code" claim (`SelectUserDialog`) is still true today by grepping the whole `src/` tree.
- Verified the "no auth" liability description against `AuthSetup.cs` and `AdminEndpoints.cs`/`SettingsEndpoints.cs` — confirmed the web app replaces it with real auth, so this row is COVERED (in the sense that the gap is closed, not reproduced).
