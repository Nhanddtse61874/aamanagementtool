# B8 тАФ Backup тАФ two separate services

**Section risk:** Database RESTORE (BK-05) тАФ reachable only through the WPF Settings tab today тАФ has **zero** web/API path (the API explicitly refuses to expose it, "NOT exposed in M8.3"), so deleting `src/TimesheetApp/` removes the only way anyone can put a `.bak`/`.db` snapshot back into production without hand-editing files on the server; the once-per-day auto-backup trigger is likewise only ever called from WPF startup and nothing on the server calls its Core method, so that automatic safety net silently stops firing too.

## Method note

Both `DbBackupHelper` (XC-10) and `BackupService` (BK-01..07) are **Core classes** (`src/TimesheetApp.Core/Services/*.cs`), so their code bodies are never deleted by M10. Per the audit framing, "lives in Core -> CORE-SURVIVES" answers step (a)/(b) for the *code*. But several of these features are two-part: a Core method **plus** a *caller* (a UI button, a startup hook, an HTTP route) that is what actually makes the behavior reachable by a user. Where the only caller was WPF-only and no web caller exists, I did not stop at "the class is in Core" тАФ I verified whether any live caller remains, per Rule 5 (a false COVERED is unrecoverable once WPF is gone). Those are marked MISSING/PARTIAL, not CORE-SURVIVES, with the distinction spelled out in the Note column.

## Headline risk for the M10 decision

Restore (BK-05) is a deliberate, documented, already-shipped decision to NOT expose DB restore over HTTP (correctly, for the stated corruption-risk reason) тАФ but no replacement mechanism (admin CLI, maintenance script, ops runbook) was found anywhere in the repo. Deleting WPF today deletes the only working restore path that exists, before any replacement lands.
