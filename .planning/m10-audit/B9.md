# B9 — Retention / Prune (P12, DESTRUCTIVE, default OFF)

**Section risk:** None of the destructive prune algorithm itself is at risk (it lives entirely in `TimesheetApp.Core/Services/RetentionService.cs` + `PruneArchiver.cs`, which survive M10) — the real risk is the **automatic silent trigger**: WPF ran retention unattended on every app launch (`App.xaml.cs:91-95`, opt-in via `RetentionEnabled`), and the web app has no scheduled/background equivalent, so if an org relied on "just leave the desktop app opt-in and it prunes itself," that automation is gone and retention now only fires when an admin manually clicks a button.

## Summary

- The destructive prune **algorithm** (guard order, contiguous-prefix archiving, backup-before-delete, children-first transaction, DEFAULT/spanning-backlog preservation, marker, journal check, no-export-root gate) is 100% Core (`TimesheetApp.Core/Services/RetentionService.cs`, `PruneArchiver.cs`) and cannot be lost by deleting `src/TimesheetApp/`.
- The two **manual admin actions** (Preview, Run) are fully re-implemented end-to-end on the web stack (route + DTO + Angular UI + confirm-before-destroy gate) — COVERED.
- Two things are **not** simple Core survivors and need a human decision before M10 ships:
  1. **MISSING** — the automatic unattended startup trigger has no web equivalent; retention is now 100% opt-in-per-click with no scheduler.
  2. **PARTIAL** — the in-app toggle/months UI is gone by deliberate design (host config, security rationale documented in code); the setting still works, but only via direct `appsettings.json` edit + restart.
