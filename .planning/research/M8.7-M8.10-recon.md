# PART 1 — The cross-cutting finding, and it governs everything

**Five files are needed by every remaining screen, and owned by none.** This is the *exact* failure that cost M8.2 an entire consolidation wave:

> *"A file that N parallel agents need is a file the controller writes FIRST. `Models/Entities.cs` was not — four agents each needed it, none owned it, so the whole mechanism was unreachable."*

| File | Who needs it | Why it collides |
|---|---|---|
| 🔴🔴 **`Api/Endpoints/SettingsEndpoints.cs`** | Users · Settings · **Task List** (tags, holidays) · **Daily Report** (all 10 standup routes) · **Reports** (teams, users) · **Backlog** (users, pca) | **47 routes. ONE file. ZERO annotated.** `grep -c "WithName\|WithTags\|Produces"` = **0**. |
| 🔴 **`ng-openapi-gen.json` line 19** | every screen adds its own tag | **One line, N writers.** |
| 🔴 **`ApiTests/OpenApiContractTests.cs`** | every screen adds `[InlineData]` | Same three `[Theory]` blocks. |
| 🔴 **`app/services/worklog.service.ts`** | every screen | The stub block `:309-329` is **one contiguous region** every screen rewrites. |
| 🔴🔴 **`app/api/**` (generated)** | every screen | **A regen is a GLOBAL event** — `npm run gen:api` rewrites the whole tree. Two agents regenerating in parallel: the second clobbers the first. |

### Therefore the only safe shape

```
PHASE 1 — CONTROLLER, sequential, single owner:
    annotate EVERY route for EVERY screen (all 47 in SettingsEndpoints.cs + the rest)
  → extend OpenApiContractTests.cs ONCE
  → widen includeTags ONCE
  → regenerate the client ONCE
  → write EVERY WorklogService method ONCE
  → build the shared components nobody owns (below)

PHASE 2 — 4 agents, TRUE parallel, zero overlap:
    A → pages/task-list/     C → pages/reports/
    B → pages/daily-report/  D → pages/users/ + pages/settings/
```

### 🔴 Two shared things nobody owns, and four screens need

1. **A team-filter component.** In WPF, `TeamFilterViewModel` + `Views/Controls/TeamFilter.xaml` are **shared by four screens** (Backlog, Task List, Reports, Daily Board — `TeamFilterViewModel.cs:32-33`). In Angular **it does not exist**; `components/` holds exactly `shell/` and `sidebar/`. Its contract must be preserved exactly: `checkedTeamIds: number[]` where **an empty array means NO teams, never "all"** (R6); default = the active team only; hidden when `memberTeamIds.length <= 1`; wire format `?teamIds=1&teamIds=2` (repeated key — that is what `EffectiveTeamIds` parses).
2. **`GET /api/teams`** (`SettingsEndpoints.cs:192`). Reports needs team **names** for the filter; `GET /api/me` returns `MemberTeamIds: int[]` with **no names**. M8.6 does not claim it. **Assign it explicitly.**

### 🔴 A shared defect that hits every screen

```typescript
// core/realtime.service.ts:63 — the server sends (kind, teamId). BOTH are discarded.
hub.on(DATA_CHANGED, () => this._dataChanged.next());
```
`dataChanged` is `Observable<void>`, so **no screen can filter by what changed.** Every change anywhere refreshes everything. WPF's ViewModels filter on specific `DataKind`s; the web cannot. **Fix it in Phase 1, before five screens depend on the broken shape.**

# PART 2 — M8.7 · Task List — **the hardest screen, and the only one missing its spine**

## 🔴 The API is missing the screen's entire read model

Verified by grep against `src/TimesheetApp.Api/`:

| Missing | Evidence | Consequence |
|---|---|---|
| **Logged hours per backlog** | `ITimeLogRepository.GetLoggedHoursByBacklogAsync()` (`:54`) is **CALLED BY NO ENDPOINT** | **The single hardest blocker.** `/api/reports/monthly` returns totals by *code + task name, month-scoped* — **not** all-time hours keyed by backlog **id**. There is no route by which the web can obtain this. |
| **`ScheduleState`** (Late / Warning / Normal) | `IScheduleStateService` is registered in DI (`Program.cs:88`) and **no endpoint calls it** | The ⚠ Late / ⚠ At-risk chips have **no wire representation at all**. |
| **Bulk tag map** | `GetTagIdsForAllAsync()` unexposed | The web must N+1 over `GET /api/backlogs/{id}/tags`. |
| **Bulk task map** | `GetActiveByBacklogsAsync(ids)` unexposed | N+1 over `GET /api/tasks?backlogId=`. |
| **Gantt** | `BuildGantt` lives in the **WPF ViewModel** (`TaskListViewModel.cs:314-376`), **not Core** | Must be moved to Core or reimplemented. |
| **"Export this month"** | `ITaskListArchiveService` registered (`Program.cs:97`), **no endpoint uses it** | `/api/export/markdown` is the *timesheet* export — a different service. |

**The good news:** `ScheduleStateService.cs` and `WorkingDayCalculator.cs` are **pure, in Core, and already DI-registered in the API**. A server-side read model needs **no new plumbing, no new math**.

**Can the client compute `ScheduleState` itself?** It has `today` ✓, `startDate` ✓, `deadlineInternal` ✓, `estimate = official ?? rough` ✓, `isDone` ✓, `holidays` ✓ — and **`loggedHours` ✗, obtainable by no route.** So client-side is blocked on exactly one field; server-side is nearly free. **→ Build a `GET /api/tasklist` read model server-side.**

## The screen, verified

- **Cards, grouped into section bands.** Grouping is **adaptive**: `showTeam = (>1 team checked)` → group by team; else group by **project**, ordered by the index in `BacklogProjects.All` (`ARCS, PlusArcs, ARMS, Other`), unknown → last.
- **Chips:** at most **one** system chip (`⚠ Late` red, or `⚠ At risk` amber — `if/else if`), then custom tags ordered by `Tag.Id`.
- **`ScheduleState` rules** (`ScheduleStateService.cs:9-45`, pure, never throws, all division cross-multiplied away): done → Normal · no internal deadline → Normal · **past deadline → Late (takes precedence)** · no start date → Normal · no estimate → Normal · **working days to deadline > 2 → Normal** · then `behind = loggedHours × total < elapsed × estimate` → **Warning**.
- **`isDone` = `tasks.Count > 0 && tasks.All(Status == "Done")`** — **zero tasks is NOT done.**
- **Working-day math is inclusive of both ends**; `from > to` → 0. Holidays loaded once per reload.
- **Inline progress** is on the **backlog** (`Backlog.ProgressPercent`), NOT the task. Escape reverts without persisting; a non-integer never commits.
- **🔴 `CommitBacklogEditAsync` writes ALL FOUR fields** (`Type`, `AssigneeUserId`, `PcaContactId`, `ProgressPercent`) whenever **any one** changes (`TaskListViewModel.cs:419-433`).
- **🔴 A deadline change REQUIRES a reason note** (`DeadlineNoteDialog`); Cancel reverts the picker. Start/End changes do **not**.
- **Reads use `GetAllAsync` deliberately** for users and PCA contacts — *"so a deactivated PCA/PCT still resolves on historical rows"* (`:188`). **But `/api/users/all` and `/api/pca-contacts/all` are ADMIN-ONLY.** A non-admin web user cannot resolve a deactivated assignee's name. *(M8.6 solves this with `GET /api/users/names` — reuse it.)*
- **Every WPF write here is UNCHECKED.** The API exposes only `*CheckedAsync` — every web write needs an `expectedVersion`.
- **No context menu anywhere** — `grep -c ContextMenu TaskListTab.xaml` = 0.

## The Angular shell — fake, and *conceptually* wrong

All three data calls are stubs: `getTaskCards()` → `of([])`, `getTags()` → `of([])`, **`saveProgress()` → `of(void 0)` — every progress edit is silently discarded.**

🔴 **`ProgressMap` is keyed per-task** (`worklog.models.ts:50`), but real progress is **per-backlog**. The component's `overall(ci)` / `doneCount(ci)` (`:44-52`) **average per-task percentages — pure invention with no WPF counterpart.** `TaskCard` has **no id**, so a backlog cannot even be addressed for a write. This is a rewrite, not an extension.

**Absent entirely from the Angular page:** month/year/team filters · Grid⇄Gantt · the Gantt · chart collapse · Export · section bands · expand/collapse · PCT · PCA · Type · Status · deadlines · start/end · logged/estimate · schedule chips · tag editing · **and `Continue`**.

## 🔴 M8.7 CANNOT run in parallel with M8.6

M8.6 is in flight and already owns **eight** files M8.7 needs: `BacklogEndpoints.cs`, `SettingsEndpoints.cs`, `Dtos.cs`, `OpenApiContractTests.cs`, `ng-openapi-gen.json`, `worklog.service.ts`, `BacklogEndpointsTests.cs`, and the generated `api/**`. M8.7 needs **8 more routes annotated in `BacklogEndpoints.cs`** alone (continue, backlog-tags ×2, tasks GET, task status/extended/tags). **Sequence it after M8.6 merges.**

# PART 3 — M8.9 · Reports — **the easiest screen. Zero routes to invent.**

## There are no charts

`[VERIFIED]` by reading all 210 lines of `ReportsTab.xaml`: four stat cards, **two DataGrids**, **one 5-level TreeView**, an amber banner. **No `Canvas`, no `Path`/`Polyline`, no chart control, no charting package.** *(The Gantt is on Task List, not here.)* **No charting library is required.**

## The API is 100% complete and 0% annotated

All five routes exist and work — `GET /api/reports/weekly` · `/monthly` · `/missing-logs` · `GET /api/export/excel` · `/markdown` (all in `TimesheetEndpoints.cs:211-287`). **Every handler ends bare at `});`** — not one has `.WithName` / `.WithTags` / `.Produces<T>()`. **Unlike M8.6, M8.9 invents ZERO routes. The annotation IS the whole job.**

**`GET /api/reports/metrics` does not exist and does not need to.** The `// TODO` at `worklog.service.ts:315` is **wrong** — delete it. All four stat cards are client-side arithmetic over the weekly response (which already bundles `DayTotals` + `DetailRows` + `DaysLoggedStat` into **one round-trip, deliberately**, so the client cannot see three different snapshots).

## 🔴 The team-scoping trap is DEFUSED — and proven

`EffectiveTeamIds` (`TimesheetEndpoints.cs:356-368`) reads `HttpContext.Request.Query` **directly**, never a bound `int[]?`, *precisely because* minimal-API array binding cannot distinguish "absent" from "empty". It returns `ctx.MemberTeamIds` when absent, `client ∩ MemberTeamIds` when present, and **never `null`** (null means *every team* to the repository). All four row-serving routes go through it; both exports pass it as the **named** `TeamIds:` argument. **Two existing passing tests prove it** (`TimesheetEndpointsTests.cs:510, 573`). **No leak.**

**Two real behavioural facts a design must state (neither is a leak):**
- **`/api/reports/missing-logs` is scoped to `ActiveTeamId`, not `MemberTeamIds`**, and takes no `teamIds`. So the banner and the ⚠ NOT LOGGED card **ignore the team filter** — in WPF *and* web. Deliberate. `N` is read server-side from `chua_log_n_days` and is **never client-supplied** (correct — otherwise an attacker could demand an arbitrarily large scan).
- **`[FromQuery] int? userId` is unvalidated.** Rows stay team-clamped, so no cross-team leak — but you can read **any user's hours within your own teams**. **This matches WPF exactly** (its target dropdown is built from *all* active users, company-wide). Note `/api/timesheet/week` made the **opposite** call and refuses `userId` entirely. **Someone should decide whether that asymmetry is intentional.**

## Other findings

- 🔴 **`GET /api/export/excel` returns `Results.File` (binary); `/markdown` returns `Results.Text`.** Neither is JSON. `ng-openapi-gen` will not type these usefully — **expect a hand-written `HttpClient.get(url, {responseType:'blob'})`**, and decide whether they join `includeTags` at all. `[ASSUMED]` — not measured.
- The Angular **tree widget is real and correct** (flatten-with-depth, expand/collapse-all, `reports.component.ts:52-77`). Only its data is fake. **Keep it.** But `TreeNode` is **single-root and homogeneous**; the API returns `TeamNode[]` — **an array of roots, five heterogeneous levels**, with **no node ids** (they must be synthesised).
- The Angular project filter is **missing `PlusArcs`** (`reports.component.ts:32` vs `Entities.cs:63-64`).
- **The tree header label is stale in BOTH apps** — it says *"Project → Backlog → Task → Date"*; the tree is actually **Team →** Project → Backlog → Task → Date (P10/TM-08 added the Team root and nobody updated the label).
- **Zero Core changes needed.** `ReportAggregator` / `ReadModels` / `TimeLogRepository` / `ExportService` are complete and already team-aware.

# PART 4 — M8.10 · Users + Settings

## 🔴 Six WPF Settings sections CANNOT exist on the web

`SettingsEndpoints.cs:45-47`, verbatim:
> *"Never call any `IAppConfig.Set*` from an endpoint. It is a process-wide singleton with ten setters; on a server every one of them is **cross-user state** — one user toggling dark mode flips it for everyone, and `SetDbPath` **repoints the whole server's database**."*

That kills, on the web: **DB path** (SET-01) · **archive path** (SET-05) · **both export roots** (EX-01) · **backup folder/auto/keep-count** (BK-01) · **retention enable/months** (RT-01) · **dark mode** (P19, must become client-side `localStorage` — the Angular mock already does this correctly).

Only the **actions** survive, as the four `/api/ops/*` routes. **None of their settings can be configured through the API.** This is a product decision the user must see.

## 🔴 Other hard gaps

| Gap | Evidence | Impact |
|---|---|---|
| **No admin promote/demote route** | No `PUT /api/users/{id}/admin`; **no `SetIsAdminAsync` anywhere in Core**. `is_admin` is written by **exactly one statement in the entire codebase**: the v10 migration (`DatabaseInitializer.cs:337`). | `UserDto.IsAdmin` is **read-only**. The Users screen can show the badge and cannot change it. **Needs a Core change.** |
| **`internal sealed record` response types** | `SettingsUserStandup`, `SettingsStandupEntryView`, `SettingsOpsResult` (`:837-843`) | **Must become `public` before `.Produces<T>()` will compile.** |
| **`POST /api/teams` skips the TM-04 bootstrap** | WPF calls `EnsureDefaultBacklogIdAsync(newId)` + `SyncAsync()` (`SettingsViewModel.cs:566-569`); the route (`:199-211`) calls **neither** | **A team created via the web gets no `DEFAULT` backlog and no default tasks.** |
| **No `GET/PUT /api/settings/{key}`** | **No route touches `ISettingsRepository`** | SET-02's "not-logged warning window" **cannot be read or written from the web at all** — and Reports reads the same key. |
| **No backup list, no restore, no SharePoint verify** | `ListBackups()` / `RestoreAsync` / `ISharePointDestinationValidator.Verify()` all unexposed | `RestoreAsync` is withheld **deliberately** — *"it overwrites the live .db in place while the API holds open connections, which corrupts live readers."* |
| **No standalone default-tasks sync** | SET-04's button is `_sync.SyncAsync()`; the API only calls it as a *side-effect* of two routes | The "Sync default tasks" button has no API equivalent. |
| **Ops leaks a Core type** | `/api/ops/retention/preview` returns `RetentionPreview`, a `Core/Services` record, not a DTO | `.Produces<RetentionPreview>()` would pull a Core type into the wire contract. **Needs a decision.** |

## 🔴 Admin gating — two mechanisms, and they disagree

- **The cookie claim** (`AdminPolicy`) is written **once, at login**, into a **30-day sliding** cookie. **33 admin-gated routes use only this.**
- **`ctx.IsAdmin`** is read **fresh from the DB on every request**.
- **⇒ A demoted admin keeps admin for up to 30 days** (indefinitely, if they keep using the app) on 29 of 33 routes. Only the **four `/api/ops/*` routes** additionally check `ctx.IsAdmin` and reject immediately.
- The literal `"1"` is load-bearing: `RequireClaim` uses `StringComparer.Ordinal`, and a bool would render `"True"` — **no admin would ever pass.**

## 🔴 No `adminGuard` exists in Angular

`app.routes.ts:45-54` puts `users` and `settings` behind **`authGuard` only**. `AuthService.currentUser().isAdmin` **exists and is read by nothing** — grepped the whole `src/`. **Today a non-admin can navigate to /users and /settings; once wired, every admin-only call 403s and the screens look broken rather than hidden.**

## The shells

**Users (~100% fake):** `addUser()` is **`this.toast.show('User added')` and nothing else** — it does not even read the name. The mock `User` is **keyed by NAME, not id** (`{ name, active }`) — every real write is `PUT /api/users/{id}` with an `expectedVersion`. A rewrite.

**Settings (~95% fake):** `notify()` is wired to **every** action button; every Edit/Delete/Rename/Members/Deactivate/Preview/Run button has **no `(click)` handler at all**. The calendar is **hard-coded to July 2026**. All inputs are `value="3"` literals, unbound. **Real:** the dark-mode + accent picker (client-side `localStorage` — correct), and the holiday-calendar toggle (wired, but the service is stubbed).

**A user created by `POST /api/users` has `username = null` and no password hash → they CANNOT LOG IN** until an admin calls `PUT /api/users/{id}/username` *and* `POST /api/auth/users/{id}/set-password`. The Users screen must do all three, or it creates ghosts.
