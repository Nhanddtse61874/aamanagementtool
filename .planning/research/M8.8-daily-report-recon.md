# M8.8 — Daily Report (Standup) · recon

**Date:** 2026-07-13 · read-only reconnaissance, nothing modified, API never run.
Every claim `[VERIFIED]` against a file that was read, with `file:line`, unless marked `[ASSUMED]`.

---

## Headline

1. **The DR-06 edit-lock IS enforced server-side.** `StandupService.CanEditDay` lives in **Core**, not the ViewModel, and gates all five entry-write paths. The API re-checks it. **A web client cannot bypass it.** *(The controller's brief assumed the opposite. It was wrong, in the safe direction.)*
2. **All 10 standup routes live in `SettingsEndpoints.cs`** — not a `StandupEndpoints.cs`. That file maps **47 routes** and **not one** is annotated. It is the single highest-collision file in the repo.
3. **DR-09 (weekly markdown archive) has NO API surface at all.** No route; and `Program.cs` never calls `BackfillMissingWeeksAsync`. WPF-only. Same class of hole as M8.6's missing audit route.
4. **`StandupEntries` genuinely has no `row_version`** — confirmed three ways. Entries are **unversioned last-write-wins BY DESIGN**. `StandupIssues` *is* versioned and has a real 409 path.
5. The Angular page is **100% fake** — hardcoded date, `of([])` stubs, and view models whose shapes cannot survive the real DTOs.

### 🔴 DR-06 edit-lock — where it lives `[VERIFIED]`

**In the Core SERVICE, not the VM.** `StandupService.cs:34-38`:
```csharp
public bool CanEditDay(DateOnly workDate)
{
    var today = _clock.Today;
    return workDate == today || workDate == today.AddDays(-1);
}
```

Enforced at **five** entry-write sites, all in `StandupService.cs`:

| Site | Line | When locked |
|---|---|---|
| `AddEntryAsync` | `:98` | `if (!CanEditDay(workDate)) return 0;` |
| `QuickImportDayAsync` | `:120` | `if (!CanEditDay(targetDate)) return 0;` |
| `UpdateEntryAsync` | `:159` | `if (!CanEditDay(existing.WorkDate)) return false;` |
| `DeleteEntryAsync` | `:181` | `if (!CanEditDay(existing.WorkDate)) return false;` |
| `ReorderEntryAsync` | `:195` | `if (!CanEditDay(dragged.WorkDate)) return;` |

Paired with an **owner gate** at `:158`, `:180`, `:194` (`existing.UserId != _currentUser.Current?.Id`).

**Enforced server-side.** The API calls `IStandupService`, never the repository, for writes. `SettingsEndpoints.cs:26-28`: *"Call `IStandupService`, never `IStandupRepository` — the service holds the OWNER GATE on entry writes. Go around it and a user edits a colleague's standup entry."* The API additionally **re-checks** the lock to turn silent no-ops into honest 400s (`:595` reorder, `:620` quick-import).

Covered by `SettingsEndpointsTests.cs`: `:335` `Adding_a_standup_entry_outside_the_edit_window_is_400_and_writes_nothing` · `:351` `A_user_cannot_edit_a_colleagues_standup_entry` · `:755` `Quick_import_into_a_locked_day_is_400_and_writes_nothing`.

The VM only **mirrors** the rule for display (`DailyReportViewModel.cs:72-73`) — it is not the enforcement point.

**⚠️ Issues are DELIBERATELY EXEMPT** from both the lock and the owner gate. `AddIssueAsync` (`:222`), `UpdateIssueAsync` (`:230`), `UpdateIssueCheckedAsync` (`:241`), `DeleteIssueAsync` (`:250`) contain **no `CanEditDay` and no owner check**. By design — `REQUIREMENTS.md` DR-06: *"Issues are exempt (anyone, anytime)."* API-side the only guard is the team gate. **Do not "fix" this in the web client.**

**🔴 Gap for a web client.** Lock state reaches the client **only** as the per-entry `Editable` bool on `StandupEntryView`. **On a day with zero entries there is no `Editable` flag anywhere in the response** — so the client cannot distinguish a locked empty day from an editable empty day without recomputing `today`/`yesterday` **against the client clock and client timezone**, while the server uses `IClock.Today`. **No route or field exposes `CanEditDay`.**

### 🔴 What is MISSING

1. **DR-09 archive: no route at all.** `[VERIFIED]` `grep -rn "BackfillMissingWeeksAsync|ExportWeekAsync"` across the API returns **only** the DI registration (`Program.cs:96`). No `/api/standup/archive`, no export, no download. And **`Program.cs` startup (`:160-168`) calls `DatabaseInitializer`, `TeamBootstrapService`, `AdminBootstrap` — not the backfill.** `App.xaml.cs:100` is the only caller anywhere.
2. **No lock-state endpoint or field.** `CanEditDay` is exposed nowhere on the wire except a per-entry `Editable` bool, which does not exist on an empty day.
3. **The picker's two routes are unannotated** — `GET /api/backlogs?term=` and `GET /api/tasks?backlogId=`. *(M8.6 annotates the first; the second is still open.)*
4. **No `GET /api/standup/entries/{id}`** single-entry read.

### 🔴 `StandupEntries` has NO `row_version` — verified three ways

1. **Migration v10** (`DatabaseInitializer.cs:321-328`) — the eight tables that gain `row_version` are `Backlogs`, `Tasks`, `TimeLogs`, **`StandupIssues`**, `Users`, `Teams`, `Tags`, `PcaContacts`. **`StandupEntries` is not among them.**
2. **Repository** (`StandupRepository.cs:21-22`) — `EntryCols` has no `row_version`. `IssueCols` (`:24`) does.
3. **DTO** (`Dtos.cs:101-104`) — `StandupEntryDto` has no `RowVersion`, under the header *"DO NOT INVENT A VERSION FOR: … `StandupEntry` — deliberately unversioned: owner-gated, last-write-wins BY DESIGN."*

**Entries are neither checked nor bump-only — genuinely unversioned.** `StandupRepository.cs:16-18`: *"any write to `StandupEntries` that tries to bump `row_version` is a SQL error ('no such column'), not a no-op."* The only protection is the **owner gate + edit-lock**, and the design argument is that they make a cross-user race unreachable.

**Issues are fully protected.** `UpdateIssueCheckedAsync` (`:173-198`) → `WHERE id = @Id AND (@expected IS NULL OR row_version = @expected) RETURNING row_version` → `ConcurrencyConflictException` → **409**. `expectedVersion` is an **explicit parameter, never read off the record** (`StandupModels.cs:18-19`) — a record rebuilt from edited fields carries default `0` and would be rejected outright. `DeleteIssueAsync` has **no** checked sibling.

**🔴 What it means for the web.** Send `expectedVersion` on issue PUTs and handle 409 (`ConflictDialogComponent` already exists). For **entries**, send **no version** and expect **no 409** — but note: **the same user in two browser tabs CAN race their own entry.** The owner gate does not stop that, and WPF was single-process. This is a **genuinely new exposure the web creates** — and it only materialises if the web screen adds an edit-entry affordance **the desktop never had**.

### 🟡 A shared defect that hits EVERY screen `[VERIFIED]`

`realtime.service.ts:63`:
```typescript
hub.on(DATA_CHANGED, () => this._dataChanged.next());
```
The server sends `SendAsync("DataChanged", kind, teamId)` — **both arguments are discarded.** `dataChanged` is `Observable<void>`, so the web client **cannot filter by `DataKind`**. Any change anywhere fires every subscriber. The WPF VM filters on three specific kinds (`DailyReportViewModel.cs:47`); the web cannot. **Pre-existing, shared, and it gets more expensive to fix with every screen that depends on it.**
