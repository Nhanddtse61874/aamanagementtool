# M8.1 — Core Extraction — PLAN (wave-grouped)

**Goal:** `TimesheetApp.Core` (`net8.0`, no `-windows`) exists and holds the business layer — Models, Config, Data, and 47 of 50 Services. WPF references it and still runs. The 548 tests still pass, unchanged, and now exercise the exact assembly the future API will consume.
**Mode B.** Spec: `docs/superpowers/specs/2026-07-12-m8-backend-foundation-design.md` §6. **Stack skill: `dotnet`.**
**Tech Stack:** .NET 8 · SDK-style csproj · Dapper · Microsoft.Data.Sqlite · ClosedXML · xUnit (548 tests, ~5 s)

> **This plan is PURE FILE MOVEMENT. Zero C# is edited.** Every Core-bound file already has a file-scoped namespace, so moving it between assemblies cannot change its namespace — not one `using` anywhere in the solution changes. If a task finds itself editing a `.cs` file, it has gone wrong: stop and report.
>
> **No parallelism is available here, by design.** Each wave depends on the previous one *compiling* — `Data/` cannot move to Core before `Models/`, because it references it. Dispatching these concurrently would only produce agents fighting over the same tree. The speed comes from elsewhere: **the full suite runs in ~5 seconds**, so a gate after every wave is nearly free and there is no excuse to skip one.

## must_haves (goal-backward)

**Observable Truths**
- **OT-1** `dotnet test src/TimesheetApp.sln` reports **548 passed, 0 failed** — the same 548 as before, none deleted, none skipped.
- **OT-2** The WPF app still launches and renders real data (window title `Worklog`, sidebar, Log Work grid populated).
- **OT-3** `TimesheetApp.Core.csproj` targets **`net8.0`** — *not* `net8.0-windows`. If it needs `-windows` to compile, a WPF dependency was missed and the extraction has failed its whole purpose.
- **OT-4** No `.cs` file is modified. `git diff --stat` on every move commit shows renames only.
- **OT-5** WPF no longer carries package references it does not use.

**Required Artifacts**
- `src/TimesheetApp.Core/TimesheetApp.Core.csproj` (new)
- `src/TimesheetApp.Core/{Models,Config,Data,Services}/**` (81 files moved: 3 + 2 + 29 + 47)
- `src/TimesheetApp/TimesheetApp.csproj` (ProjectReference → Core; dead packages dropped)
- `src/TimesheetApp.Tests/TimesheetApp.Tests.csproj` (ProjectReference → Core added; **stays `net8.0-windows`**, keeps its WPF reference)
- `src/TimesheetApp.sln` (Core added)

**Required Wiring**
- Core → `InternalsVisibleTo("TimesheetApp")` **and** `InternalsVisibleTo("TimesheetApp.Tests")`.
- WPF → ProjectReference → Core. Tests → ProjectReference → Core **and** WPF.

**Key Links**
`InternalsVisibleTo("TimesheetApp")` on Core → lets WPF's `ReportsViewModel` / `TimesheetViewModel` keep calling `internal static DateHelpers` / `FormatHelpers` after they move. **Without it the Services move is a hard `CS0122` on 2 ViewModels.** This is why the wiring wave comes *before* any file moves.

## Three files stay behind in WPF (verified, not assumed)

Of the 50 files in `Services/`, exactly **three** do not move:

| File | Why it stays |
|---|---|
| `ThemeService.cs` | `using System.Windows;` · `Application.Current` · `ResourceDictionary`. **The only real WPF dependency in the entire business layer.** |
| `IThemeService.cs` | Clean (no `using` at all), but nothing outside `App` and `SettingsViewModel` consumes it. Keep it with its implementation; theming becomes CSS on the web. |
| `CurrentTeamService.cs` | `using CommunityToolkit.Mvvm.Messaging;` — its constructor takes `IMessenger`. **This is the only reason Core would otherwise need `CommunityToolkit.Mvvm`.** Leaving it behind keeps that package out of Core entirely. |

`ICurrentTeamService.cs` **does** move — `TimeLogService` and `StandupService` depend on the *interface*, and the API will supply its own scoped implementation in M8.2 (spec §6.2). `DataChangedMessage.cs` also moves: despite the word "CommunityToolkit" appearing in its doc comment, it has **no `using` at all** — it is a bare enum plus a record.

## Wave 1 — Scaffold Core and wire it up. Nothing moves yet.  `[sonnet]`

- **W1-T1** `[sonnet]` **Create the Core project, add it to the solution, wire all three references and both `InternalsVisibleTo`.**

  · **read_first:** `src/TimesheetApp/TimesheetApp.csproj` (packages + the existing `InternalsVisibleTo`); `src/TimesheetApp.Tests/TimesheetApp.Tests.csproj` (note `net8.0-windows` + `UseWPF=false`); `src/TimesheetApp.sln`.

  · **action:**

  1. Record the baseline. Run `dotnet test src/TimesheetApp.sln` and confirm it reports **548 passed**. If it does not, **stop and report** — everything downstream is gated on this number and a wrong baseline makes the gate meaningless.

  2. Create `src/TimesheetApp.Core/TimesheetApp.Core.csproj` with exactly this content:

  ```xml
  <Project Sdk="Microsoft.NET.Sdk">

    <PropertyGroup>
      <TargetFramework>net8.0</TargetFramework>
      <Nullable>enable</Nullable>
      <ImplicitUsings>enable</ImplicitUsings>
      <RootNamespace>TimesheetApp</RootNamespace>
    </PropertyGroup>

    <ItemGroup>
      <!-- DateHelpers/FormatHelpers are `internal static` and are called by
           ReportsViewModel + TimesheetViewModel, which stay in WPF. Without the
           first line, moving Services/ is a hard CS0122. -->
      <InternalsVisibleTo Include="TimesheetApp" />
      <InternalsVisibleTo Include="TimesheetApp.Tests" />
    </ItemGroup>

    <ItemGroup>
      <PackageReference Include="Dapper" Version="2.1.79" />
      <PackageReference Include="Microsoft.Data.Sqlite" Version="8.0.10" />
      <PackageReference Include="ClosedXML" Version="0.105.0" />
    </ItemGroup>

  </Project>
  ```

  `RootNamespace` is `TimesheetApp` (not `TimesheetApp.Core`) so that any file added later defaults to the same namespace family the existing code uses. Existing files are unaffected either way — they all declare their namespace explicitly.

  3. Add Core to the solution: `dotnet sln src/TimesheetApp.sln add src/TimesheetApp.Core/TimesheetApp.Core.csproj`

  4. In `src/TimesheetApp/TimesheetApp.csproj`, add a ProjectReference to Core. Add it as a **new `<ItemGroup>`** placed after the existing `InternalsVisibleTo` group:

  ```xml
    <ItemGroup>
      <ProjectReference Include="..\TimesheetApp.Core\TimesheetApp.Core.csproj" />
    </ItemGroup>
  ```

  Do **not** touch its `<PackageReference>` list in this wave — the code that needs those packages has not moved yet. Dropping them now would break the build.

  5. In `src/TimesheetApp.Tests/TimesheetApp.Tests.csproj`, add Core alongside the existing WPF reference, so the group reads:

  ```xml
    <ItemGroup>
      <ProjectReference Include="..\TimesheetApp\TimesheetApp.csproj" />
      <ProjectReference Include="..\TimesheetApp.Core\TimesheetApp.Core.csproj" />
    </ItemGroup>
  ```

  Leave `<TargetFramework>net8.0-windows</TargetFramework>` **exactly as it is.** 23 of the 61 test files need WPF — 13 ViewModel tests, 8 STA render tests (`WpfStaCollection.cs` exists because WPF permits one `Application` per AppDomain), and `DependencyInjectionTests`, which calls `App.ConfigureServices`. Retargeting this project to `net8.0` fails ~200 of the 511 test attributes.

  · **AVOID:** removing any `PackageReference` from WPF (wave 3's job). Retargeting the test project. Adding `CommunityToolkit.Mvvm` or `Microsoft.Extensions.DependencyInjection` to Core — verified, Core needs neither.

  · **verify (auto, <60s):**
  ```
  dotnet build src/TimesheetApp.sln
  dotnet test  src/TimesheetApp.sln
  ```
  Expected: build clean; **548 passed, 0 failed**. Core is empty, so nothing can have changed — this run proves the new references and `InternalsVisibleTo` broke nothing.

  · **done:** `TimesheetApp.Core.csproj` exists targeting `net8.0`; it is in the `.sln`; WPF and Tests both reference it; **548 passed**. Commit: `refactor(M8.1): add empty TimesheetApp.Core and wire references`

## Wave 2 — Move the layers, bottom-up. Pure `git mv`.  `[sonnet]`

> Order is not cosmetic. Dependencies flow **upward** (`Services` → `Data` → `Models`), so each layer must land in Core before anything that references it does. Move top-down instead and Core will not compile.
>
> SDK-style projects glob `**/*.cs` automatically, so **a `git mv` alone moves a file between assemblies.** No csproj edit accompanies any move in this wave.

- **W2-T1** `[sonnet]` **Move `Models/` (3 files) and `Config/` (2 files).**

  · **read_first:** `src/TimesheetApp/Models/` and `src/TimesheetApp/Config/` — list the files, confirm the counts are 3 and 2.

  · **action:**
  ```bash
  cd "E:/Learning/AAM 2nd/aamanagementtool"
  git mv src/TimesheetApp/Models  src/TimesheetApp.Core/Models
  git mv src/TimesheetApp/Config  src/TimesheetApp.Core/Config
  ```
  `Config/` contains `IAppConfig.cs` **and `JsonAppConfig.cs`** — the implementation moves too. It is `System.IO` + `System.Text.Json` only (fully portable), and it is `new`-ed at 25 call sites across 9 test files, 8 of which are otherwise pure-Core. Leaving it in WPF would make Core's own tests transitively depend on the WPF assembly for no benefit. The API simply does not register it.

  · **AVOID:** editing any `.cs` file. Splitting `Config/` — both files move.

  · **verify (auto, <60s):**
  ```
  dotnet build src/TimesheetApp.sln
  dotnet test  src/TimesheetApp.sln
  git diff --stat HEAD
  ```
  Expected: build clean; **548 passed**; the diff shows **renames only** (`R100`), zero content changes.

  · **done:** 548 passed; `src/TimesheetApp/Models` and `src/TimesheetApp/Config` no longer exist. Commit: `refactor(M8.1): move Models + Config to Core (git mv, no code changes)`

- **W2-T2** `[sonnet]` **Move `Data/` (29 files).**

  · **read_first:** `src/TimesheetApp/Data/` — confirm 29 `.cs` files (14 repositories + their interfaces + `IConnectionFactory`, `SqliteConnectionFactory`, `DatabaseInitializer`, `IDatabaseInitializer`, `SqliteMaintenance`).

  · **action:**
  ```bash
  git mv src/TimesheetApp/Data src/TimesheetApp.Core/Data
  ```

  · **AVOID:** touching `SqliteConnectionFactory.cs`. Its `Pooling=false` / `journal_mode=DELETE` settings are **correct for WPF** and are the default profile; the API's WAL profile arrives in M8.2 as an optional constructor parameter. Changing it here breaks `SqliteConnectionFactoryTests`, which hard-asserts `journal_mode=delete`, no `-wal`/`-shm`, and pooling-off.

  · **verify (auto, <60s):**
  ```
  dotnet build src/TimesheetApp.sln
  dotnet test  src/TimesheetApp.sln
  git diff --stat HEAD
  ```
  Expected: build clean; **548 passed**; renames only.

  · **done:** 548 passed; `src/TimesheetApp/Data` no longer exists. Commit: `refactor(M8.1): move Data to Core (git mv, no code changes)`

- **W2-T3** `[sonnet]` **Move 47 of the 50 files in `Services/`.**

  · **read_first:** `src/TimesheetApp/Services/` — confirm 50 `.cs` files.

  · **action:** Move the whole directory, then move the three WPF-bound files back:

  ```bash
  git mv src/TimesheetApp/Services src/TimesheetApp.Core/Services

  mkdir -p src/TimesheetApp/Services
  git mv src/TimesheetApp.Core/Services/ThemeService.cs        src/TimesheetApp/Services/ThemeService.cs
  git mv src/TimesheetApp.Core/Services/IThemeService.cs       src/TimesheetApp/Services/IThemeService.cs
  git mv src/TimesheetApp.Core/Services/CurrentTeamService.cs  src/TimesheetApp/Services/CurrentTeamService.cs
  ```

  Those three, and only those three. `ICurrentTeamService.cs` and `DataChangedMessage.cs` **stay in Core** — `TimeLogService` and `StandupService` depend on the interface, and `DataChangedMessage` is a bare enum + record with no `using` at all.

  · **AVOID:** moving `ICurrentTeamService.cs` or `DataChangedMessage.cs` back to WPF — that would drag Core's own services with them. Editing any `.cs` file.

  · **verify (auto, <60s):**
  ```
  dotnet build src/TimesheetApp.sln
  dotnet test  src/TimesheetApp.sln
  git diff --stat HEAD
  ```
  Expected: build clean; **548 passed**; renames only.
  Then assert the extraction actually achieved its purpose:
  ```bash
  # Core must contain no WPF reference whatsoever.
  grep -rn "using System.Windows" --include=*.cs src/TimesheetApp.Core/ ; echo "exit=$?"
  ```
  Expected: **no matches** (`exit=1`). A match means a WPF dependency was missed and Core would need `net8.0-windows` — the extraction has failed its purpose. Stop and report.

  · **done:** 548 passed; `src/TimesheetApp/Services/` holds exactly 3 files; `src/TimesheetApp.Core/Services/` holds 47; zero `using System.Windows` in Core. Commit: `refactor(M8.1): move 47 of 50 Services to Core (git mv, no code changes)`

## What this plan deliberately does NOT do

Each of these is real work in the spec, and each is excluded on purpose:

| Excluded | Why, and where it lands |
|---|---|
| `SqliteOptions` (WAL/pooling profile) | A C# change with new tests. It cannot be verified by "548 unchanged". **M8.2.** |
| Removing `ActiveTeamId` from `IAppConfig` (spec §6.2) | A C# change touching `CurrentTeamService` + `TeamBootstrapService` and requiring schema v10's `Users.active_team_id`. **M8.2.** |
| The three bug fixes (spec §13) | They **delete code that existing tests cover** (`SmartInputPanelVm.BuildPlan` — 9 attributes; `SelectUserDialog` — 1). Mixing them in makes the 548 gate impossible, and *"the cheapest way to turn a red gate green is to delete tests."* They land after M8.1 as their own step, with the test-count change stated up front. |
| Deleting `SelectUserDialog` (dead code) | Same reason. **After the gate.** |

A pure move that goes red is trivially diagnosable. A move plus three behaviour changes is not. That separation is the entire value of this plan.
