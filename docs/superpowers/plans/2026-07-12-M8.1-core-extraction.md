## The one thing rev. 1 got wrong, and why it matters more than it looks

**XAML resolves types by *assembly*, not by namespace import.**

`src/TimesheetApp/Views/Tabs/ReportsTab.xaml:4` declares:

```xml
xmlns:m="clr-namespace:TimesheetApp.Models"      <!-- no ;assembly= → "look in THIS assembly" -->
```

and genuinely binds five types from it — `m:TeamNode`, `m:ProjectNode`, `m:BacklogNode`, `m:TaskNode`, `m:DateEntry` (lines 165, 173, 181, 189, 197). Those records live in `Models/ReadModels.cs:74-78`, which move in **W2-T1 — the very first move**. The moment they land in Core:

```
error MC3050: Cannot find the type 'm:TeamNode'.
  src/TimesheetApp/Views/Tabs/ReportsTab.xaml(165,39)
```

**Of the project's 14 `clr-namespace` declarations, exactly one is affected.** The other 13 name `TimesheetApp.Views.*` or `TimesheetApp.ViewModels`, which stay in WPF. That one-in-fourteen ratio is precisely why a spot-check misses it — and why the STEP 4 research did: `clr-namespace` and `xmlns` return **zero hits** across all four research reports, the feature inventory, and rev. 1–2 of the spec. Nobody opened a `.xaml` file.

Rev. 1 then compounded the miss: its tripwire said *"if you find yourself editing a `.cs` file, you have gone wrong — stop"*, which **does not fire on a `.xaml`**, and its `done` criterion demanded *"renames only"*, which **forbids the one edit that makes the build work**. An executor would have been cornered into either stalling or improvising.

## Three of the 50 `Services/` files stay in WPF (verified, not assumed)

| File | Why it stays |
|---|---|
| `ThemeService.cs` | `using System.Windows;` · `Application.Current` · `ResourceDictionary`. **The only real WPF dependency in the whole business layer.** |
| `IThemeService.cs` | Clean (no `using` at all), but only `App` and `SettingsViewModel` consume it. Keep it with its implementation — theming becomes CSS on the web. |
| `CurrentTeamService.cs` | `using CommunityToolkit.Mvvm.Messaging;` — its constructor takes `IMessenger`. **This is the sole reason Core would otherwise need `CommunityToolkit.Mvvm`.** Leaving it behind keeps that package out of Core entirely. |

> **Disclosed divergence from spec §6.** The spec says Core takes **48** of the 50 (`ThemeService` + `IThemeService` excluded). This plan moves **47**, additionally retaining `CurrentTeamService.cs` — because moving it would drag `CommunityToolkit.Mvvm` into Core for one file. This is a strict improvement on the spec and costs nothing: the API needs its own scoped `ICurrentTeamService` implementation regardless (spec §6.2, `Users.active_team_id`).

`ICurrentTeamService.cs` **does** move — `TimeLogService` and `StandupService` depend on the *interface*. `DataChangedMessage.cs` also moves: despite naming CommunityToolkit in a doc comment, it has **no `using` at all** — a bare enum plus a record.

## Waves are one task each, on purpose

`config.json` has `parallelization: true`, and STEP 7 dispatches **same-wave** plans into **parallel worktrees**. Rev. 1 put three strictly-sequential moves under one wave label — prose warning against parallelism while the *structure* invited it. Rev. 2 gives each move its own wave so the machinery cannot do the wrong thing.

**There is no parallelism to be had here, by construction.** Dependencies flow upward — `Services` → `Data` → `Models` — so each layer must land in Core before anything referencing it does. The speed comes from the suite running in **~5 seconds** (measured), which makes a gate after every wave nearly free.

Models are tagged `[haiku]`; with `model_profile: quality` that resolves to **sonnet**. Rev. 1 tagged `[sonnet]`, which the quality profile promotes to **opus** — pure waste for five `git mv`s.

## Wave 1 — Scaffold Core and wire it up. Nothing moves.  `[haiku]`

- **W1-T1** `[haiku]` **Create the Core project, add it to the solution, wire three references and both `InternalsVisibleTo`.**

  · **read_first:** `src/TimesheetApp/TimesheetApp.csproj`; `src/TimesheetApp.Tests/TimesheetApp.Tests.csproj` (note `net8.0-windows`); `src/TimesheetApp.sln`.

  · **action:**

  1. Create `src/TimesheetApp.Core/TimesheetApp.Core.csproj` with exactly this content:

  ```xml
  <Project Sdk="Microsoft.NET.Sdk">

    <PropertyGroup>
      <TargetFramework>net8.0</TargetFramework>
      <Nullable>enable</Nullable>
      <ImplicitUsings>enable</ImplicitUsings>
      <RootNamespace>TimesheetApp</RootNamespace>
    </PropertyGroup>

    <ItemGroup>
      <!-- DateHelpers is `internal static` and is called by ReportsViewModel and
           TimesheetViewModel, which stay in WPF. Without the first line, moving
           Services/ is a hard CS0122 x3. -->
      <InternalsVisibleTo Include="TimesheetApp" />
      <InternalsVisibleTo Include="TimesheetApp.Tests" />
    </ItemGroup>

    <ItemGroup>
      <PackageReference Include="Dapper" Version="2.1.79" />
      <PackageReference Include="Microsoft.Data.Sqlite" Version="8.0.10" />
      <PackageReference Include="ClosedXML" Version="0.105.0" />
    </ItemGroup>

  </Project>
  ```

  2. `dotnet sln src/TimesheetApp.sln add src/TimesheetApp.Core/TimesheetApp.Core.csproj`

  3. In `src/TimesheetApp/TimesheetApp.csproj`, add a **new `<ItemGroup>`** after the existing `InternalsVisibleTo` group:

  ```xml
    <ItemGroup>
      <ProjectReference Include="..\TimesheetApp.Core\TimesheetApp.Core.csproj" />
    </ItemGroup>
  ```

  Do **not** touch its `<PackageReference>` list yet — the code needing those packages has not moved. Dropping them now breaks the build.

  4. In `src/TimesheetApp.Tests/TimesheetApp.Tests.csproj`, add Core alongside the existing WPF reference:

  ```xml
    <ItemGroup>
      <ProjectReference Include="..\TimesheetApp\TimesheetApp.csproj" />
      <ProjectReference Include="..\TimesheetApp.Core\TimesheetApp.Core.csproj" />
    </ItemGroup>
  ```

  Leave `<TargetFramework>net8.0-windows</TargetFramework>` **exactly as it is.** 23 of the 61 test files need WPF — 13 ViewModel tests, 8 STA render tests (`WpfStaCollection.cs` exists because WPF permits one `Application` per AppDomain), and `DependencyInjectionTests`, which calls `App.ConfigureServices`. Retargeting to `net8.0` fails ~200 of the 511 test attributes.

  · **AVOID:** removing any `PackageReference` from WPF (Wave 5's job). Retargeting the test project. Adding `CommunityToolkit.Mvvm` or `Microsoft.Extensions.DependencyInjection` to Core — verified, Core needs neither.

  · **verify (auto, <60s):**
  ```
  dotnet build src/TimesheetApp.sln
  dotnet test  src/TimesheetApp.sln
  ```
  Expected: build clean; **548 passed**. Core is empty, so nothing *can* have changed — this run proves the new references and `InternalsVisibleTo` broke nothing.

  · **done:** Core exists on `net8.0`, is in the `.sln`, and both projects reference it; **548 passed**. Commit: `refactor(M8.1): add empty TimesheetApp.Core and wire references`

## What this plan deliberately does NOT do

| Excluded | Why, and where it lands |
|---|---|
| `SqliteOptions` (WAL/pooling profile) | A C# change with new tests. It cannot be verified by "548 unchanged". **M8.2.** |
| Removing `ActiveTeamId` from `IAppConfig` (spec §6.2) | A C# change touching `CurrentTeamService` + `TeamBootstrapService`, and it needs schema v10's `Users.active_team_id`. **M8.2.** |
| The three bug fixes (spec §13) | They **delete code that existing tests cover** (`SmartInputPanelVm.BuildPlan` — 9 attributes; `SelectUserDialog` — 1). Bundling them makes the 548 gate unreachable — and *the cheapest way to turn a red gate green is to delete tests.* They land after M8.1 as their own step, with the test-count change stated up front. |

A pure move that goes red is trivially diagnosable. A move plus three behaviour changes is not. **That separation is the entire value of this plan** — and it is exactly what let the Plan Checker localise `MC3050` to a single XAML line instead of hunting through a mixed diff.
