# M8.2 — Core Hardening — PLAN (wave-grouped)

**Goal:** `TimesheetApp.Core` gains everything the web API needs and the desktop app never had: schema v10 (`row_version` + auth columns), optimistic concurrency, a WAL-capable connection profile, an online backup that does not corrupt the database, and the three bugs the survey found. **No API yet** — that is M8.3. This slice hardens the shared layer both apps sit on.
**Mode B.** Spec: `docs/superpowers/specs/2026-07-12-m8-backend-foundation-design.md` (rev. 3) §6–§9, §13. **Stack skill: `dotnet`.**
**Tech Stack:** .NET 8 · Dapper · Microsoft.Data.Sqlite 8.0.10 (bundles SQLite **3.41.2** — has `RETURNING` and `ON CONFLICT … WHERE`) · xUnit
**Branch:** `feature/m8-core-extraction-2026-07-12` (continues)
**Baseline:** 548 passed, 0 failed, 0 skipped (~5 s)

> **This is where parallelism starts.** M8.1 was a dependency chain and could not be split. Here, Waves 2 and 3 fan out to agents working on **disjoint file sets** — the wave map below states the file ownership explicitly, because `parallelization: true` means same-wave tasks are dispatched into concurrent worktrees and **any file shared between two same-wave tasks is a merge conflict waiting to happen.**

## must_haves (goal-backward)

**Observable Truths**
- **OT-1** Two concurrent updates to the same row: exactly one succeeds, the other gets a conflict. Proven by a **deterministic** test (no threads — see §Testing).
- **OT-2** A cell that one user **deleted** cannot be silently resurrected by another user holding a stale version. *(This is the fifth case rev. 1 of the spec missed.)*
- **OT-3** Smart Fill **bumps** `row_version` on every cell it writes, even though it does not check it. Otherwise a stale client still matches and overwrites Smart Fill's result.
- **OT-4** A backup taken while the database is in WAL mode restores to a correct database. `File.Copy` is gone from every live-DB path.
- **OT-5** `SqliteConnectionFactory` still defaults to the **desktop profile** (`Pooling=false`, `journal_mode=DELETE`) — `SqliteConnectionFactoryTests` hard-asserts it — and accepts a WAL profile via an **optional constructor parameter**.
- **OT-6** `ActiveTeamId` is per-**user** (`Users.active_team_id`), not per-**process**. On a server, one user switching team must not re-scope everyone else's data.
- **OT-7** Migration v10 runs on the existing database without data loss, and `CreateTables`' baseline DDL still says `windows_username` (migrations replay on a fresh DB — "tidying" it breaks fresh installs only).
- **OT-8** The WPF app still launches and works. It runs on this Core until M8.10.

**Key Links**
1. Client's `expectedVersion` → repository's `WHERE row_version = @expected` → `rowsAffected == 0` → `ConcurrencyConflictException` → (M8.3) HTTP 409.
2. `Users.active_team_id` → `ICurrentTeamService` → `TimeLogService` / `StandupService` team scoping (**11 read sites**) → R6 no-leak rule.
3. WAL enabled → `File.Copy` of a live `.db` becomes corrupting → **every** live-DB copy must become `BackupDatabase()`.

# Wave 2 — 3 agents **IN PARALLEL**. Disjoint files.  `[opus]` `[sonnet]` `[opus]`

> **File ownership is exclusive.** No file appears in two tasks. Verified against the wave-2 file list before dispatch.

- **W2-A** `[sonnet]` **`SqliteOptions` — a WAL profile, without disturbing the desktop default.**

  **Owns:** `src/TimesheetApp.Core/Data/SqliteConnectionFactory.cs` · `src/TimesheetApp.Tests/Data/SqliteConnectionFactoryTests.cs`

  · **read_first:** `SqliteConnectionFactory.cs` (all of it — 50 lines); `SqliteConnectionFactoryTests.cs` (**it hard-asserts `journal_mode=delete`, no `-wal`/`-shm`, pooling-off — all 6 tests break if you change the default**).

  · **action:** Add an **optional constructor parameter** carrying the profile. Verified against MS.DI 8.0.1: an optional param works on all three call paths — unregistered (WPF → desktop default), registered (API → WAL), and direct `new SqliteConnectionFactory(cfg)` (three test fixtures). **`IOptions<T>` breaks the direct-`new` sites. Do not use it.** Do **not** hang it off `IAppConfig` — that interface is **frozen**: four test files hand-implement it, so any new member fails compilation on all 548.

  | Setting | Desktop (**default**) | Server (explicit) |
  |---|---|---|
  | `Pooling` | `false` | `true` |
  | `journal_mode` | `DELETE` | **`WAL`** |
  | `busy_timeout` | — | **`1000`** |
  | `DefaultTimeout` | — | **`5`** |
  | `synchronous` | default | `NORMAL` |

  🚨 **`DefaultTimeout=5` is not optional.** `busy_timeout` does **not** bound how long a blocked writer waits: Microsoft.Data.Sqlite **auto-retries** to `CommandTimeout`, default **30 s**. Measured with WAL + `busy_timeout=5000`: a blocked writer failed after **33,940 ms**. `DefaultTimeout` is a valid connection-string keyword and flows into every Dapper call, so no repository changes.

  · **AVOID:** changing the default profile (6 tests hard-assert it). `IOptions<T>`. Touching `IAppConfig`.

  · **verify:** existing 6 `SqliteConnectionFactoryTests` **still pass unchanged**; new tests prove the server profile yields `journal_mode=wal` and that `-wal`/`-shm` appear.

  Commit: `feat(M8.2): SqliteOptions — optional WAL/pooling profile; desktop default unchanged`

- **W2-B** `[opus]` **BLOCKER B1 — replace every live-DB `File.Copy` with an online backup.**

  **Owns:** `src/TimesheetApp.Core/Services/BackupService.cs` · `DbBackupHelper.cs` · `PruneArchiver.cs` · their test files

  · **read_first:** `BackupService.cs` (**:9 states its own precondition verbatim**, and :47/:103/:105); `DbBackupHelper.cs:36`; `PruneArchiver.cs:151-167`.

  · **Why this is a blocker, not a cleanup.** `BackupService.cs:9` says:

  > *"File-level `File.Copy` **is safe at idle** — the app uses **short connections + journal_mode=DELETE**."*

  W2-A enables WAL and pooling — **deleting both premises**. Per [sqlite.org/wal](https://www.sqlite.org/wal.html), copying a WAL database **without its `-wal`** means *"transactions that were previously committed … might be lost, or the database file might become corrupted."* And `PruneArchiver` validates its pre-prune snapshot with:

  ```csharp
  return File.Exists(snapPath) && new FileInfo(snapPath).Length > 0 ? snapPath : null;
  ```

  Exists and non-empty → *"snapshot OK"* → `RetentionService` then **permanently DELETEs the originals**. Under WAL that snapshot can be silently corrupt. **Three months of real data destroyed, with a backup that cannot restore it.**

  · **action:**
  1. Replace all five live-DB `File.Copy` calls with **`SqliteConnection.BackupDatabase()`** (SQLite's online-backup API; 3.41.2 is bundled). It is safe against a live, WAL-mode database.
  2. **Restore must be offline.** With pooling on, `File.Copy(backup, dbPath, overwrite:true)` throws `IOException` on pooled handles or replays a stale `-wal`. Restore-while-serving cannot be done by file copy at all: close the connection pool (`SqliteConnection.ClearAllPools()`), restore, and require a restart — the desktop app already tells the user to restart.
  3. **Verify the snapshot for real.** `Exists && Length > 0` is not proof. Open the snapshot and run `PRAGMA integrity_check` — Retention deletes data on the strength of this answer.
  4. Register **no-op** `IDbBackupHelper` and `IJournalWarningSink` for the future API (spec §9). They are constructor-injected into **five** services (`TimeLogService`, `RetentionService`, `TeamBootstrapService`, `DefaultTaskSyncService`, `PruneArchiver`), so left alone on a server they would copy the **entire database on every Smart Fill request**, and the XC-09 journal check would pass forever looking for a `-journal` file WAL never creates. **A DI swap — do not change any of the five services.**

  · **AVOID:** changing the five services' constructors. Removing the desktop backup behaviour — WPF still needs it until M8.10.

  · **verify:** a new test takes a backup **while a write transaction is open in WAL mode**, restores it, and asserts the data is intact and `integrity_check` returns `ok`. Existing backup/retention tests still pass.

  Commit: `fix(M8.2): online backup instead of File.Copy — WAL made the old path a data-loss route`

# Wave 3 — Repositories. **n agents in parallel**, one per file group.  `[opus]`

> `row_version` touches **22 `UPDATE` sites**. **10 of the 16 on versioned tables carry no client version** — soft-deletes, reorders, admin edits. Rev. 1's single template applied to `TaskRepository.SetOrderAsync` (`:114`), which runs **once per row during a drag**, would **409-storm on an ordinary reorder**.

**Two templates. The rule is: always bump; check selectively.**

```sql
-- check-and-bump  (user edits carrying an expectedVersion)
UPDATE <t> SET …, row_version = row_version + 1
 WHERE id = @id AND row_version = @expected;      -- rowsAffected == 0 → conflict

-- bump-only  (reorders, soft-deletes, system writes)
UPDATE <t> SET …, row_version = row_version + 1
 WHERE id = @id;
```

A write that bumps without checking is safe. **A write that checks without bumping is the Smart Fill bug** (below).

| **W3-B** `[opus]` | `BacklogRepository.cs` + `TaskRepository.cs` + tests |
| **W3-C** `[sonnet]` | `StandupRepository.cs` + `UserRepository.cs` + tests |
| **W3-D** `[sonnet]` | `TeamRepository.cs` + `TagRepository.cs` + `PcaContactRepository.cs` + tests |

### W3-A — `TimeLogs`: the upsert has **five** cases

| Client sends | Row on server | Result |
|---|---|---|
| `expectedVersion = null` | absent | INSERT, `row_version = 1` |
| `expectedVersion = null` | **present** | **conflict** — someone filled this cell while you looked at it |
| `expectedVersion = N` | present, version `N` | UPDATE, `row_version = N + 1` |
| `expectedVersion = N` | present, version `≠ N` | **conflict** |
| **`expectedVersion = N`** | **absent (deleted)** | **conflict** ← *the case rev. 1 of the spec missed* |

The naive `INSERT … ON CONFLICT … DO UPDATE … WHERE row_version = @expected` gets the **last case wrong**: with no row there is no conflict, so the `WHERE` never runs, the `INSERT` succeeds, and the row is **resurrected at v=1, reported as success**. Alice deletes a cell; Bob, holding v=3, types a number; **Alice's delete evaporates with no error and no trace.** The `WHERE` guards the UPDATE branch — **not the INSERT branch.**

Use this (verified 5/5 cases; 12 threads racing one empty cell → exactly 1 winner, 11 conflicts, 0 exceptions):

```sql
INSERT INTO TimeLogs (user_id, task_id, work_date, hours, created_at, row_version)
SELECT @UserId, @TaskId, @WorkDate, @Hours, @CreatedAt, 1
 WHERE @Expected IS NULL
    OR EXISTS (SELECT 1 FROM TimeLogs
                WHERE user_id = @UserId AND task_id = @TaskId
                  AND work_date = @WorkDate AND row_version = @Expected)
    ON CONFLICT(user_id, task_id, work_date) DO UPDATE
   SET hours = excluded.hours, row_version = TimeLogs.row_version + 1
 WHERE TimeLogs.row_version = @Expected
RETURNING row_version;
```

`RETURNING` emits **zero rows** when the `DO UPDATE`'s `WHERE` makes it a no-op — the SQLite docs are silent on this; it was measured. So one statement both detects the conflict and returns the new version: `QueryFirstOrDefaultAsync<long?>` → **`null` means conflict**.

**Smart Fill (`UpsertBatchAsync`) BUMPS but does not CHECK.** If it bumps nothing, a client holding the old version still matches and **silently overwrites Smart Fill's result** — the very lost update this slice exists to kill. Its protection stays the server-side re-validation already inside the apply transaction.

Also add `INSERT … SELECT … WHERE … ON CONFLICT` to the spec's §15 porting table — it is a **new SQLite-only construct**, and §15 commits to staying accurate.

### W3-B — `Backlogs` needs a transaction *before* it can be version-checked

`BacklogRepository.UpdateAsync` (`:140-192`) currently does read → `UPDATE` → N× audit `INSERT` **with no transaction**. Bolt a version check onto that and a conflict still writes the audit rows — history describing a change that never happened. **Wrap it first.**

`TaskRepository.SetOrderAsync:114` → **bump-only**. It runs once per row during a drag.

### W3-C — the rename's silent failure

`UserRepository.cs:88` — `UserRaw.windows_username` binds **by column name**. Rename the column (Wave 1) but not the DTO and **Dapper silently returns `null`**: no exception, no log, **broken login**. Rename the DTO property and the interface methods (`GetByUsernameAsync` / `SetUsernameAsync`) — leaving `Windows` in the name of a method that has nothing to do with Windows is how stale vocabulary calcifies. *(This codebase already carries that scar: files named `Requests*` containing classes named `Backlogs*`, three migrations after the rename.)*

Also add `GetActiveTeamIdAsync` / `SetActiveTeamIdAsync` — Wave 4 needs them.

### All Wave-3 agents

· **verify (auto):** the **deterministic** concurrency test — no threads. The conflict window *is* the stale version number, so "two simultaneous updates" reproduces perfectly sequentially: both clients read v=1; apply A (succeeds); apply B with `expected=1` (**conflict**); assert **A's value survived**. Fast, and never flaky.

🚨 **Never use `:memory:` for a concurrency test.** Each connection gets its **own** database, so a conflict can never occur — the test **passes while asserting nothing**. The existing 548 use temp-file databases; keep that convention.

· **AVOID:** touching any file owned by another Wave-3 agent. Using the `deferred: true` transaction overload — it reproduces `SQLITE_BUSY_SNAPSHOT` (517), which hangs for the full timeout on a conflict that can never resolve. (`BeginTransaction()` already defaults to `BEGIN IMMEDIATE`, so the default is safe.)

# Wave 4 — `ActiveTeamId` becomes per-user. One agent (depends on W3-C).  `[opus]`

- **W4-T1** `[opus]` **Move `ActiveTeamId` out of `IAppConfig` and into `Users.active_team_id`.**

  **Owns:** `Core/Config/IAppConfig.cs` · `JsonAppConfig.cs` · `Core/Services/ICurrentTeamService.cs` · `Core/Services/TeamBootstrapService.cs` · `src/TimesheetApp/Services/CurrentTeamService.cs` · tests

  · **Why.** `IAppConfig.ActiveTeamId` is read and written **by Core services** — `CurrentTeamService:72,94` and `TeamBootstrapService:66,76` write it; `TimeLogService` (4 sites) and `StandupService` (7 sites) read it to scope **every** query. The original authors documented the hazard themselves at `IAppConfig.cs:24-27`:

  > *"the active team is an app-local, **per-machine/user** UI preference — never in the shared DB, else two users fight over one active team"*

  That is **correct for WPF**, where one process serves one user. In an API, one process serves everyone: user A switches team → the singleton says team X for **everybody** → user B's next request is served team X's timesheets. This violates the R6 no-leak rule the whole codebase is built around. The root defect is that `IAppConfig` mixes per-**app** state (`DbPath`, `BackupFolderPath`, `ExportRoot*`) with per-**user** state. On a desktop those coincide; on a server they do not.

  · **action:** Remove `ActiveTeamId` / `SetActiveTeamId` from `IAppConfig`. `CurrentTeamService` reads/writes `Users.active_team_id` via `IUserRepository` (W3-C added the methods). Startup order already works: `MainViewModel.InitializeAsync` resolves the **user** before the **team**, so there is no chicken-and-egg.

  ⚠️ **`IAppConfig` is frozen for *additions* — four test files hand-implement it.** *Removing* two members is safe (their fakes simply keep an unused public member), but **run the full suite to confirm**.

  · **verify:** two users on the same database hold **independent** active teams. Existing team-scope tests pass. WPF still switches teams correctly.

  Commit: `fix(M8.2): ActiveTeamId is per-user (Users.active_team_id), not per-process`
