# M8.3 — API + Auth + SignalR Implementation Plan

> **REQUIRED SUB-SKILL:** `superpowers:subagent-driven-development`.

**Goal:** Stand up `TimesheetApp.Api` so the Angular app has something real to talk to — authenticated, team-scoped, conflict-aware, pushing live changes.

**Architecture:** Minimal APIs over the existing `TimesheetApp.Core` services. Cookie auth on the existing `Users` table. SignalR replaces the in-process `WeakReferenceMessenger`. The API is the **only writer process** — that is what restores the single-writer property SQLite needs (spec §2).

**Tech Stack:** ASP.NET Core 8, `Microsoft.Extensions.Identity.Core` (the hasher class only — no EF Core, no Identity schema), SignalR, Swashbuckle (OpenAPI → M8.4 generates its client from it), `WebApplicationFactory`.

**Spec:** `docs/superpowers/specs/2026-07-12-m8-backend-foundation-design.md` §6.2, §8, §10, §11.

## rev. 2 — this plan was BLOCKED by the Plan Checker and rewritten

Rev. 1 rested on five premises about the codebase. **Four were false**, and they were false in the same direction: they assumed Core already had what the API needs. It does not.

| rev. 1 assumed | Ground truth |
|---|---|
| Auth can be built on existing Core methods | **`IUserRepository` has ZERO credential methods.** `password_hash` and `is_admin` exist *only* in the `DatabaseInitializer` DDL and a migration test. No repository method, no entity property. |
| `ICurrentTeamService` is usable from the API | **Its only implementation lives in the WPF project** (`net8.0-windows`) and its ctor takes `IMessenger` (CommunityToolkit). The API cannot reference it. Yet `TimeLogService` and `StandupService` — both Core — **require** it. |
| "A checked write never takes an optional param and never returns void" | **False for the one entity that matters most.** `UpsertCheckedAsync(TimeLog, long? expectedVersion)` is nullable *by design* (the five-case table), and `DeleteCheckedAsync(…)` returns `Task`. An agent following rev. 1's rule would have "fixed" `TimeLogRepository` and destroyed the five-case behaviour W3-A measured. |
| The 409 body can carry the current row | **`ConcurrencyConflictException` does not carry it** — and for `TimeLogs`, `Id` is `0` (the key is the natural triple), so a global handler cannot even re-read it. |
| Zero file overlap ⇒ safe to parallelize | **Route overlap is invisible to a file-overlap check.** Rev. 1 gave `tags` to two agents in different files; both would map `/api/tags` → `AmbiguousMatchException` → 500 on every tags request, discovered only at the merge gate. |

**And the one that would have failed 100% of the tests:** the WPF root registers ~35 services as `AddSingleton`. In the API, `ICurrentUserService` and `ICurrentTeamService` **must be scoped** — so everything injecting them must be too. `WebApplicationFactory` defaults to the Development environment, where **scope validation is on**: the host throws `Cannot consume scoped service … from singleton …` at build time, and *every* API test fails, including the auth tests meant to prove Wave 1 works. Turn validation off and it is worse: one singleton captures one user's team and serves it to everyone — the R6 cross-team leak, which already has a regression test (`TM-06`).

**The structural lesson, now twice paid for:** M8.2 fanned four agents out over four repositories and got three different concurrency APIs plus an `Entities.cs` nobody would touch. Rev. 1 repeated it one layer up. So: **Wave 0 and Wave 1 are single agents that build nothing but what more than one later agent needs.** A file — or a *route* — that N parallel agents touch is the controller's to fix first.

```csharp
// New record. The hash does NOT go on `User` — `User` is returned by a dozen endpoints and
// a hash must never be one field away from a DTO by accident.
public sealed record UserCredentials(int Id, string Name, string? PasswordHash, bool IsAdmin, bool IsActive);
```

## W1a — The host boots, with scope validation ON

**Owns:** `src/TimesheetApp.Api/TimesheetApp.Api.csproj` · `Program.cs` · `Infrastructure/IClientContext.cs` · `ApiClientContext.cs` · `ApiCurrentTeamService.cs` · `ClientContextFilter.cs` · `src/TimesheetApp.ApiTests/*.csproj` · `ApiFactory.cs` · `HostBootTests.cs` · `src/TimesheetApp.sln`

### The lifetime graph is the deliverable, not a detail

The WPF composition root (`App.xaml.cs:134-196`) registers ~35 services as `AddSingleton`. **Do not copy it.** On a desktop, one process serves one user, so singleton per-user state is correct. On a server it is a cross-user data leak.

Anything that holds per-user state — `ICurrentUserService` (holds `Current`), `ICurrentTeamService` (holds the active team) — is **Scoped**. Anything that injects those transitively (`TimeLogService`, `StandupService`, `BacklogContinuationService`, …) is **Scoped** too. Repositories are Scoped or Transient. Only genuinely stateless, config-only services stay Singleton.

**`ICurrentTeamService` has no server-usable implementation.** Its only impl is in the WPF project (`net8.0-windows`, ctor takes `IMessenger`) — the API cannot reference it. Spec §6.2 requires: *"make `ICurrentTeamService` **scoped per-request** in the API, resolving from the authenticated user."* Write `ApiCurrentTeamService`, backed by `IUserRepository.GetActiveTeamIdAsync` / `SetActiveTeamIdAsync` (M8.2 W3-C added them; W4 wired them).

**Nothing calls `CurrentUserService.ResolveAsync()` for you.** Without it, `.Current` stays `null` and `StandupService` throws a `NullReferenceException` on the first request. `ClientContextFilter` — an endpoint filter running on every authenticated request — resolves the identity once per request and populates:

```csharp
public interface IClientContext            // scoped; the ONE place identity, team scope and connection id live
{
    int UserId { get; }
    string UserName { get; }
    bool IsAdmin { get; }
    IReadOnlyList<int> TeamIds { get; }    // R6: every query is scoped to these
    string? ConnectionId { get; }          // SignalR caller, from the X-Connection-Id header. See Wave 3.
}
```

`ConnectionId` is here for a reason found by the Plan Checker: `IChangeNotifier.DataChangedAsync(..., exceptConnectionId)` needs the caller's SignalR connection id, **and it is not on `HttpContext`.** The JS client must send it as a header, and something must read it out. If that reader is not in Wave 1, all four Wave-2 agents invent their own and Wave 3 discovers the divergence.

### The four silent failures

Each was found by reading framework source. None throws. None logs. Each looks like something else.

1. **Cookie auth does not 401 on .NET 8 — it 302s.** The *only* trigger for a 401 is the literal request header `X-Requested-With: XMLHttpRequest`. Everything else gets `Response.Redirect("/Account/Login")` → a route that does not exist → **Angular sees 404, never 401.** Microsoft changed this default only in .NET 10.
   ```csharp
   o.Events.OnRedirectToLogin        = ctx => { ctx.Response.StatusCode = 401; return Task.CompletedTask; };
   o.Events.OnRedirectToAccessDenied = ctx => { ctx.Response.StatusCode = 403; return Task.CompletedTask; };
   ```
   **The twist that costs a day:** the SignalR JS client *does* send that header — so **the hub 401s correctly while the API 404s.** The asymmetry sends you hunting in the wrong file.

2. **Data Protection is the highest-severity gap in the design, and it is invisible until production.** The auth cookie is encrypted with a Data Protection key ring. Unconfigured, that ring lives **in memory** — so **every process restart logs everyone out.** The host is a workstation: it reboots, it updates, it recycles. That is mass logouts roughly daily, **with nothing in the logs**, silently defeating the "stay logged in" requirement that is the whole reason we chose a persistent cookie. `PersistKeysToFileSystem`, into a folder that survives restarts and is backed up with the database.

3. **`FallbackPolicy = DefaultPolicy` locks the user out of the login page.** It applies to *"requests served by other middleware after the authorization middleware, such as static files"* — so a logged-out user requesting Angular's `index.html` gets 401 and can **never reach the form that would log them in**. It also blocks `MapFallbackToFile`, health checks, and `/api/auth/login` itself. `[AllowAnonymous]` on login; `.AllowAnonymous()` on the SPA fallback and static files. (SignalR's `negotiate` being blocked is **correct** — leave it.)

4. **`RequireClaim("is_admin", "1")` compares with `StringComparer.Ordinal`.** Write the claim from a `bool` and you get `"True"` — the policy then **always fails** and your admin gets a **silent 403 with nothing logged**. Write the literal `"1"`. Test that a real admin *passes*, not merely that a non-admin fails.

### `busy_timeout` does not bound how long a request hangs

Microsoft.Data.Sqlite **auto-retries** busy/locked errors up to `CommandTimeout` — default **30 s**. Measured with the previous settings: a blocked writer failed after **33,940 ms**, not 5,000. Put **`DefaultTimeout=5`** in the connection string (a valid keyword; it flows into every Dapper call, so no repository changes) and `busy_timeout=1000` → worst-case hang ≈ 5.2 s.

### The 409 contract — and the field rev. 1 dropped

```csharp
public sealed record ConflictBody(string Table, long Id, bool Deleted, string? Detail, string Message);
```

Every field comes straight off `ConcurrencyConflictException`. **`Detail` is not optional garnish** — W3.5 added it *specifically* so that a 409 on a timesheet cell can say *which* cell: `TimeLogs`' key is the natural triple `(user_id, task_id, work_date)`, so its `Id` is **`0`** and without `Detail` the client is told only "something in TimeLogs changed". Rev. 1 dropped it one hour after W3.5 added it. Do not drop it again.

**DEVIATION from spec §8.1 — recorded, not hidden.** The spec's 409 body also carries `current` (the server's current row). **This plan omits it.** The exception does not carry the row, and a global handler cannot re-read it: it would need a `Table → repository` dispatch map, and for `TimeLogs` — `Id = 0` — `GetByIdAsync` is not even expressible. The alternatives were to put `Current` on the exception (re-opening all 8 repositories one wave after consolidating them) or to parse `Detail` strings in a dispatch map (fragile). **Instead: on 409 the client re-fetches.** One extra round-trip on a path that is by definition rare, and it is not racy — a re-fetch returns a coherent (state, rowVersion) pair, and if someone writes again in between, the next save 409s again, which is correct. Nothing is lost but a round-trip.

Known limitation, by design, not a bug to hunt later: **`TimeLogs` has no `changed_by` column**, so the API *cannot name who* changed a cell. "Someone else just changed this" can name the person for `Backlogs` (via `BacklogAudit`) but **not** for timesheet cells.

### DTOs — and the rule that is *not* universal

**Seven records carry `RowVersion`:** `User`, `Team`, `Backlog`, `TaskItem`, `TimeLog`, `Tag`, `PcaContact`, `StandupIssue`. Their DTOs **must** expose it as `rowVersion` — drop it and the client cannot send back the `expectedVersion` a checked write requires, and the mechanism M8.2 spent four waves building is dead on arrival for that entity.

**`TaskTemplate`, `DefaultTask` and `Holiday` have no `row_version` column at all.** Do **not** invent one for them. (W3-D already invented a convention when handed an over-broad instruction — it justified an edit by "this file's existing convention", and the convention did not exist.)

Also: the *column* is `username` (v10) but the *record property* is still `WindowsUsername` — renaming it was judged out of scope at 28+ consumers. `Dtos.cs` maps `WindowsUsername` → `username`.

### Admin bootstrap — no persistent secret

Rev. 1 of the spec read a bootstrap password from configuration; that puts a credential in a file that tends to get committed. Instead: on startup, if the designated admin's `password_hash IS NULL`, generate a random password, apply it via `TryBootstrapAdminPasswordAsync`, and write it **once** to the console. The operator reads it, logs in, changes it. **There is no secret to leak, because it exists only in that one log line.**

**No "first login claims the account" flow.** On a shared internal network that lets anyone claim a colleague's account before they do.

**Login must check `is_active`.** Omit it and a soft-deleted user who still has a password can log in.

`CurrentUserService` is **not modified** — it already takes its identity through a `Func<string>` seam (`CurrentUserService.cs:17`):
```csharp
new CurrentUserService(users, () => Environment.UserName)                   // WPF, unchanged
new CurrentUserService(users, () => ctx.HttpContext!.User.Identity!.Name!)  // API
```

### OpenAPI — M8.4 depends on this and rev. 1 forgot it

Add Swashbuckle and expose the OpenAPI document. **M8.4 generates its TypeScript client from it.** The vendored Angular bundle's models are UI-shaped and unusable as a wire contract: `User` has no `id`, `Backlog` has no `id`, `LogGroup.tasks` is `string[]`, and `HoursMap` is keyed by `` `${groupIndex}-${taskIndex}-${dayIndex}` `` — **array indices**. One sort or filter and hours land on the wrong task; and with no id there is nowhere to put a `rowVersion`. Hand-maintaining those types across the wire is how you rebuild that bug on purpose. Without OpenAPI here, M8.4 has nothing to generate from.

- [ ] **Verify:** the five auth tests from spec §11 — login · bad password · **inactive user** · persistent cookie · **admin policy admits a real admin AND denies a non-admin**.
- [ ] **Verify (automated, not by hand):** `ConflictContractTests` asserts the **raw 409 JSON** — `deleted` correct in *both* cases (changed vs deleted), `detail` populated for a `TimeLogs` conflict. Rev. 1 gated the single highest-value contract in this milestone on a manual eyeball; "no test proves the wire format" was a choice, not a fact.
- [ ] **Verify (automated):** the Data Protection key ring survives a restart — factory A → log in → dispose → factory B over the **same key-ring folder** → replay the cookie → **200**. This is the one bug that is invisible until production, so it is the one that most needs a test.
- [ ] **Done:** logged-out `GET /api/me` → **401**. Not 302. Not 404.

Commit: `feat(M8.3): auth, Data Protection, 409 contract, OpenAPI — every shape Wave 2 needs`

### Restated verbatim in every Wave-2 brief — because an agent sees its brief, not this plan

M8.2 proved that a decision recorded centrally but not repeated in each brief **is not a decision**: three of four agents independently re-derived a different concurrency API from a rule that was already written down.

1. **Use the repository's real signature. Read it; do not assume it.** Most checked writes are `Task<long> <Verb>CheckedAsync(entity, long expectedVersion)` and return the new `row_version`. **`TimeLogs` is deliberately different** and W2-B owns it:
   ```csharp
   Task<long> UpsertCheckedAsync(TimeLog log, long? expectedVersion);              // NULLABLE — the five-case table
   Task       DeleteCheckedAsync(int u, int t, DateOnly d, long expectedVersion);  // returns Task, not Task<long>
   ```
   `expectedVersion` is nullable there because an upsert has **five** outcomes, not four: with a version in hand and the row **deleted**, a naive upsert resurrects it and reports success. Do **not** "normalize" these signatures. They were measured.
2. **Never call a bump-only method on a user edit; never call a checked method on a system write** (reorder, soft-delete, backfill). *Bumping without checking is safe; checking without bumping is the bug the mechanism exists to prevent.*
3. **Never re-read the version after a write.** Use the `long` the checked call returned. Re-reading is racy: between your write committing and your re-read, another client can write — you would then hold *their* version with *your* data, and your next save would pass the version check and silently overwrite them. That is the exact lost update this whole thing exists to prevent, laundered through a read-back. `GetRowVersionAsync` was deleted for this reason. Do not reintroduce it.
4. **The seven versioned entities' DTOs carry `rowVersion`.** `TaskTemplate`, `DefaultTask` and `Holiday` have no `row_version` column — do not invent one.
5. **Team scope (R6) is not optional.** Every query is scoped to `IClientContext.TeamIds`. There is already a regression test for a cross-team leak (`TM-06`).
6. After **every** successful mutation: `IChangeNotifier.DataChangedAsync(kind, teamId, exceptConnectionId: clientContext.ConnectionId)`. Wave 3 makes it real.

### W2-D only — `RetentionService` must not run in the request path

Spec §8.4: it holds one `BEGIN IMMEDIATE` across six bulk DELETEs, **blocking every writer app-wide**. Wired straight into `POST /api/ops/retention/run`, **one admin click 500s everyone else.** Return **202 Accepted** and run it on a background queue.

# Wave 3 — SignalR. ONE agent. `[sonnet]`

**W3-T1 — Replace `NoopChangeNotifier` with the real hub.**

**Owns:** `Hubs/DataHub.cs` · `Infrastructure/SignalRChangeNotifier.cs` · `Program.cs` (**the one `MapHub` line**) · its tests. **Touches no endpoint file** — that is what the Wave-1 `IChangeNotifier` seam and `IClientContext.ConnectionId` bought.

Replaces `DataChangedMessage` (`WeakReferenceMessenger` — in-process only, which is precisely why **two users never see each other's changes without reloading today**).

- Hub at `/hubs/data`; clients joined to a **group per team**, preserving R6.
- After each successful mutation: broadcast `DataChanged(DataKind, teamId)`.
- `DataKind` has **exactly 11 values** (verified, in `Core/Services/DataChangedMessage.cs`); the listener table in `.planning/M8-FEATURE-INVENTORY.md` §0.3 ports 1:1. Templates / DefaultTasks / Holidays are owned by W2-D, so all 11 are raised by something.

### Two hub facts that bite silently

- **Group membership is lost on every reconnect.** Rejoin in `OnConnectedAsync`, or cross-user sync dies after any Wi-Fi blip — **and appears to work perfectly until then.**
- **`IHubContext` has no `Others`.** The editing user receives their own `DataChanged` echo, which re-fetches and **clobbers the very conflict dialog the 409 just raised.** Exclude the caller — that is what `exceptConnectionId` is for, and what `IClientContext.ConnectionId` (read from the `X-Connection-Id` header the JS client sends) exists to supply.

- [ ] **Verify:** two clients in one team → A's mutation reaches B and **not** A. **Then force B to reconnect and repeat — B must still receive it.** That second half is the test that otherwise gets written months later, after the bug.
- [ ] **Done:** full suite green.

Commit: `feat(M8.3): SignalR — group per team, rejoin on reconnect, no self-echo`
