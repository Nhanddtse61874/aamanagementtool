# M8.3 — API + Auth + SignalR Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development`.

**Goal:** Stand up `TimesheetApp.Api` so the Angular app has something real to talk to — authenticated, team-scoped, conflict-aware, and pushing live changes.

**Architecture:** Minimal APIs over the existing `TimesheetApp.Core` services. Cookie auth on the existing `Users` table. SignalR replaces the in-process `WeakReferenceMessenger`. The API is the **only writer process** — that is what restores the single-writer property SQLite needs (spec §2).

**Tech Stack:** ASP.NET Core 8, `Microsoft.Extensions.Identity.Core` (hasher only — no EF Core, no Identity schema), SignalR, `WebApplicationFactory`.

**Spec:** `docs/superpowers/specs/2026-07-12-m8-backend-foundation-design.md` §8, §10, §11.
**Depends on:** M8.2 complete (schema v10, `row_version` reachable end to end, `ActiveTeamId` per-user).

---

## The wave structure exists to prevent one specific failure

M8.2's Wave 3 put four agents on four repositories. They wrote **three** different concurrency APIs, and left `Models/Entities.cs` — the file all four needed — completely untouched, because none of them owned it. Consolidating that cost an extra wave.

The same trap is waiting here, and it is bigger: **four agents writing endpoints would all edit `Program.cs`, and all invent their own DTO shape, their own 409 body, and their own way to broadcast a change.**

So **Wave 1 is one agent and it builds nothing but the shared surface.** Every contract that more than one later agent touches is fixed, in code, before any fan-out. Waves 2 and 3 then own strictly disjoint files.

**This is the rule the rest of this plan is designed around: a file that N parallel agents need is a file the controller writes first.**

### The five things that fail SILENTLY if you get them wrong

Each of these was found by research reading the framework source. They do not throw, they do not log, and they all look like something else.

1. **Cookie auth does not 401 on .NET 8 — it 302s.** The *only* trigger for a 401 is the literal request header `X-Requested-With: XMLHttpRequest`. Everything else gets `Response.Redirect("/Account/Login")` → a route that does not exist → **Angular sees 404, never 401.** Microsoft changed this default only in .NET 10. Required:
   ```csharp
   o.Events.OnRedirectToLogin        = ctx => { ctx.Response.StatusCode = 401; return Task.CompletedTask; };
   o.Events.OnRedirectToAccessDenied = ctx => { ctx.Response.StatusCode = 403; return Task.CompletedTask; };
   ```
   **The twist that will cost a day:** the SignalR JS client *does* send that header, so **the hub 401s correctly while the API 404s.** The asymmetry sends you hunting in the wrong file.

2. **Data Protection is the highest-severity gap in the whole design, and it is invisible until production.** The auth cookie is encrypted with a Data Protection key ring. Unconfigured, that ring lives **in memory** — so **every app restart logs everyone out.** The host is a workstation: it reboots, it updates, it recycles. That is mass logouts roughly daily, **with nothing in the logs**, silently defeating the "stay logged in" requirement that is the entire reason we chose a persistent cookie. `PersistKeysToFileSystem` into a folder that survives restarts and is backed up with the database.

3. **`FallbackPolicy = DefaultPolicy` locks the user out of the login page.** It applies to *"requests served by other middleware after the authorization middleware, such as static files"* — so a logged-out user requesting Angular's `index.html` gets 401 and can **never reach the form that would log them in**. It also blocks `MapFallbackToFile`, health checks, and `/api/auth/login` itself. Required: `[AllowAnonymous]` on login, `.AllowAnonymous()` on the SPA fallback and static files. (SignalR's `negotiate` being blocked is *correct* — leave it.)

4. **`RequireClaim("is_admin", "1")` compares with `StringComparer.Ordinal`.** Write the claim from a `bool` and you get `"True"` — the policy then **always fails** and your admin gets a **silent 403 with nothing logged**. Write the literal `"1"`. Add a test that a real admin passes, not just that a non-admin fails.

5. **`busy_timeout` does not bound how long a request hangs.** Microsoft.Data.Sqlite auto-retries busy/locked errors up to `CommandTimeout` — **default 30 s**. Measured with the previous settings: a blocked writer failed after **33,940 ms**, not 5,000. Put **`DefaultTimeout=5`** in the connection string (a valid keyword; it flows into every Dapper call, so no repository changes) and `busy_timeout=1000` → worst-case hang ≈ 5.2 s.

### Contracts you are fixing for everyone downstream

**Every entity DTO carries `rowVersion`.** M8.2 made `row_version` reachable — the entities carry it and every SELECT projects it. If a DTO drops it, the client cannot send back the `expectedVersion` a checked write requires, and the entire concurrency mechanism is dead on arrival. This is exactly the gap that cost M8.2 an extra wave; do not recreate it one layer up.

**The 409 body** (spec §8.1). `rowsAffected == 0` conflates *"your version is stale"* with *"the row is gone"*, and the user-facing wording differs:
```csharp
public sealed record ConflictBody(string Table, long Id, bool Deleted, object? Current, string Message);
```
Known limitation to design around, not fix: **`TimeLogs` has no `changed_by` column**, so the API *cannot name who* changed a cell. The "Someone else just changed this" dialog can name the person for `Backlogs` (via `BacklogAudit`) but **not** for timesheet cells.

**`IChangeNotifier`** — Wave 2 endpoints call it after every successful mutation; Wave 3 swaps the implementation for the real hub. Wave 1 ships a no-op. This is what keeps Wave 3 from having to edit all four endpoint files:
```csharp
public interface IChangeNotifier { Task DataChangedAsync(DataKind kind, int teamId, string? exceptConnectionId = null); }
```

**`CurrentUserService` is NOT modified.** It already takes its identity through a `Func<string>` seam:
```csharp
new CurrentUserService(users, () => Environment.UserName)                    // WPF, unchanged
new CurrentUserService(users, () => ctx.HttpContext!.User.Identity!.Name!)   // API
```

**Admin bootstrap — no persistent secret.** Migration v10 promoted the first user to `is_admin = 1`, but gave them no `password_hash`, so today **nobody can log in and nobody can set one.** On startup, if the designated admin's `password_hash IS NULL`, generate a random password, apply it, and write it **once** to the console. The operator reads it, logs in, changes it. There is no secret to leak because it exists only in that one log line.
- Applied as a single atomic `UPDATE … WHERE password_hash IS NULL` — which also neutralises the race under an overlapped service restart.
- **No "first login claims the account" flow.** On a shared internal network that lets anyone claim a colleague's account before they do.

**Login must check `is_active`.** Omit it and a soft-deleted user who still has a password can log in.

- [ ] **Done:** a logged-out `GET /api/me` returns **401** (not 302, not 404). ### Restated verbatim in every Wave 2 brief — because an agent sees its brief, not this plan

M8.2 proved that a decision written down centrally but not repeated in each brief **is not a decision**: three of four agents independently re-derived a different concurrency API from a rule that was already recorded.

1. **A checked write is `<Verb>CheckedAsync`, takes `long expectedVersion`, and returns `Task<long>` (the new `row_version`).** Never an optional parameter. Never a `void` return.
2. **Never call a bump-only method on a user edit**, and never call a checked method on a system write (reorder, soft-delete, backfill). *Bumping without checking is safe; checking without bumping is the bug the mechanism exists to prevent.*
3. **Never re-read the version after a write.** Use the `long` the checked call returned. Re-reading is racy: between your write committing and your re-read, another client can write — you would then hold *their* version with *your* data, and your next save would pass the version check and silently overwrite them. That is the exact lost update this whole thing exists to prevent, laundered through a read-back. `GetRowVersionAsync` was deleted for this reason; do not reintroduce it.
4. **Every response DTO carries `rowVersion`.** A DTO that drops it kills the mechanism for that entity.
5. **Team scope (R6) is not optional.** Every query is scoped to the caller's teams. This is the rule the whole codebase is built around, and there is already a regression test for a cross-team leak (`TM-06`).
6. Call `IChangeNotifier.DataChangedAsync(kind, teamId, exceptConnectionId: <caller's>)` after **every** successful mutation. Wave 3 makes it real.

# Wave 3 — SignalR. ONE agent. `[sonnet]`

**W3-T1 — Replace `NoopChangeNotifier` with the real hub.**

**Owns:** `Hubs/DataHub.cs` · `Infrastructure/SignalRChangeNotifier.cs` · `Program.cs` (the one `MapHub` line) · its tests.
**Touches no endpoint file** — that is what the Wave-1 `IChangeNotifier` seam bought.

Replaces `DataChangedMessage` (`WeakReferenceMessenger` — in-process only, which is why **two users never see each other's changes without reloading today**).

- Hub at `/hubs/data`; clients joined to a **group per team**, preserving the R6 no-leak rule.
- After each successful mutation: broadcast `DataChanged(DataKind, teamId)`.
- `DataKind` keeps all 11 values; the listener table in `.planning/M8-FEATURE-INVENTORY.md` §0.3 ports 1:1.

### Two hub facts that bite silently

- **Group membership is lost on every reconnect.** Rejoin in `OnConnectedAsync`, or cross-user sync dies after any Wi-Fi blip — **and appears to work perfectly until then.**
- **`IHubContext` has no `Others`.** The editing user receives their own `DataChanged` echo, which re-fetches and **clobbers the very conflict dialog the 409 just raised.** Exclude the caller's connection — that is what `exceptConnectionId` on `IChangeNotifier` is for.

- [ ] **Verify:** two clients, one team → A's mutation reaches B and **not** A. Then force a reconnect on B and repeat — B must still receive it. That second half is the test that would otherwise be written months later, after the bug.
- [ ] **Done:** full suite green.

Commit: `feat(M8.3): SignalR — group per team, rejoin on reconnect, no self-echo`
