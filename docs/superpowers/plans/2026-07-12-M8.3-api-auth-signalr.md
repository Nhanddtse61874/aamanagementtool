# M8.3 — API + Auth + SignalR Implementation Plan

> **REQUIRED SUB-SKILL:** `superpowers:subagent-driven-development`.

**Goal:** Stand up `TimesheetApp.Api` so the Angular app has something real to talk to — authenticated, team-scoped, conflict-aware, rule-preserving, pushing live changes.

## rev. 3 — BLOCKED twice. Both times for the same reason, and it is worth naming.

Rev. 1 and rev. 2 each assumed **Core already exposes what the API needs**. Rev. 1 was wrong about credentials. Rev. 2 fixed credentials and was wrong about **versions** and **team scope** — the same error, one layer over.

The root cause is now clear, and it is the most important sentence in this document:

> **M8.2 built the concurrency mechanism across eight repositories, five waves and +80 tests — and stopped one layer short of the only caller that needs it.**

```
$ grep -rln 'expectedVersion' src/TimesheetApp.Core/Services/
(no hits)

$ sed -n '58p' src/TimesheetApp.Core/Services/TimeLogService.cs
        await _logs.UpsertAsync(new TimeLog(0, userId, taskId, date, rounded, _clock.UtcNow));
```

`UpsertAsync`'s own XML doc — written by the agent that built the checked path — says:

> *"BUMP-ONLY: always increments row_version, never compares it, so it can never conflict. This is the system/legacy write. **A version-aware caller (the web API) must use `UpsertCheckedAsync` instead; this overload silently wins every race.**"*

The doc names the API as the caller that must not use it. **And there is no service method that does.** Left unnamed, four Wave-2 agents would each pick a side, and **both sides are broken**:

- **Through the service:** `ConcurrencyConflictException` is never thrown. `ExceptionMapper` never fires. OT-5 fails, `rowVersion` on every DTO becomes write-only decoration, and M8.2's four waves are unreachable from the web.
- **Around the service, straight to the repository:** concurrency works — and you bypass the 8h/day cap (XC-03), the holiday guard (HOL-02), the weekend guard, the 1-decimal rule, the XC-10 pre-bulk backup, the XC-09 journal check, and the standup **owner gate** (`StandupService.cs:157` — `if (existing.UserId != _currentUser.Current?.Id) return false;`). **The API would let a user log 40 hours on a Sunday holiday and edit a colleague's standup entry.** Every existing test still passes, because the WPF path is untouched.

So **Wave 0 now has two tasks**, and the second one is the missing layer.

### The other thing two rounds of review found: `teamIds = null` means *every team*

```csharp
// TimeLogRepository.cs — GetExportRowsAsync has NO userId parameter at all
Task<IReadOnlyList<TimeLogReportRow>> GetExportRowsAsync(
    DateOnly from, DateOnly to, string? projectFilter, IReadOnlyList<int>? teamIds = null);

var noTeam = teamIds is null;
//   AND (@noTeam OR r.team_id IN @teamIds)      <-- null => no filter => ALL teams, ALL users
```

On `/api/export/*`, forgetting the parameter exports the whole company. Passing the client's `?teamIds=` straight through — **the natural implementation, and what the WPF TeamFilter does** — lets any authenticated user export any team by guessing an id. `TM-06` does not catch this: it tests the WPF path, where the filter is sourced from `AvailableTeams` and is therefore already membership-bounded by construction. **On the wire, the filter is attacker-controlled.** Same shape on `IBacklogRepository.SearchAsync` and `IStandupRepository.GetEntriesForDayAsync`.

### And the structural lesson, now paid for three times

**A file — or a route, or a layer — that N parallel agents need is the controller's to fix BEFORE the fan-out.** `ConcurrencyConflictException` was done that way and cost nothing. `Models/Entities.cs` was not, and cost a consolidation wave. The service layer was not, and would have cost either the whole concurrency mechanism or every business rule in the product.

# Wave 0 — What Core does not have. ONE agent, two tasks, sequential. `[opus]`

## W0-T1 — `IUserRepository` grows a credential surface

**Owns:** `src/TimesheetApp.Core/Data/Repositories/IUserRepository.cs` · `UserRepository.cs` · `src/TimesheetApp.Core/Models/Entities.cs` · their tests.

`password_hash` and `is_admin` are columns that **nothing reads**. They appear only in the `DatabaseInitializer` DDL and one migration test — no repository method, no entity property. Migration v10 promoted the first user to `is_admin = 1` and gave them no hash, so **as it stands nobody can log in and nobody can set a password.**

```csharp
// The hash does NOT go on `User`. `User` is returned by a dozen endpoints, and a password
// hash must never be one careless projection away from a DTO.
public sealed record UserCredentials(int Id, string Name, string? PasswordHash, bool IsAdmin, bool IsActive);

Task<UserCredentials?> GetCredentialsAsync(string username);
Task SetPasswordHashAsync(int userId, string hash);                   // BUMP-ONLY — no client version
Task<bool> TryBootstrapAdminPasswordAsync(int userId, string hash);   // atomic: UPDATE … WHERE password_hash IS NULL
```

`TryBootstrapAdminPasswordAsync` returns `rowsAffected == 1`. The `WHERE password_hash IS NULL` is what makes it safe under an overlapped service restart: two processes racing produce one winner and one no-op, not two passwords.

**Also add `bool IsAdmin` to the `User` record** and project it. `/api/users` needs to show who is an admin, and Wave 2 is forbidden from touching Core — so if it is not added here it cannot be added at all.

- [ ] **Verify:** a user with a hash round-trips; `GetCredentialsAsync` on an unknown username → `null`; `TryBootstrapAdminPasswordAsync` → `true` once, `false` every time after.

## W0-T2 — The service layer learns about versions. **This is the task the whole milestone rests on.**

**Owns:** `src/TimesheetApp.Core/Services/ITimeLogService.cs` · `TimeLogService.cs` · `IStandupService.cs` · `StandupService.cs` · their tests.

Add **checked overloads** that thread a version into the `*CheckedAsync` repository methods **that already exist**. Keep every existing overload exactly as-is — WPF calls them and must not change (it is deleted at M8.10).

```csharp
// ITimeLogService — new. Validation is a RESULT (-> 400); a conflict is an EXCEPTION (-> 409).
public readonly record struct SaveCellResult(bool Ok, string? Error, long RowVersion);

Task<SaveCellResult> SaveCellCheckedAsync(int userId, int taskId, DateOnly date, decimal hours, long? expectedVersion);
Task ClearCellCheckedAsync(int userId, int taskId, DateOnly date, long expectedVersion);

// IStandupService — new.
Task<long> UpdateIssueCheckedAsync(StandupIssue issue, long expectedVersion);
```

**The validation must run first, then the checked write.** Same order as today — the overload changes *which repository method* is called, nothing else. `SaveCellCheckedAsync` runs the 8h cap, holiday, weekend and rounding rules, returns `Ok=false` with the message on failure, and only then calls `UpsertCheckedAsync`.

`expectedVersion` is `long?` on `SaveCellCheckedAsync` because the underlying `UpsertCheckedAsync(TimeLog, long?)` is nullable **by design** — an upsert has five outcomes, not four, and `null` legitimately means *"I believe this cell is empty"*. Do not normalize it.

**Do NOT add a checked variant for:**
- **Smart Fill** (`ApplySmartFillAsync`, `ApplySmartInputAsync`) — the recorded decision is that Smart Fill **bumps but does not check**. If it bumped nothing, a client holding a stale version would still match and silently overwrite Smart Fill's result — the very lost update this mechanism exists to kill, reintroduced by its own exception.
- **`StandupService.UpdateEntryAsync`** — `StandupEntry` is deliberately **not versioned** (it has no `RowVersion`; `IStandupRepository.UpdateEntryAsync` has no checked sibling). It is protected by an **owner gate** instead, so it is last-write-wins *by design*. Do not invent a version for it.

- [ ] **Verify:** two clients read the same cell at v1; A saves; **B's save at v1 throws `ConcurrencyConflictException`** — *through the service*, with the 8h/holiday/weekend rules still enforced. Then prove the rules still bite: a checked save of 40h on a holiday returns `Ok=false`, **not** a conflict and **not** a write.
- [ ] **Done:** full suite green (628 + yours). Every pre-existing overload untouched.

Commit: `feat(M8.3): Core services learn about row_version — the mechanism stopped one layer short`

## W1a — The host boots, with scope validation ON

**Owns:** `src/TimesheetApp.Api/TimesheetApp.Api.csproj` · `Program.cs` · `Infrastructure/IClientContext.cs` · `ApiClientContext.cs` · `ApiCurrentTeamService.cs` · `ClientContextFilter.cs` · `src/TimesheetApp.ApiTests/*.csproj` · `ApiFactory.cs` · `HostBootTests.cs` · `src/TimesheetApp.sln`

### The lifetime graph is the deliverable

The WPF composition root (`App.xaml.cs:134-196`) registers ~35 services as `AddSingleton`. **Do not copy it.** On a desktop, one process serves one user, so singleton per-user state is correct. On a server it is a cross-user data leak.

Anything holding per-user state — `ICurrentUserService` (holds `Current`), `ICurrentTeamService` (holds the active team) — is **Scoped**. Anything injecting those transitively (`TimeLogService`, `StandupService`, `BacklogContinuationService`, …) is **Scoped** too. Only genuinely stateless, config-only services stay Singleton.

`ICurrentTeamService` **has no server-usable implementation** — its only impl is in the WPF project (`net8.0-windows`, ctor takes `IMessenger`), which `net8.0` cannot reference; yet Core's `TimeLogService` and `StandupService` both require it. Spec §6.2: *"make `ICurrentTeamService` **scoped per-request** in the API, resolving from the authenticated user."* Write `ApiCurrentTeamService` over `IUserRepository.GetActiveTeamIdAsync` / `SetActiveTeamIdAsync` (they exist — W3-C added them, W4 wired them). Its `ActiveTeamChanged` event is a **no-op** in the API: a team switch is per-user state and must **not** broadcast to the team group.

### `SqliteProfile.Server` already exists — and the default will silently give you `Desktop`

```csharp
public SqliteConnectionFactory(IAppConfig config, SqliteProfile profile = SqliteProfile.Desktop)
```

`SqliteProfile.Server` (WAL, `busy_timeout=1000`, **`DefaultTimeout=5`**) is already implemented and referenced by **exactly one test and zero production code**. A plain `AddScoped<IConnectionFactory, SqliteConnectionFactory>()` gets the **default — Desktop** — which means `journal_mode=DELETE`, no `DefaultTimeout`, and no WAL. Silently. Two consequences, both bad:

- Microsoft.Data.Sqlite **auto-retries** busy/locked errors up to `CommandTimeout` (default **30 s**). Measured: a blocked writer failed after **33,940 ms**, not 5,000.
- Without WAL, **readers block writers** — the API serialises every request.

Register it explicitly: `new SqliteConnectionFactory(cfg, SqliteProfile.Server)`, and **assert `PRAGMA journal_mode` returns `wal`** from a connection resolved out of the API's own container.

### The per-request initializer — without it, every timesheet endpoint returns empty

**Nothing calls `CurrentUserService.ResolveAsync()` for you, and nothing calls `ICurrentTeamService.InitializeAsync(userId)`.** `ActiveTeamId` is a **sync property** whose resolution needs an **async DB read**, so it cannot lazily resolve — it must be eagerly initialized per request.

Skip it and: `ActiveTeamId` stays `0` → `GetActiveForTimesheetAsync(0)`, whose own doc says *"teamId 0 ⇒ no tasks (empty, R6)"* → **every timesheet and standup endpoint returns empty**, and `StandupService.AddEntryAsync` writes `TeamId: 0`, **poisoning data**. Meanwhile `ICurrentUserService.Current` is `null`, so the standup owner gate compares against nothing.

`ClientContextFilter` runs on every authenticated request and does, **in this order**:
1. `await currentUser.ResolveAsync()`
2. `await currentTeam.InitializeAsync(user.Id)`
3. populate `IClientContext`

```csharp
public interface IClientContext        // scoped; the ONE place identity, authz bound and connection id live
{
    int UserId { get; }
    string UserName { get; }
    bool IsAdmin { get; }
    IReadOnlyList<int> MemberTeamIds { get; }   // AUTHORIZATION BOUND — from ITeamRepository.GetTeamIdsForUserAsync
    string? ConnectionId { get; }               // SignalR caller, from the X-Connection-Id header
}
```

**`MemberTeamIds`, not `TeamIds`.** The name matters because three different things were being conflated:
1. **Authorization bound** — the user's memberships. Source: **`ITeamRepository.GetTeamIdsForUserAsync`** — *not* `IUserRepository`, where an agent would naturally look after Wave 0 added things there.
2. **Working scope** — `ICurrentTeamService.ActiveTeamId`, a **single int**, read *internally* by `TimeLogService` (4 sites) and `StandupService` (7 sites). For service-routed queries R6 is enforced **there**.
3. **Filter** — the client's multi-select. Attacker-controlled. See Wave 2 rule #5.

`ConnectionId` is here because `IChangeNotifier.DataChangedAsync(…, exceptConnectionId)` needs the caller's SignalR connection id **and it is not on `HttpContext`.** The JS client sends it as a header; something must read it out. If that reader is not in Wave 1, four Wave-2 agents each invent one and Wave 3 discovers the divergence.

### The four silent failures

Each was found by reading framework source. None throws. None logs. Each looks like something else.

1. **Cookie auth does not 401 on .NET 8 — it 302s.** The *only* trigger for a 401 is the literal header `X-Requested-With: XMLHttpRequest`. Everything else gets `Response.Redirect("/Account/Login")` → a route that does not exist → **Angular sees 404, never 401.** Fixed by Microsoft only in .NET 10.
   ```csharp
   o.Events.OnRedirectToLogin        = ctx => { ctx.Response.StatusCode = 401; return Task.CompletedTask; };
   o.Events.OnRedirectToAccessDenied = ctx => { ctx.Response.StatusCode = 403; return Task.CompletedTask; };
   ```
   **The twist that costs a day:** the SignalR JS client *does* send that header — so **the hub 401s correctly while the API 404s.** The asymmetry sends you hunting in the wrong file.

2. **Data Protection is the highest-severity gap in the design, and it is invisible until production.** The auth cookie is encrypted with a Data Protection key ring. Unconfigured, that ring lives **in memory** — so **every process restart logs everyone out.** The host is a workstation: it reboots, it updates, it recycles. Mass logouts, roughly daily, **with nothing in the logs**, silently defeating the "stay logged in" requirement that is the whole reason we chose a persistent cookie. `PersistKeysToFileSystem`, into a folder that survives restarts and is backed up with the database.

3. **`FallbackPolicy = DefaultPolicy` locks the user out of the login page.** It applies to *"requests served by other middleware after the authorization middleware, such as static files"* — so a logged-out user requesting Angular's `index.html` gets 401 and can **never reach the form that would log them in**. It also blocks `MapFallbackToFile`, health checks, and `/api/auth/login` itself. `[AllowAnonymous]` on login; `.AllowAnonymous()` on the SPA fallback and static files. (SignalR's `negotiate` being blocked is **correct** — leave it.)

4. **`RequireClaim("is_admin", "1")` compares with `StringComparer.Ordinal`.** Write the claim from a `bool` and you get `"True"` — the policy then **always fails** and your admin gets a **silent 403 with nothing logged**. Write the literal `"1"`. Test that a real admin *passes*, not merely that a non-admin fails.

### TWO error channels, not one

This is a contract, and if Wave 1 does not freeze it, four agents will invent four different 400 bodies.

| Failure | Mechanism | Status |
|---|---|---|
| Business rule (8h cap, holiday, weekend, decimals) | **`SaveResult(bool Ok, string? Error)`** — a **return value**, not an exception (`ReadModels.cs:45`) | **400** + `ValidationBody` |
| Version conflict | **`ConcurrencyConflictException`** — thrown | **409** + `ConflictBody` |

`ExceptionMapper` covers only the second. The first never throws, so an endpoint that ignores `SaveResult.Ok` **returns 200 on a rejected write** and the user watches their hours vanish.

```csharp
public sealed record ValidationBody(string Error);
public sealed record ConflictBody(string Table, long Id, bool Deleted, string? Detail, string Message);
```

**`Detail` is not garnish.** W3.5 added it *specifically* so a 409 on a timesheet cell can say **which** cell: `TimeLogs`' key is the natural triple `(user_id, task_id, work_date)`, so its `Id` is **`0`**, and without `Detail` the client is told only "something in TimeLogs changed". Rev. 1 dropped it one hour after W3.5 added it. Do not drop it again.

**DEVIATION from spec §8.1, recorded:** the 409 body omits `current` (the server's current row). The exception does not carry it, and a global handler cannot re-read it — that needs a `Table → repository` dispatch map, and for `TimeLogs` (`Id = 0`) `GetByIdAsync` is not even expressible. The alternatives were to put `Current` on the exception (re-opening all eight repositories one wave after consolidating them) or to parse `Detail` strings (fragile). **Instead the client re-fetches on 409.** One round-trip on a by-definition-rare path, and it is not racy: a re-fetch returns a coherent (state, version) pair, and if someone writes again in between, the next save 409s again — which is correct.

Known limitation, by design: **`TimeLogs` has no `changed_by` column**, so the API cannot name *who* changed a cell. "Someone else just changed this" can name the person for `Backlogs` (via `BacklogAudit`) but **not** for timesheet cells.

### DTOs — eight versioned records, and three that are not

**Eight carry `RowVersion`:** `User`, `Team`, `Backlog`, `TaskItem`, `TimeLog`, `Tag`, `PcaContact` (in `Entities.cs`) and **`StandupIssue`** (in `StandupModels.cs`). Their DTOs must expose it as `rowVersion` — drop it and the client cannot send back the `expectedVersion` a checked write requires.

**Do NOT invent a version for:** `TaskTemplate`, `DefaultTask`, `Holiday` (no `row_version` column) or **`StandupEntry`** (deliberately unversioned — owner-gated, last-write-wins by design). W3-D already invented a convention once when handed an over-broad instruction; it justified an edit by "this file's existing convention", and the convention did not exist.

Also: the *column* is `username` (v10) but the *record property* is still `WindowsUsername` — renaming it was judged out of scope at 28+ consumers. `Dtos.cs` maps `WindowsUsername` → `username`.

### Admin bootstrap — no persistent secret

On startup, if the designated admin's `password_hash IS NULL`, generate a random password, apply it via `TryBootstrapAdminPasswordAsync`, and write it **once** to the console. The operator reads it, logs in, changes it. **There is no secret to leak, because it exists only in that one log line.** (An earlier draft read it from configuration — that puts a credential in a file that tends to get committed.)

**No "first login claims the account" flow.** On a shared internal network that lets anyone claim a colleague's account before they do.

**Login must check `is_active`.** Omit it and a soft-deleted user who still has a password can log in.

`CurrentUserService` is **not modified** — it already has the seam (`CurrentUserService.cs:17`):
```csharp
new CurrentUserService(users, () => Environment.UserName)                   // WPF, unchanged
new CurrentUserService(users, () => ctx.HttpContext!.User.Identity!.Name!)  // API
```

### OpenAPI — M8.4 depends on it

Add Swashbuckle and expose the document. **M8.4 generates its TypeScript client from it.** The vendored Angular bundle's models are unusable as a wire contract: `User` has no `id`, `Backlog` has no `id`, `LogGroup.tasks` is `string[]`, and `HoursMap` is keyed by `` `${groupIndex}-${taskIndex}-${dayIndex}` `` — **array indices**. One sort or filter and hours land on the wrong task; and with no id there is nowhere to put a `rowVersion`. Hand-maintaining those types across the wire rebuilds that bug on purpose.

## The seven rules — restated verbatim in every Wave-2 brief

An agent sees its brief, not this plan. M8.2 proved that a decision recorded centrally but not repeated in each brief **is not a decision**: three of four agents independently re-derived a different concurrency API from a rule that was already written down.

**1 — Which layer you call is not your choice. It is a split, and the line is where the business rules are.**

| Entity | You MUST call | Why |

Go around it and a user logs **40 hours on a Sunday holiday**. |
| **Standup** (W2-D) | **`IStandupService`** — `UpdateIssueCheckedAsync`, `UpdateEntryAsync`. **Never `IStandupRepository`.** | The service holds the **owner gate** (`StandupService.cs:157`). Go around it and a user **edits a colleague's standup entry**. |
| **Backlogs · Tasks · Teams · Tags · PcaContacts · Users** | the repository's `*CheckedAsync` **directly** | **There is no service for these** — the WPF ViewModels call the repositories directly too. There is no logic to bypass. But see rule #4: the audit parameters *are* the logic. |

**2 — Read the real signature. Do not assume a shape.** Only **4 of 15** checked writes take `(entity, long expectedVersion)`. Most take `(int id, field…, long expectedVersion)`. And `TimeLogs` is deliberately different:
```csharp
Task<long> UpsertCheckedAsync(TimeLog log, long? expectedVersion);              // NULLABLE — the five-case table
Task       DeleteCheckedAsync(int u, int t, DateOnly d, long expectedVersion);  // returns Task, NOT Task<long>
```
`expectedVersion` is nullable there because an upsert has **five** outcomes, not four: with a version in hand and the row **deleted**, a naive upsert resurrects it and reports success. **Do not "normalize" these.** They were measured.

**3 — Never re-read the version after a write.** Use what the checked call returned. Re-reading is racy: between your write committing and your re-read, another client can write — you would hold *their* version with *your* data, and your next save would pass the check and silently overwrite them. That is the exact lost update this exists to prevent, laundered through a read-back. `GetRowVersionAsync` was deleted for this reason. Do not reintroduce it.

**4 — Every audited checked write passes `changedByUserId: ctx.UserId, changedByName: ctx.UserName`.** Five checked writes carry these as **optional params defaulting to `null`**:
```csharp
Task<long> UpdateCheckedAsync(Backlog b, long expectedVersion,
    int? changedByUserId = null, string? changedByName = null, string? auditNote = null);
Task<long> UpdateStatusCheckedAsync(int taskId, string status, long expectedVersion,
    int? changedByUserId = null, string? changedByName = null);
// … and SetTagsChecked, UpdateExtendedChecked, SetTaskTagsChecked
```
Omit them and **it compiles, the tests pass, and every web edit writes an anonymous audit row.** The product feature "who changed this backlog" quietly dies, and nothing tells you.

**5 — R6. Any client-supplied `teamIds` is INTERSECTED with `ctx.MemberTeamIds` before it reaches a repository. Absent ⇒ default to `ctx.MemberTeamIds`. NEVER pass `null`.**
```csharp
var noTeam = teamIds is null;
//   AND (@noTeam OR r.team_id IN @teamIds)     <-- null means NO FILTER: every team, every user
```
`GetExportRowsAsync` has **no `userId` parameter at all**. Forget the filter on `/api/export/*` and you export the whole company. Pass the client's `?teamIds=` straight through — **the natural implementation** — and any authenticated user exports any team by guessing an id. `TM-06` does **not** cover this: it tests the WPF path, where the filter comes from `AvailableTeams` and is membership-bounded by construction. On the wire, the filter is attacker-controlled.
**Each Wave-2 agent adds one negative test:** authenticated user + a team id they are not a member of → **empty or 403, never rows.**

**6 — Two error channels.** `SaveResult.Ok == false` → **400** + `ValidationBody`. `ConcurrencyConflictException` → **409** + `ConflictBody` (the mapper handles it; let it throw). An endpoint that ignores `SaveResult.Ok` **returns 200 on a rejected write**, and the user watches their hours vanish.

**7 — After every successful mutation:** `IChangeNotifier.DataChangedAsync(kind, teamId, exceptConnectionId: ctx.ConnectionId)`. Wave 3 makes it real.

# Wave 3 — SignalR. ONE agent. `[sonnet]`

**W3-T1 — Replace `NoopChangeNotifier` with the real hub.**

**Owns:** `Hubs/DataHub.cs` · `Infrastructure/SignalRChangeNotifier.cs` · `Program.cs` (**the one `MapHub` line**) · its tests. **Touches no endpoint file** — that is what the Wave-1 `IChangeNotifier` seam bought.

Replaces `DataChangedMessage` (`WeakReferenceMessenger` — in-process only, which is precisely why **two users never see each other's changes without reloading today**).

- Hub at `/hubs/data`; clients joined to a **group per team**, preserving R6.
- After each successful mutation: broadcast `DataChanged(DataKind, teamId)`.
- `DataKind` has **exactly 11 values** (verified, `Core/Services/DataChangedMessage.cs`); the listener table in `.planning/M8-FEATURE-INVENTORY.md` §0.3 ports 1:1. W2-D owns templates/defaults/holidays, so every value is raised by something.

### Three facts that bite silently

- **SignalR hubs do NOT run endpoint filters.** So `DataHub.OnConnectedAsync` **cannot use `IClientContext`** — it will be null, the hub will join **zero groups**, and cross-user sync will simply never work, with no error. The hub must read the user's teams itself, via `ITeamRepository.GetTeamIdsForUserAsync` off `Context.User`.
- **Group membership is lost on every reconnect.** Rejoin in `OnConnectedAsync`, or cross-user sync dies after any Wi-Fi blip — **and appears to work perfectly until then.**
- **`IHubContext` has no `Others`.** The editing user receives their own `DataChanged` echo, which re-fetches and **clobbers the very conflict dialog the 409 just raised.** Exclude the caller — that is what `exceptConnectionId` and `IClientContext.ConnectionId` (read from the `X-Connection-Id` header) exist for.

- [ ] **Verify:** two clients in one team → A's mutation reaches B and **not** A. **Then force B to reconnect and repeat — B must still receive it.** That second half is the test that otherwise gets written months later, after the bug.
- [ ] **Done:** full suite green.
