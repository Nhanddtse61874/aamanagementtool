# Wave 1 — The shared files nobody owns. **CONTROLLER, not an agent.**

`STATE.md`'s own lesson, and rev. 1 broke it again: **a file that N later agents need is the controller's to write FIRST.** Rev. 1 left `angular.json`, `proxy.conf.json` and the test config owned by nobody.

## W1a — Same-origin, or nothing works at all

`ng serve` (:4200) → Kestrel (:5xxx) is **cross-origin**, and it fails in a three-link chain where **each link is fatal on its own**:

1. `Program.cs` has **no `AddCors`/`UseCors`** → the browser blocks every response. `PUT`/`DELETE` with a JSON body are **preflighted**, and there is no `OPTIONS` handler → **blocked before the request ever reaches C#.**
2. Add CORS and it **still** fails: **a `SameSite=Lax` cookie is not sent on a cross-site `fetch`/XHR.** Login would 200 and set the cookie; every subsequent call would go out **anonymous** → 401 → the interceptor bounces to `/login` → **redirect loop.**
3. Fix *that* with `SameSite=None` and it **requires `Secure`**, which requires **HTTPS** — which this project **explicitly rejected** (spec §D8, recorded accepted risk). Chrome rejects a `SameSite=None` cookie without `Secure` outright.

**So `withCredentials: true` — the only thing rev. 1 named — is necessary and nowhere near sufficient. The only viable transport is same-origin.**

- `proxy.conf.json` → forward `/api` and `/hubs` to Kestrel.
- `angular.json` → `proxyConfig` on the serve target.
- **Every API call in Angular is a same-origin relative path** (`/api/...`), **never** `http://localhost:5xxx/...`.

⚠️ **Production same-origin is still unanswered**, and it ties straight into the open *"no host machine exists"* blocker. The dev loop will work. **How this reaches a colleague's browser is a separate, unsolved problem.** Do not let a wave invent an answer.

## W1b — Stand up `ng test`. This is a decision, and it is being made on purpose.

**Measured:** no `test` script, no `"test"` target in `angular.json`, no `karma.conf.js`, no `tsconfig.spec.json`, **zero `.spec.ts` files**. `tsconfig.app.json` is `"files": ["src/main.ts"]` — it would not even *collect* a spec. **You cannot run one Angular test today.**

Rev. 1's entire gate was OT-1..OT-7, *"the user's to click."* After 809 automated backend tests, shipping the frontend with **zero** automated coverage is a real regression — and rev. 1 made that choice **by omission**, which is the one way it must not be made.

**The three things most worth testing here need no browser at all** — they are pure functions:
1. the `(taskId, isoDate)` cell key,
2. **the Smart Fill merge**,
3. the interceptor's 401→`/login` and 409→dialog branches.

**A unit test on (2) is precisely what would have caught the "replace the grid" bug this very review found.** That is the argument, and it is sufficient.

Karma + Jasmine, ~6 devDeps, a `test` target, `tsconfig.spec.json`. Nothing else contends for those files.

# Wave 4 — Log Work, for real. ONE agent. `[opus]`

**Owns:** `src/app/pages/log-work/*` · `src/app/core/conflict-dialog/*` (new) · `src/app/core/realtime.service.ts` (new)

- **Re-key `HoursMap`** from `` `${groupIndex}-${taskIndex}-${dayIndex}` `` to **`` `${taskId}-${isoDate}` ``**. Array indices *are* the bug; ids are the fix. **This is a pure function — unit-test it** (W1b stood up `ng test` for exactly this).
- **`WEEK_DAYS` is a hard-coded 5-day literal** (`worklog.service.ts:36-40`), and **`prevWeek()`/`nextWeek()` are `toast.show('Loaded previous week')` — pure theatre.** Derive the day axis from the selected Monday and make week navigation actually re-fetch. Rev. 1 never mentioned the day axis at all.
- **`expectedVersion` is `long?` and `null` is MEANINGFUL, not "unknown".** `null` asserts *"I believe this cell is empty."* Send `null` **only** for a cell with no existing log; for a cell that has hours, send **its actual `rowVersion`**. **Never send `0` as a stand-in for "I don't know"** — under the five-case table that is a *different assertion*, and it will either 409 spuriously or silently overwrite.
- **Store the `rowVersion` the write RETURNS.** Never re-read it. A read-back is racy: another client can write in between, and you would then hold *their* version with *your* data and silently overwrite them on your next save.
- **400 → show the message.** The 8h cap / holiday / weekend guards return `400 + { error }`. They are a **return value**, not an exception. (The grid has **no weekend columns**, so the weekend guard is unreachable from this UI — handle it defensively, don't burn UAT on it.)
- **409 → conflict dialog.** Body: `{ table, id, deleted, detail, message }`. **`detail` is the ONLY thing that identifies WHICH cell** — a timesheet cell has no id (its key is the natural triple), so `id` is `0`. Re-fetch, show both values, let the user choose. There is deliberately no `current` in the body (recorded deviation — the client re-fetches; one round-trip on a rare path, and not racy).
- **404 is reachable and rev. 1 never mentioned it.** `PUT /api/timesheet/cell` returns 404 when the task was deleted or is not in your team. Handle it: *"This task is no longer available"* + re-fetch. Same on `DELETE /cell` and both smartfill routes.
- 🔴 **Smart Fill returns a FLAT `TimeLogDto[]` — NOT a week grid.** It is `GetByUserAndRangeAsync(userId, min(filledDates), max(filledDates))`, so it spans **only the dates that were filled**, ungrouped. **MERGE it into the cell map by `(taskId, workDate)`** — patch `hours` + `rowVersion` per returned row.
  **Never *replace* grid state from it:** fill Wed–Thu only and a replace **wipes Mon/Tue/Fri off the screen**, and a flat list cannot reconstruct the backlog grouping anyway. (Rev. 1 said "returns the refreshed grid… replace state from it." **Both halves were wrong.**)
  **And ignoring it is also wrong:** your next inline edit then sends a stale version and **409s against your own Smart Fill, on the happy path, every time.** **Unit-test this merge** — it is the seam most likely to silently lose data.
- **SignalR** at `/hubs/data`, `withAutomaticReconnect`. Send the connection id as the **`X-Connection-Id` header on every mutating request** — that is how the server excludes you from your own echo. Without it you receive your own change, re-fetch, and **clobber the conflict dialog the 409 just raised.**

- [ ] **Verify (two browsers — this IS the milestone):** A types 4h → B sees it without reloading, and **A gets no echo** · both open the same cell, A saves, **B gets the conflict dialog** and A's hours survive · **editing a cell that already has hours just works** (this is what W0 exists for) · 40h on a holiday → the message, nothing written · Smart Fill, then immediately edit a filled cell → **200, not 409**, and **Mon/Tue/Fri are still on screen.**
