# M8.4 — Angular shell + Log Work

> **REQUIRED SUB-SKILL:** `superpowers:subagent-driven-development`.

**Goal:** The first slice the user can actually click. A browser, a login, and a working timesheet against the real API.

**Base:** M8.3 complete — **80 routes, 809 tests green**, OpenAPI exposed at `/swagger/v1/swagger.json` (anonymously, deliberately, so codegen can read it).

**Stack:** Angular 17.3 standalone components + signals, already vendored at `src/timesheet-web/`. Node 24 / npm 11.

---

## What is actually there, and what is not

The design bundle is **real and good** — 9 standalone components, signals, `@if`/`@for`, a `ThemeService` with light/dark CSS custom properties. **The visual layer needs almost nothing.**

**The data layer does not exist.** `worklog.service.ts` has **20 methods and every one is `return of(...)`** — an honest empty shell. And the models are a mockup's models, not a wire contract:

```typescript
export interface User { name: string; active: boolean; }        // no id
export interface Backlog { code: string; project: string; ... } // no id
export interface LogGroup { tasks: string[]; ... }              // tasks are STRINGS
/** hours keyed by `${groupIndex}-${taskIndex}-${dayIndex}` */
export type HoursMap = Record<string, string>;                  // ARRAY INDICES
/** per-task progress keyed by `${cardIndex}-${taskIndex}` */
export type ProgressMap = Record<string, number>;
```

**Every entity is identified by its position in an array, or by a display string.** Three consequences, and they are not stylistic:

1. **There is nowhere to put `rowVersion`.** The concurrency mechanism M8.2 spent five waves building is unreachable from this model layer.
2. **There is no id, so there is no way to address a row on the server.**
3. **One sort, filter or reorder and every key is wrong** — hours land on the wrong task, silently.

This is exactly the class of bug this project has been bitten by three times: a representation that cannot express the thing it must express, so the error is invisible until it is data loss.

**Therefore: `src/app/api/` is GENERATED from OpenAPI and never hand-edited.** That is the whole reason Wave 1b added Swashbuckle. Hand-maintaining these types across the wire is rebuilding the bug on purpose.

**Good news, measured:** only **3 of 9** component files use index-keys. The other six bind to display data and port largely unchanged.

# Wave 2 — Login, and the two interceptors everything else depends on. ONE agent. `[sonnet]`

**W2-T1 — Auth shell.**

**Owns:** `src/app/pages/login/*` (new) · `src/app/core/auth.interceptor.ts` · `auth.guard.ts` · `auth.service.ts` · routing

- `POST /api/auth/login { username, password, rememberMe }`. `rememberMe` **is** the "stay logged in" mechanism — it sets `IsPersistent` on the cookie.
- **`withCredentials: true` on every request.** The session is a cookie; without this Angular sends nothing and every call 401s.
- **401 interceptor → redirect to `/login`.** The API returns a real 401 (Wave 1b overrode ASP.NET's default, which was a **302 to a page that does not exist** — Angular would have seen a 404).
- **Route guard** on everything except `/login`.

- [ ] **Verify:** log in → land on Log Work · wrong password → an error, not a redirect loop · **restart the API process → still logged in** (that is the Data Protection key ring; it is the one bug invisible until production) · log out → `/login`.

# Wave 3 — Log Work against the real API. ONE agent. `[opus]`

**W3-T1 — The week grid, for real.**

**Owns:** `src/app/pages/log-work/*` · `src/app/core/conflict-dialog/*` (new) · `src/app/core/realtime.service.ts` (new)

- Re-key `HoursMap` from `${groupIndex}-${taskIndex}-${dayIndex}` to **`${taskId}-${isoDate}`**. Array indices are the bug; ids are the fix.
- Every cell carries its **`rowVersion`**. `PUT /api/timesheet/cell` sends it as `expectedVersion` and **stores the `rowVersion` the response returns** — never re-reads it (a read-back is racy: another client can write in between, and you would then hold *their* version with *your* data and silently overwrite them on your next save).
- **400 → show the message.** The 8h/day cap, the holiday guard and the weekend guard all come back as `400 + { error }`. They are a **return value**, not an exception, and an endpoint that ignores them returns 200 on a rejected write.
- **409 → conflict dialog.** The body is `{ table, id, deleted, detail, message }`. **`detail` is the only thing that identifies WHICH cell** — a timesheet cell has no id (its key is the natural triple), so `id` is `0`. Re-fetch, show both values, let the user choose. There is deliberately **no `current` in the body**; the client re-fetches (recorded deviation — one round-trip on a rare path, and not racy).
- **Smart Fill returns the refreshed grid with new `rowVersion`s in its 200 body. Replace state from it.** If you ignore it, the user's very next inline edit sends a stale version and gets *"someone else changed this"* — **about their own Smart Fill result, on the happy path, every single time.**
- **SignalR** at `/hubs/data`. Send the connection id as the **`X-Connection-Id` header on every mutating request** — that is how the server excludes you from your own echo. Without it you receive your own change, re-fetch, and **clobber the conflict dialog the 409 just raised**.
- **Group membership is lost on every reconnect** — the server rejoins in `OnConnectedAsync`, but the client must reconnect (`withAutomaticReconnect`). Cross-user sync appears to work perfectly until the first Wi-Fi blip.

- [ ] **Verify (two browsers, and this IS the milestone):** A types 4h → B sees it without reloading, and **A does not get an echo** · both open the same cell, A saves, **B gets the conflict dialog** and A's hours survive · 40h on a holiday → the message, and nothing is written · Smart Fill, then immediately edit a filled cell → **saves, no 409**.
