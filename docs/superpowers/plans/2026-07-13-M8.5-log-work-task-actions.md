### The reorder bug, because it is the one worth remembering

Rev. 1 "optimised" the reorder: write only the rows between the drag's two positions, at absolute index `lo + i`. That is correct **only if** the group's active `order_index` values are already exactly `0..n-1`.

**They are not — because of the other feature in this same milestone.** `SetActiveAsync` (`TaskRepository.cs:123`) sets `is_active = 0` and **leaves `order_index` untouched**; the read is `WHERE is_active = 1 ORDER BY order_index`.

```
A,B,C,D at order_index 0,1,2,3
Drag A to the trash        -> actives are B(1), C(2), D(3)  -> array positions 0,1,2
Drag C (position 1) to 2   -> reorderPlan(rows,1,2): lo=1,hi=2 -> writes D->1, C->2

DB:  B=1,  D=1,  C=2      <-- B and D TIE
ORDER BY order_index with a tie -> arbitrary. The order silently scrambles on reload.
```

**WPF rewrites every row** (`TimesheetViewModel.cs:421` — `for (var i = 0; i < ids.Count; i++) await SetOrderAsync(ids[i], i)`), which is **self-healing**: it renormalises gaps and ties on every drag. The spec already accepts N writes per drag (§8: *"WPF has the identical shape"*). Rev. 2 does the same.

And rev. 1's second test — `it('touches only the rows between the two positions')` — **asserted the buggy behaviour**, so it would have actively resisted the fix. **A test that agrees with a broken implementation is worse than no test.** It is gone.

### Why this wave exists

`BacklogEndpoints.cs` has **zero `.Produces<T>()`**. Every route returns `IResult` via `Results.Ok(x)`, and **ApiExplorer cannot infer a response schema from that** — so OpenAPI describes all 17 as `{"200": {"description": "OK"}}`: no content, no schema. The generated client therefore does not contain them, and `ng-openapi-gen.json` says why, in a comment M8.4/W2 left deliberately:

> *"A route joins this list when, and only when, its C# gains a `.Produces<T>()`. **Annotate first, then generate. Never widen this list to 'fix' a missing method.**"*

Widening the tag list instead emits a method typed `void` for an endpoint that **does** return data — **a client that lies, committed to the repo, that a later wave would reasonably trust.**

using System.Net;
using System.Text.Json;
using Xunit;                       // NOT covered by ImplicitUsings -- every test file here has it explicitly

namespace TimesheetApp.ApiTests;

/// <summary>
/// The generated TypeScript client can only contain a route whose C# DECLARES a response type. A route
/// that returns IResult via Results.Ok(x) and says nothing else is described by OpenAPI as an empty 200,
/// and codegen then emits a method typed `void` for an endpoint that returns data. That failure is
/// SILENT -- codegen SUCCEEDS. This test is what makes the rule enforceable.
/// </summary>
public sealed class OpenApiContractTests : IAsyncLifetime
{
    private ApiFactory _factory = null!;
    private JsonElement _paths;

    public async Task InitializeAsync()
    {
        _factory = new ApiFactory();
        var res = await _factory.AnonymousClient().GetAsync("/swagger/v1/swagger.json");  // deliberately anonymous
        Assert.Equal(HttpStatusCode.OK, res.StatusCode);
        _paths = JsonDocument.Parse(await res.Content.ReadAsStringAsync())
                             .RootElement.GetProperty("paths").Clone();
    }

    public Task DisposeAsync() { _factory.Dispose(); return Task.CompletedTask; }

    [Theory]
    [InlineData("/api/backlogs/{id}", "get", "200")]
    [InlineData("/api/backlogs/{id}", "put", "200")]
    [InlineData("/api/tasks", "post", "200")]
    public void Route_declares_a_response_SCHEMA_not_just_a_status(string path, string verb, string status)
    {
        var response = _paths.GetProperty(path).GetProperty(verb)
                             .GetProperty("responses").GetProperty(status);

        // `description: "OK"` with no `content` IS the silent failure. Assert the schema exists.
        Assert.True(response.TryGetProperty("content", out var content),
            $"{verb.ToUpperInvariant()} {path} declares {status} with NO response body. " +
            "Codegen will emit a method typed `void` for an endpoint that returns data.");
        Assert.True(content.GetProperty("application/json").TryGetProperty("schema", out _));
    }

    [Theory]
    [InlineData("/api/tasks/{id}/active", "put")]
    [InlineData("/api/tasks/{id}/order", "put")]
    public void Bump_only_route_declares_204(string path, string verb)
    {
        var responses = _paths.GetProperty(path).GetProperty(verb).GetProperty("responses");
        Assert.True(responses.TryGetProperty("204", out _),
            $"{verb.ToUpperInvariant()} {path} must declare 204 -- it returns Results.NoContent().");
    }

    [Theory]
    [InlineData("/api/backlogs/{id}", "get", "Backlogs")]
    [InlineData("/api/backlogs/{id}", "put", "Backlogs")]
    [InlineData("/api/tasks", "post", "Tasks")]
    [InlineData("/api/tasks/{id}/active", "put", "Tasks")]
    [InlineData("/api/tasks/{id}/order", "put", "Tasks")]
    public void Route_is_TAGGED_so_ng_openapi_gen_can_include_it(string path, string verb, string tag)
    {
        var tags = _paths.GetProperty(path).GetProperty(verb).GetProperty("tags")
                         .EnumerateArray().Select(t => t.GetString()).ToList();
        Assert.Contains(tag, tags);
    }

    // ng-openapi-gen names each generated FUNCTION from the operationId, which in a minimal API comes
    // from .WithName(...). Omit it and Swashbuckle invents one -- so the function gets a name nobody
    // chose, and it churns whenever an unrelated route is added.
    [Theory]

    [InlineData("/api/backlogs/{id}", "put", "BacklogUpdate")]
    [InlineData("/api/tasks", "post", "TaskCreate")]
    [InlineData("/api/tasks/{id}/active", "put", "TaskSetActive")]
    [InlineData("/api/tasks/{id}/order", "put", "TaskSetOrder")]
    public void Route_has_the_operationId_the_generated_function_is_named_from(

    {
        var op = _paths.GetProperty(path).GetProperty(verb);
        Assert.True(op.TryGetProperty("operationId", out var id),
            $"{verb.ToUpperInvariant()} {path} has no operationId -- add .WithName(\"{operationId}\").");
        Assert.Equal(operationId, id.GetString());

/**
 * The month a "Move to next month" lands in.
 *
 * WPF uses `backlog.PeriodMonth ?? SelectedMonth`, where SelectedMonth is its month FILTER. The web Log
 * Work grid is a WEEK view and has no month filter, so the fallback is the month of the Monday currently
 * on screen. (User's explicit choice.)
 */
export function nextMonthFrom(periodMonth: string | null, displayedMonday: string): string {
  const base = periodMonth ?? displayedMonday.slice(0, 7);
  const [y, m] = base.split('-').map(Number);
  const next = m === 12 ? { y: y + 1, m: 1 } : { y, m: m + 1 };
  return `${next.y}-${String(next.m).padStart(2, '0')}`;
}

`reorder.ts`:

```typescript
import { TaskRow } from './grid-state';

/** One row's new position on the server. */
export interface OrderWrite {
  readonly taskId: number;
  readonly orderIndex: number;
}

/**
 * A drag, expressed as the new `orderIndex` of EVERY row in the group.
 *
 * 🔴 Every row, not just the ones that moved -- and that is not belt-and-braces, it is the fix for a real
 * corruption. `SetActiveAsync` soft-deletes by setting `is_active = 0` and LEAVES `order_index` alone,
 * while the read is `WHERE is_active = 1 ORDER BY order_index`. So one delete leaves the survivors at
 * 1,2,3 -- a GAP. A reorder that writes only the displaced rows, at absolute index lo+i, then produces a
 * TIE, and `ORDER BY order_index` with a tie is arbitrary: the order silently scrambles on the next
 * reload. Rewriting every row renormalises the gap on every drag. This is exactly what WPF does
 * (`TimesheetViewModel.cs:421`).
 *
 * The writes are BUMP-ONLY (`PUT /api/tasks/{id}/order`, no expectedVersion). That is a recorded M8.2
 * decision, not an oversight: SetOrderAsync runs once per row, so a checked variant would 409-storm on
 * an ordinary drag. Do NOT add a version to these.
 */
export function reorderPlan(rows: readonly TaskRow[], from: number, to: number): OrderWrite[] {
  if (from === to) return [];

  const next = [...rows];
  const [moved] = next.splice(from, 1);
  next.splice(to, 0, moved);

  return next.map((row, i) => ({ taskId: row.taskId, orderIndex: i }));
}
```

### 🔴 Three conventions you will get wrong if you guess

**1. `mutatingHttp`, not `http`.** `WorklogService` holds **two** `HttpClient`s. Reads use `this.http`; **every mutation uses `this.mutatingHttp`** — a `ConnectionIdHttpClient` that attaches **`X-Connection-Id`**, which is how the server **excludes you from your own SignalR echo**. Send a write on the plain client and you receive your own change back, re-fetch, and clobber your own screen. **All of M8.5 is mutations.**

**2. There is no dialog service.** The pattern is a **signal plus `@if`**, exactly as the existing conflict dialog is rendered:
```html
@if (conflict(); as c) { <app-conflict-dialog [conflict]="c" ... /> }
```

**3. Import from the barrel.** The eight existing functions come from `'../api/functions'`. Match that; do not deep-import.
