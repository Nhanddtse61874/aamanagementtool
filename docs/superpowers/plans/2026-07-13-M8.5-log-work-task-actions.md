### Why this wave exists at all

`BacklogEndpoints.cs` has **zero `.Produces<T>()`**. Every route returns `IResult` via `Results.Ok(x)`, and **ApiExplorer cannot infer a response schema from that**. So OpenAPI describes all 17 as:

```json
"/api/backlogs/{id}": { "get": { "responses": { "200": { "description": "OK" } } } }
```

No content. No schema. The generated client therefore **does not contain them**, and `ng-openapi-gen.json` says why, in a comment M8.4/W2 left deliberately:

> *"A route joins this list when, and only when, its C# gains a `.Produces<T>()`. **Annotate first, then generate. Never widen this list to 'fix' a missing method.**"*

Widening the tag list instead would emit a method typed `void` for an endpoint that **does** return data — **a client that lies, committed to the repo, that a later wave would reasonably trust.**

```csharp
/// <summary>
/// The generated TypeScript client can only contain a route whose C# DECLARES a response type.
/// A route that returns IResult via Results.Ok(x) and says nothing else is described by OpenAPI as
/// an empty 200 -- and codegen then emits a method typed `void` for an endpoint that returns data.

/// </summary>
public sealed class OpenApiContractTests : IAsyncLifetime
{
    private ApiFactory _factory = null!;
    private JsonElement _paths;

    public async Task InitializeAsync()
    {
        _factory = new ApiFactory();
        var client = _factory.AnonymousClient();          // /swagger is deliberately anonymous
        var res = await client.GetAsync("/swagger/v1/swagger.json");
        Assert.Equal(HttpStatusCode.OK, res.StatusCode);
        var doc = JsonDocument.Parse(await res.Content.ReadAsStringAsync());
        _paths = doc.RootElement.GetProperty("paths").Clone();
    }

    public Task DisposeAsync() { _factory.Dispose(); return Task.CompletedTask; }

    [InlineData("/api/backlogs/{id}", "put", "200")]

    {

                             .GetProperty("responses").GetProperty(status);

        // `description: "OK"` with no `content` is exactly the silent failure. Assert the schema exists.
        Assert.True(response.TryGetProperty("content", out var content),
```

### 🔴 `.WithName(...)` is NOT optional. It becomes the generated function's NAME.

`ng-openapi-gen` emits **tree-shakable functions**, not services, and it names each one from the OpenAPI **`operationId`** — which in a minimal API comes from **`.WithName(...)`**:

```
TimesheetEndpoints.cs:74   .WithName("TimesheetWeek")
                              ↓  operationId
src/app/api/fn/timesheet/timesheet-week.ts   →   export function timesheetWeek(...)
```

**Omit `.WithName` and Swashbuckle invents an operationId**, so the generated function gets a name nobody chose and which will churn the next time an unrelated route is added. Every existing generated function in this repo has a `.WithName`. Match it.

```typescript
import { fromIsoDate } from './week';

/**
 * The month a "Move to next month" lands in.
 *
 * WPF uses `backlog.PeriodMonth ?? SelectedMonth`, where SelectedMonth is its month FILTER. The web
 * Log Work grid is a WEEK view and has no month filter, so the fallback is the month of the Monday
 * currently on screen. (User's explicit choice.)
 *
 * @param periodMonth       the backlog's own "yyyy-MM", or null if it has never been assigned one
 * @param displayedMonday   the Monday the grid is currently showing, as "yyyy-MM-dd"
 */
export function nextMonthFrom(periodMonth: string | null, displayedMonday: string): string {
  const base = periodMonth ?? displayedMonday.slice(0, 7);
  const [y, m] = base.split('-').map(Number);
  const next = m === 12 ? { y: y + 1, m: 1 } : { y, m: m + 1 };
  return `${next.y}-${String(next.m).padStart(2, '0')}`;
}

/**
 * The hidden DEFAULT backlog holds the recurring default tasks and must appear in EVERY month -- so it
 * must never belong to one. WPF guards this twice: `CanMoveMonth => BacklogCode != "DEFAULT"` hides the
 * button, AND MoveMonthAsync refuses outright. Keep both guards. Defence in depth here is deliberate.
 */
export function canMoveMonth(backlogCode: string): boolean {
  return backlogCode !== 'DEFAULT';
}
```

`reorder.ts`:

```typescript
import { TaskRow } from './grid-state';

/** One row's new position on the server. */
export interface OrderWrite {
  readonly taskId: number;
  readonly orderIndex: number;
}

/**
 * Which rows a drag actually moved, and what their new `orderIndex` is.
 *
 * A drag does not move one row -- it displaces every row between the two positions. Writing only the
 * dragged row leaves the others with stale indices, and the order silently reverts on the next reload.
 *
 * The writes are BUMP-ONLY (`PUT /api/tasks/{id}/order`, no expectedVersion). That is a recorded M8.2
 * decision, not an oversight: SetOrderAsync runs once per row, so a checked variant would 409-storm on
 * an ordinary drag. Do not add a version to these.
 */
export function reorderPlan(rows: readonly TaskRow[], from: number, to: number): OrderWrite[] {
  if (from === to) return [];

  const next = [...rows];
  const [moved] = next.splice(from, 1);
  next.splice(to, 0, moved);

  const lo = Math.min(from, to);
  const hi = Math.max(from, to);

  return next
    .slice(lo, hi + 1)
    .map((row, i) => ({ taskId: row.taskId, orderIndex: lo + i }));
}
```

**1. `mutatingHttp`, not `http`.** `WorklogService` holds **two** `HttpClient`s. Reads go through `this.http`; **every mutation goes through `this.mutatingHttp`** — a `ConnectionIdHttpClient` that attaches the **`X-Connection-Id`** header. That header is how the server **excludes you from your own SignalR echo.** Send a write on the plain `http` and you receive your own change back, re-fetch, and clobber your own screen. **All three M8.5 features are mutations.**

**2. There is no dialog service.** The established pattern is a **signal plus `@if`** in the template — see the existing conflict dialog:
```html
@if (conflict(); as c) { <app-conflict-dialog [conflict]="c" ... /> }
```
**Do not invent a `MatDialog`-style `dialog.open()`.** Follow the pattern that is already there.

**3. The refresh Subject is called `refresh`, not `refresh$`.** It is `private readonly refresh = new Subject<void>()`, and W4 left a comment on it explaining why it is a `Subject` and not a signal (`toObservable()` replays a signal's current value on subscribe, so the week was fetched twice per load and a 404's re-fetch silently never happened). **A refresh is an EVENT, not state.**

- [ ] **Step 2: Map the DTO onto the request body**

`PUT /api/backlogs/{id}` takes **every field**, not a patch — so an omitted field is written as `null`. M8.3 already recorded what that costs: *"a DTO that merely omits `teamId` sets `team_id = NULL` and the backlog drops out of every team, invisible to everyone, permanently, while every test still passes."* `BacklogUpdateRequest` has **no `teamId` property at all** — structurally absent, so that particular bug cannot compile back in. **Every other field still must be carried across unchanged.**

```typescript
import { BacklogDto, BacklogUpdateRequest } from '../../api/models';

/**
 * PUT /api/backlogs/{id} replaces the WHOLE record -- an omitted field is written as null, not left
 * alone. So a Move must carry every other field across untouched. Only `periodMonth` and
 * `expectedVersion` are ours to set.
 */
export function toUpdateRequest(b: BacklogDto, periodMonth: string): BacklogUpdateRequest {
  return {
    backlogCode: b.backlogCode!,
    project: b.project!,
    startDate: b.startDate ?? null,
    endDate: b.endDate ?? null,
    periodMonth,                                   // <-- the ONLY field this action changes
    type: b.type ?? null,

    note: b.note ?? null,
```

```typescript
  async onMoveMonth(group: Group): Promise<void> {
    if (!canMoveMonth(group.code)) return;        // the SECOND guard. Both, deliberately.

    // The GET is mandatory, not an optimisation: the PUT needs the full body AND an expectedVersion,
    // and WeekBacklogGroup carries neither the backlog's rowVersion nor its other fields. There is
    // nowhere else to get them.
    const b = await firstValueFrom(this.api.getBacklog(group.backlogId));
    const periodMonth = nextMonthFrom(b.periodMonth ?? null, this.monday());

    try {

    }
```

- [ ] **Step 4: The 409 does NOT reuse the cell conflict dialog**

That dialog is a **merge** between two numbers — *keep theirs / overwrite with mine*. **A moved backlog has nothing to merge.**

```typescript
  /** 409: someone else changed this ticket between the read and the write. Say so, re-fetch, stop.
   *  DO NOT silently retry -- their change may ITSELF have been a Move, and a blind retry would push the
   *  ticket TWO months forward. */
  private onMoveConflict(err: unknown): void {
    if (err instanceof HttpErrorResponse && err.status === 409) {
      this.toast.show('Someone else just changed this ticket. Reloaded — try again if you still want to move it.');
      this.refresh.next();
      return;
    }
    throw err;
  }
```
