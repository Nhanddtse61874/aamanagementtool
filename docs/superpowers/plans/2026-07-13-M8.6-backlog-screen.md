## 🔴 READ THIS FIRST — it applies to every task

Each is a bug already paid for once in this project. **They are restated verbatim inside each task that can touch them, because an agent sees its brief and nothing else.**

### R1 — The API must NEVER run against the real database

`Program.cs` defaults `DbPath` to `C:\Users\Admin\Documents\TimesheetApp\timesheet.db` — the user's **live production data** — and startup runs `DatabaseInitializer` + `TeamBootstrapService` + `AdminBootstrap` against whatever it points at.

**Pin ALL THREE seams** to a throwaway temp dir: `TimesheetApp:DbPath`, `TimesheetApp:ConfigPath`, `TimesheetApp:KeyRingPath`.
**`DbPath` alone is NOT enough.** `Program.cs:34` uses `||`, so setting `DbPath` *without* `ConfigPath` silently falls back to the **fully default production config**. *(Plan Checker verified this line. The claim is true.)*

**Proof it worked — grep the startup log for this EXACT substring:**
```
no users yet, nothing to bootstrap
```
🔴 **Grep the substring, NOT a prefixed form.** The real line (`AdminBootstrap.cs:41`) reads *"**Admin bootstrap:** no users yet, nothing to bootstrap. The first user must be created by…"* — **space, lowercase `b`**. `AdminBootstrap` (one word) is only the *logger category*, rendered on the preceding line as `info: AdminBootstrap[0]`. **Rev. 1 searched for the wrong string, which would have aborted a perfectly safe run.**

**Do NOT try to prove safety by asserting the real DB has no `-wal` sidecar.** A live SQLite database in WAL mode **always** has one. That check aborts on a false alarm.

### R2 — A running API makes the .NET gate LIE

If an API is listening on 5080 it **holds `TimesheetApp.Core.dll` open**, so `dotnet test` fails to *build* `TimesheetApp.Api` (`MSB3027: the file is locked by TimesheetApp.Api (<pid>)`) — **and still exits 0 and prints `Passed!`** for whichever assembly did build.

**This happened today.** The gate reported `Passed! … 658 … TimesheetApp.Tests.dll` while `TimesheetApp.ApiTests.dll` **never appeared at all** — 172 tests silently did not run.

**Before any .NET gate: nothing may listen on 5080. After it, BOTH lines must be present:**
```
Passed! ... TimesheetApp.Tests.dll        <-- 658, unchanged
Passed! ... TimesheetApp.ApiTests.dll     <-- ABSENT = THE GATE DID NOT RUN
```

### R3 — `.Produces<T>()` is the only thing that makes a route exist to the client

ApiExplorer **cannot infer a response schema from a handler returning `IResult` via `Results.Ok(x)`**. An unannotated route is described as `"200": {"description":"OK"}` — no content — and `ng-openapi-gen` **omits it entirely**.

> *"A route joins this list when, and only when, its C# gains a `.Produces<T>()`. **Annotate first, then generate. Never widen this list to 'fix' a missing method.**"*

🔴 **And a route that IS annotated but ADMIN-GATED is a different lie: correctly typed, and it 403s for every ordinary user.** Check `.RequireAuthorization(AuthSetup.AdminPolicy)` before you annotate anything. **`/api/users/all` and `/api/pca-contacts/all` are admin-only and are NOT part of this milestone.**

### R4 — A green build proves nothing about wiring

M8.5 Task 7: an agent mutated the template to the *broken* form and re-ran. **Six of seven `onTrash` tests stayed GREEN against a completely dead feature.** Only the test asserting the *wiring* went red. **A test that calls a handler directly cannot tell you the handler is reachable.**

### R5 — Every `async` handler bound to a template output needs a `catch` that never re-throws

Anything that escapes is an **unhandled promise rejection**: it surfaces in the console and **nowhere the user can see**. *(Shipped three times during M8.5 — in a plan that diagnoses the exact hazard.)*

### R6 — Never `!`, never `0`, on a `rowVersion`

**`requireRowVersion(rowVersion: number | undefined): number` is exported from `src/app/services/worklog.service.ts:343`** — not a util module. `move-month.ts:2` imports it as `from '../../services/worklog.service'`. It **throws** rather than defaulting. Use it.
