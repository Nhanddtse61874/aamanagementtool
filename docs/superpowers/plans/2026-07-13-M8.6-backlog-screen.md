## 🔴 READ THIS FIRST — it applies to every task in this plan

These are not preamble. Each is a bug that has already been paid for once in this project, and each will be re-paid by anyone who skips this section. **They are restated verbatim inside each task that can touch them, because an agent sees its brief and nothing else.**

### R1 — The API must NEVER run against the real database

`Program.cs` defaults `DbPath` to `C:\Users\Admin\Documents\TimesheetApp\timesheet.db` — the user's **live production data** — and startup runs `DatabaseInitializer` + `TeamBootstrapService` + `AdminBootstrap` against whatever it points at.

**Pin ALL THREE seams** to a throwaway temp directory:
```
TimesheetApp:DbPath        TimesheetApp:ConfigPath        TimesheetApp:KeyRingPath
```
**`DbPath` alone is NOT enough.** `Program.cs` uses `||`, so setting `DbPath` *without* `ConfigPath` silently falls back to the **fully default production config**.

**Proof it worked:** the startup log must read `AdminBootstrap: no users yet, nothing to bootstrap`. That line is unforgeable evidence the process opened an **empty** database.

**Do NOT try to prove safety by asserting the real DB has no `-wal` sidecar.** A live SQLite database in WAL mode **always** has one, and a checkpoint changes the file hash. That check aborts on a false alarm.

### R2 — The API process locks `TimesheetApp.Core.dll`

If an API is running (port 5080), `dotnet build`/`dotnet test` of `TimesheetApp.Api` **fails** with `MSB3027: the file is locked by TimesheetApp.Api (<pid>)` — **and `dotnet test` still exits 0 and prints `Passed!`** for whichever assembly *did* build. A build failure in one project plus a pass in another reads as success.

**Before any .NET gate: confirm nothing is listening on 5080.** After the run, confirm **both** assemblies reported:
```
Passed! ... TimesheetApp.Tests.dll
Passed! ... TimesheetApp.ApiTests.dll     <-- if this line is absent, THE GATE DID NOT RUN
```

### R3 — `.Produces<T>()` is the only thing that makes a route exist to the client

ApiExplorer **cannot infer a response schema from a handler returning `IResult` via `Results.Ok(x)`**. An unannotated route is described as `"200": {"description":"OK"}` — no content — and `ng-openapi-gen` **omits it entirely**.

From `ng-openapi-gen.json`, and it is load-bearing:
> *"A route joins this list when, and only when, its C# gains a `.Produces<T>()`. **Annotate first, then generate. Never widen this list to 'fix' a missing method.**"*

`OpenApiContractTests.cs` turns this into a build failure. **Every route this plan adds gets its `[InlineData]` rows there.**

### R4 — A green build proves nothing about wiring

M8.5 Task 7: an agent mutated the template to the *broken* form and re-ran the suite. **Six of seven `onTrash` tests stayed GREEN against a completely dead feature.** Only the test that asserted the *wiring* went red.

**A test that calls a handler directly cannot tell you the handler is reachable.** Where a control's behaviour depends on wiring (a dialog opening, a drop list registering, a form binding), assert the wiring, not just the handler.

### R5 — Every `async` handler bound to a template output needs a `catch` that never re-throws

Anything that escapes is an **unhandled promise rejection**: it surfaces in the console and **nowhere the user can see**. The row simply never appears, with no error, and the user sits there clicking the button again.

*(This bug was shipped three times during M8.5 — in a plan that diagnoses the exact hazard.)*

### R6 — Never `!`, never `0`, on a `rowVersion`

`requireRowVersion(dto.rowVersion)` **throws** rather than defaulting. Use it. `rowVersion!` reintroduces, through TypeScript's front door, the exact bug the whole mechanism exists to prevent.

### 1a — Two new DTOs (`Contracts/Dtos.cs`)

```csharp
/// The LIST shape. Deliberately NOT BacklogDto: BacklogDto is the EDITOR's shape (what GET /{id} and
/// POST return), and putting a TaskCount on it would force every single-backlog read to compute a
/// number nobody uses.
///
/// No AssigneeName: the client already fetches the full user list for the editor's dropdown, so it
/// resolves the name itself — and resolving from /api/users/all (which includes DEACTIVATED users) is
/// precisely what stops a departed assignee's name from vanishing off the grid.
///
/// No RowVersion: the Edit button does a fresh GET /{id}, which returns the authoritative version. A
/// version carried on a list row is STALE BY CONSTRUCTION, and a stale version is exactly the thing
/// that gets narrowed with a `!` and silently overwrites somebody.
public sealed record BacklogListItemDto(
    int Id, string BacklogCode, string Project, int TaskCount,
    string? PeriodMonth, string? Type, int? AssigneeUserId);

/// Projected from BacklogAuditEntry (Entities.cs:69-72). BacklogId is dropped (it is the path param)
/// and ChangedByUserId is dropped (the panel renders the NAME — and the repository audits by name
/// precisely so a deleted user's history still reads).
public sealed record BacklogAuditDto(
    int Id, string Field, string? OldValue, string? NewValue,
    string? ChangedByName, DateTimeOffset ChangedAt, string? Note);
```

### 1b — `GET /api/backlogs` (`:38`) — annotate + return the list shape

The `TASKS` column needs a count of **active** tasks per backlog. **Use the batched repository method that already exists** — `ITaskRepository.GetActiveByBacklogsAsync(backlogIds)`, a single `IN` query. WPF added it as an explicit N+1 fix. **Do not add a repository method. Do not loop.**

```csharp
api.MapGet("/api/backlogs", async (
        HttpContext http, string? term, IBacklogRepository backlogs, ITaskRepository tasks, IClientContext ctx) =>
    {
        var rows = await backlogs.SearchAsync(term, EffectiveTeamIds(http, ctx));
        var counts = await tasks.GetActiveByBacklogsAsync(rows.Select(b => b.Id).ToList());
        return Results.Ok(rows.Select(b => new BacklogListItemDto(
            b.Id, b.BacklogCode, b.Project,
            counts.TryGetValue(b.Id, out var t) ? t.Count : 0,
            b.PeriodMonth, b.Type, b.AssigneeUserId)).ToList());
    })
    .WithName("BacklogList").WithTags("Backlogs")
    .Produces<List<BacklogListItemDto>>();
```

🔴 **`EffectiveTeamIds` must stay exactly as it is** (`:438-450`). It reads `http.Request.Query` **by hand** and never passes `null`, because:
- `[FromQuery] int[]?` **never binds to `null`** — both "key absent" and "key present but empty" become an **empty array**. That was measured, not assumed.
- `SearchAsync(term, null)` means **EVERY TEAM, UNFILTERED** (`IBacklogRepository.cs:8-9`).

Together those two facts are a data leak. **Do not "simplify" it into a `[FromQuery]` parameter.**

🔴 **Verify `GetActiveByBacklogsAsync`'s exact return type before you write the projection above.** Read `ITaskRepository`. If it is not `IReadOnlyDictionary<int, IReadOnlyList<TaskItem>>`, adapt — but **do not** replace it with an N+1 loop.

### 1c — `POST /api/backlogs` (`:62`) — annotate + refuse the orphan

**The bug.** `:73` reads:
```csharp
int? teamId = currentTeam.ActiveTeamId > 0 ? currentTeam.ActiveTeamId : null;
```
`ApiCurrentTeamService.InitializeAsync` resolves `ActiveTeamId` to the persisted team if still a membership, else the first available, **else `0`**. So a caller in **zero teams** inserts `team_id = NULL`, and then:
- `GET /api/backlogs/{id}` 404s it — `AuthorizedBacklogAsync` rejects `TeamId is not { } teamId` **for everyone, admins included**;
- `GET /api/backlogs` never returns it — `team_id IN (…)` cannot match `NULL`;
- no SignalR fires (`if (teamId is { } tid)`, `:89`);
- **and the handler returns `200 OK` with the full `BacklogDto`.**

The UI renders it, then loses it on the next refresh. **Permanently.** This is the same end state M8.3 already paid for through `UPDATE` — reached this time through `INSERT`.

**The fix — refuse, immediately after the existing code/project validation:**
```csharp
int? teamId = currentTeam.ActiveTeamId > 0 ? currentTeam.ActiveTeamId : null;
if (teamId is null)
    return Results.BadRequest(new ValidationBody(
        "You are not a member of any team, so this backlog would be invisible to everyone. " +
        "Ask an admin to add you to a team."));
```
Then annotate:
```csharp
.WithName("BacklogCreate").WithTags("Backlogs")
.Produces<BacklogDto>().Produces<ValidationBody>(StatusCodes.Status400BadRequest);
```

### 1d — `GET /api/backlogs/{id}/audit` — **a new route**

It does not exist. `IBacklogRepository.GetAuditAsync(int)` (`:46`) has been sitting unexposed since v2. Without this the change-history panel renders an empty box forever.

```csharp
api.MapGet("/api/backlogs/{id:int}/audit", async (
        int id, IBacklogRepository backlogs, IClientContext ctx) =>
    {
        if (await AuthorizedBacklogAsync(id, backlogs, ctx) is null) return Results.NotFound();
        var entries = await backlogs.GetAuditAsync(id);
        return Results.Ok(entries.Select(e => new BacklogAuditDto(
            e.Id, e.Field, e.OldValue, e.NewValue, e.ChangedByName, e.ChangedAt, e.Note)).ToList());
    })
    .WithName("BacklogAudit").WithTags("Backlogs")
    .Produces<List<BacklogAuditDto>>().Produces(StatusCodes.Status404NotFound);
```
`AuthorizedBacklogAsync` (`:407-424`) is the same team guard every other single-backlog route uses, so a 404 for another team's backlog comes for free. **Check its exact signature before calling it.**

### 🔴 TRAP 1 — `toUpdateRequest` MUST start from the loaded DTO, never from the form

`PUT /api/backlogs/{id}` **replaces the whole record.** `BacklogRepository.cs:147-157` writes all 15 data columns unconditionally, every call.

**And SIX fields are hidden on the edit form** — `startDate`, `endDate`, `deadlineInternal`, `deadlineExternal`, `progressPercent`, `pcaContactId` (they are create-only, because they are edited inline on the Task List). Build the request from the form and **every one of them is written as `NULL`.**

This is the same trap that once nulled a `team_id` and made a backlog invisible to everyone, permanently, **while every test passed.**

```ts
/** Start from the DTO. Override ONLY what the edit form actually shows. */
export function toUpdateRequest(dto: BacklogDto, form: EditForm): BacklogUpdateRequest {
  return {
    // --- the edit form owns these ---
    backlogCode: form.code.trim(),
    project: form.project,
    periodMonth: periodMonth(form.year, form.month),
    type: form.type,
    assigneeUserId: form.assigneeUserId,
    roughEstimateHours: form.roughEstimate,
    officialEstimateHours: form.officialEstimate,
    note: form.note,

    // --- 🔴 HIDDEN ON EDIT. Round-tripped from the DTO. Omit one and it is NULLED. ---
    startDate: dto.startDate ?? null,
    endDate: dto.endDate ?? null,
    deadlineInternal: dto.deadlineInternal ?? null,
    deadlineExternal: dto.deadlineExternal ?? null,
    progressPercent: dto.progressPercent ?? null,
    pcaContactId: dto.pcaContactId ?? null,

    expectedVersion: requireRowVersion(dto.rowVersion),   // R6: never `!`, never 0
    auditNote: null,
  };
}
```
**`teamId` is not on `BacklogUpdateRequest` at all — by design. Do not add it.** The server re-reads the entity and `with{}`-patches only the DTO's fields, which is what makes `team_id` unnullable from the wire.

### 🔴 TRAP 2 — `status` must be round-tripped, and the form does not have it

`PUT /api/tasks/{id}` writes name, order **and status** in one checked call:
```csharp
public sealed record TaskUpdateRequest(string TaskName, int OrderIndex, string Status, long ExpectedVersion);
```
```sql
UPDATE Tasks SET task_name = @TaskName, order_index = @OrderIndex, status = @Status,
                 row_version = row_version + 1
  WHERE id = @Id AND (@ExpectedVersion IS NULL OR row_version = @ExpectedVersion)
```
**The Backlog editor never shows a task's status.** Build the request from the form and `Status` binds to `null`, so the save writes **`status = NULL`** — silently wiping `Todo` / `In-process` / `Done` / `Pending`. **The entire Task List screen is built on that column.**

It is *harder* to catch than Trap 1, because the field is not on the form at all — so **forgetting it is the default outcome, not an unlikely slip.**

```ts
// The ONLY correct shape.
{ taskName: row.name.trim(),
  orderIndex: row.orderIndex,
  status: loaded.status ?? 'Todo',                      // 🔴 ROUND-TRIPPED. Never from the form.
  expectedVersion: requireRowVersion(loaded.rowVersion) }
```

**A gift:** because rename and reorder are **one** write, `PUT /api/tasks/{id}/order` is **not used by this screen at all**, and the "rename-before-reorder" ordering hazard cannot exist.

### `task-edit.ts` — `planTaskWrites(loaded, rows, backlogId)`

Returns `{ deletes: number[], inserts: TaskCreateRequest[], updates: {id, body}[] }`.

**The gap.** `SetActiveAsync` soft-deletes by setting `is_active = 0` and **leaves `order_index` untouched** (`TaskRepository.cs:123-129`), while the read is `WHERE is_active = 1 ORDER BY order_index`. Survivors sit at 1,2,3 — a **gap** — and `ORDER BY` with a **tie** is arbitrary. *This trap ate two revisions of the M8.5 plan.*

> **Reindex EVERY surviving row 0..n.** Self-healing; it cannot tie. A new task's `orderIndex` comes from the same reindex. **Only rows whose `(name, orderIndex)` actually changed produce an update** — an untouched, already-contiguous list must emit **zero** writes.

**Write order** (steps 2–4 touch **disjoint rows**, so no step can invalidate another's `expectedVersion`; each row is written at most once):
1. `PUT /api/backlogs/{id}` — checked, the **backlog's own** version
2. **deletes** — `PUT /api/tasks/{id}/active { isActive: false }` — **bump-only, NO version.** Sending one makes every delete fail.
3. **inserts** — `POST /api/tasks { backlogId, taskName, orderIndex }` — no status; the server defaults it to `Todo`
4. **updates** — `PUT /api/tasks/{id}` — checked

### Field visibility — WPF's rule, verbatim

| Create + Edit | **Create only** | **Edit only** |
|---|---|---|
| Code · Project · Assignee · Month/Year · Type · Rough estimate · Official estimate · Note · Tasks | Start date · End date · Internal deadline (PCT) · External deadline (PCA) · Progress · PCA contact | **Change history** |

The create-only fields are *operational* — they are edited inline on the Task List. 🔴 **They are still WRITTEN on edit — round-tripped from the DTO by `toUpdateRequest`. Hiding a field is not the same as omitting it.**

### The rules

🔴 **409 → toast + re-read. NOT the merge dialog.** That dialog reconciles two *numbers* in a timesheet cell — *"keep theirs / overwrite mine"*. A backlog edit has no single number to merge. Say so, re-read, stop. **Do not silently retry**: their change may itself have been a `Continue` or a `Move`, and a blind retry would apply this edit on top of it. `log-work.component.ts:606-612` is the precedent.

🔴 **Every `async` handler bound to a template output needs a `catch` that never re-throws** (R5).

🔴 **A re-entrancy guard (`saving`) on the save.** It is a **corruption guard, not politeness**: a second click's GET can land *after* the first chain's PUT, read the already-bumped version, and apply the mutation twice.

🔴 **The assignee and PCA dropdowns keep the current value even when inactive**, rendered as `Name (inactive)`, so an untouched save round-trips it. New assignments still offer only active entries. Without this, opening and saving a backlog whose assignee has since left **silently clears the assignee** — WPF bug #6.

🔴 **House button classes: `btn`, `btn-primary`, `btn-ghost`, `btn-soft`, `btn-danger`, `btn-sm`. Nothing else.** `.mini-ghost` does not exist.
