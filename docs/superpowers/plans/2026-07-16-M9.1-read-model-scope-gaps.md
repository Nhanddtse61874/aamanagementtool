## Reconciliation — where the real code differs from the spec's assumptions (READ FIRST)

The spec was written before this plan re-verified every line. Five places diverge; each is handled below, but the controller should know them up front:

1. **DR-11 — `me()` is already read, but `activeTeamId` is DISCARDED.** `DailyReportComponent.boot()` (`daily-report.component.ts:199-201`) reads `me()` **only** to set `isAdmin`. This plan captures `activeTeamId` into a new signal there and threads it into the picker filter. The filter itself is added to the **pure** `pickableBacklogs` (standup-entry.ts), whose **signature changes** — so `standup-entry.spec.ts` and the daily-report `BACKLOGS` fixture must be updated. The big class-doc block (`daily-report.component.ts:65-81`, "THE BACKLOG PICKER'S TEAM SCOPE DIVERGES FROM WPF, AND CANNOT BE MADE TO MATCH FROM HERE") and `pickableBacklogs`' own doc are now **FALSE** and are rewritten — the DR-11 equivalent of deleting the "DELIBERATELY ABSENT" comment.

2. **TL-12 — the task-list component does NOT hold a `@ViewChild` on `<app-team-filter>`**, so the spec's `showTeamColumn()` is not directly readable from the parent. **Decision:** derive "team mode" from the component's own `teamIds()` signal (`(this.teamIds()?.length ?? 0) > 1`), which is exactly `showTeamColumn`'s `_checked.size > 1` — identical except when the filter is unloaded/failed (`undefined` → 0 → **project bands**, the safe default). The id→name map is built by the parent's **own** `getTeamsActive()` call (the task-list spec already stubs `getTeamsActive`), mirroring WPF's `AvailableTeams`. This is a deliberate, functionally-equivalent deviation from "use `showTeamColumn()`".

3. **`groupRows` signature changes** from `groupRows(rows)` to `groupRows(rows, byTeam, teamNames)`. The three existing `groupRows` tests in `task-list.model.spec.ts` and the `bands` computed in `task-list.component.ts:126` are the only call sites and are both updated.

4. **SET-05 — the Settings load uses `getDefaultTasks()`** (active-only). B3 adds a new `getDefaultTasksAll()` service method + import and swaps the `forkJoin` over to it. The existing `settings.component.spec.ts` "permanent REMOVE" test (`:337`) and the HTML `warn-note` "cannot be restored" copy (`settings.component.html:227-230`) are **replaced**.

5. **`TaskListRow` / `TaskListRowDto` — `int? TeamId` is appended TRAILING** (after `Tasks`), matching the `TimeLogReportRow` precedent ("projected trailing so existing positional ctors keep compiling"). The two records are each constructed in exactly one place (`TaskListService.GetRowsAsync`, `DtoMappings.ToDto`), both edited here.

```typescript
// =====================================================================================================
// GROUPING — ADAPTIVE (TL-12).
//
// With >1 team checked the grid bands by TEAM (team name from the id->name map, alpha order, a null team
// last); otherwise by PROJECT in the fixed order. This matches WPF (`GroupByTeam = showTeam`,
// TaskListViewModel:212). The server sends only `teamId` (decision #2 in the spec); the NAME is resolved
// client-side from /api/teams, exactly as WPF resolves it from `AvailableTeams`. `byTeam` and the name map
// are supplied by the component's `bands` computed.
// =====================================================================================================

export interface Band {
  readonly key: string;
  readonly rows: readonly TaskListRowDto[];
}

/** Index of a project in the fixed order; unknown / null sorts last. */
export function projectOrder(project: string | null | undefined): number {
  const i = PROJECTS.indexOf(project ?? '');
  return i === -1 ? Number.MAX_SAFE_INTEGER : i;
}

/** The team band a row belongs to: the resolved name, or "—" for a backlog with no team. */
function teamBandKey(row: TaskListRowDto, teamNames: ReadonlyMap<number, string>): string {
  const id = row.teamId;
  if (id === null || id === undefined) return '—';
  return teamNames.get(id) ?? `#${id}`;
}

/**
 * Group the rows into bands. `byTeam` chooses the axis:
 *   - by TEAM: key on the resolved team name, alpha order, a "—" (no-team) band last.
 *   - by PROJECT: key on the project, ordered ARCS -> PlusArcs -> ARMS -> Other, unknown last.
 *
 * Row order WITHIN a band is preserved — the server already sorts every row by `backlogCode`.
 */
export function groupRows(
  rows: readonly TaskListRowDto[],
  byTeam: boolean,
  teamNames: ReadonlyMap<number, string>,
): Band[] {
  const bands = new Map<string, TaskListRowDto[]>();

  for (const row of rows) {
    const key = byTeam ? teamBandKey(row, teamNames) : (row.project ?? '—');
    const bucket = bands.get(key);
    if (bucket) bucket.push(row);
    else bands.set(key, [row]);
  }

  const entries = [...bands.entries()].map(([key, bandRows]) => ({ key, rows: bandRows }));

  if (byTeam) {
    // "—" (no team) always sorts last; everything else is alpha by team name.
    return entries.sort((a, b) =>
      a.key === '—' ? 1 : b.key === '—' ? -1 : a.key.localeCompare(b.key));
  }
  return entries.sort((a, b) => projectOrder(a.key) - projectOrder(b.key) || a.key.localeCompare(b.key));
}
```

**Replace** `askRemoveDefaultTask` (`:399-423`) with a reversible toggle — same shape as `toggleTeamActive` / `toggleContactActive`, no confirm (it is reversible now):
```typescript
  /**
   * 🔴 REVERSIBLE now — the one-way door is gone. `getDefaultTasksAll()` (admin) lists inactive rows, so a
   * deactivated default task is visible and can be brought back. `setDefaultTaskActive` flips is_active in
   * either direction; the flag is passed through, never hard-coded. No confirm — this is a soft toggle,
   * exactly like the Team / PCA-contact toggles above.
   */
  toggleDefaultTaskActive(d: DefaultTaskDto): void {
    const id = d.id;
    if (this.busy() || typeof id !== 'number') return;   // ← re-entrancy guard
    const next = !(d.isActive === true);
    this.run(
      this.api.setDefaultTaskActive(id, next),
      next ? 'Default task activated' : 'Default task deactivated');
  }
```
