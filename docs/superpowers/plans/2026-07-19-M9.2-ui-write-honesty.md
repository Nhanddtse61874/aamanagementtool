<action>
Replace the exported `parseHours` with:

```ts
export type InvalidReason = 'not-a-number' | 'not-positive' | 'over-cap' | 'too-precise';
export type CellInput =
  | { kind: 'clear' }
  | { kind: 'value';   hours: number }
  | { kind: 'invalid'; reason: InvalidReason };

export function readCell(text: string): CellInput;
```

Rules, in this order: blank/whitespace → `clear`; not a finite number → `not-a-number`;
`<= 0` → `not-positive`; `> 8` → `over-cap`; more than 1 decimal place → `too-precise`; else `value`.

Rewrite the `parseHours` describe block as a `readCell` table covering
`''`, `'  '`, `'4'`, `'4.5'`, `'0'`, `'-1'`, `'9'`, `'2.55'`, `'abc'` — asserting `kind` and `reason`.

🔴 The existing comment at `grid-state.spec.ts:194-195` argues that `'0'` SHOULD be sent to the server.
That rationale is being reversed on purpose (spec §5.3). **Replace the comment** — do not leave prose
that contradicts the code. Note in your report that you reversed a documented decision.

Do NOT touch `log-work.component.ts` — that is B1's file. Leaving the build red at the end of A1 is
expected and correct; B1 closes it.
</action>

<done>readCell exports the three-way result and its spec table passes.</done>

  src/timesheet-web/src/app/pages/settings/settings.component.spec.ts

<action>
In `runBackup()`, branch on the returned value:
- non-empty → success exactly as today, reporting the path.
- null / empty → **failure**: no success toast; surface `Backup did not run — no file was written.`
  through the same channel this component already uses for failures (`this.fail(...)` / `opsResult`,
  whichever matches the existing pattern — follow it, do not invent a second one).

🔴 The message must NOT name a cause. `BackupService.cs:43` returns null for three different
preconditions and the client cannot tell them apart. Saying "no backup folder configured" would replace
one false claim with another — which is the exact defect this milestone exists to remove.

Add specs: `value: null` → failure, no success toast; `value: '<path>'` → success.
</action>
<verify>npx ng test --watch=false --include='**/settings.component.spec.ts'</verify>
<done>A null backup result renders failure; a real path still renders success.</done>

### B1 — `commitCell`, the totals, and the affordance

  spec §5.4, §5.5, §5.6, §5.7
</read_first>
<action>
1. `commitCell` (:291) branches on `readCell(draft).kind`:
   `clear` → `clearCell` · `value` → `saveCell` · `invalid` → keep the draft, record the cell as invalid,
   **make no API call**, show the status line. The `wanted === current` short-circuit at :299 applies
   only to `clear` and `value`.
2. 🔴 `cellHours` (:252) currently does `parseHours(text) ?? 0`, so gibberish drops the day/week totals
   to 0 while the DB still holds the real hours — the same lie, one layer up. An invalid cell must
   contribute its **last committed value** (`this.cells()[key]?.hours ?? 0`), never the parse of the draft.
3. Affordance: conditional class + `aria-invalid` on the cell input; a `role="alert"` region showing
   exactly `Not saved — fix the highlighted cell` (WPF's own string, `TimesheetViewModel.cs:374`),
   following the shape at `backlog-editor.component.html:24-27`. Styling from the `--danger-*` tokens only.
4. Specs — the two that would have caught the original bug:
   - typing gibberish over an existing value issues **no request** (assert the API spy was never called);
   - an invalid cell contributes its last committed value to `dayTotal`, not 0.
</action>
<verify>npx ng test --watch=false --include='**/log-work.component.spec.ts'</verify>
<done>Gibberish over a filled cell writes nothing, shows red + the status line, and leaves the totals honest.</done>
