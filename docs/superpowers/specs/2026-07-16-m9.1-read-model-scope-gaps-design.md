# M9.1 — Close the 3 read-model / scope gaps

**Date:** 2026-07-16
**Status:** Approved (brainstorm)
**Milestone:** M9.1 (follow-up to M9; before M10 = delete WPF)
**Mode:** TBD at Mode Gate

## Context

M9 shipped every screen against the real API; UAT round-1 (2026-07-15) fixed 5 defects. A
2026-07-16 code audit confirmed the three "silent data-loss" UAT items (OT-16/17/20) are already
SAFE in code. What remains genuinely unfinished in **code** are three fidelity/read-model gaps that
the web app inherited by leaving fields off the wire. Each is a WPF-parity gap the desktop already
implements. `RestoreAsync`, the admin-demotion window, and remote multi-user hosting are explicitly
**out of scope** (deliberate non-goals / separate risk).

All three gaps add fields to (or add a route on) the API contract, so the generated TypeScript client
must be regenerated. Because "a regeneration is a global event", they are done as **ONE milestone with
ONE regen** — annotate/change all C# → run API → `npm run gen:api` once → wire each Angular screen.
This is the exact model M9 Phase 1 used.

## Goals (REQ-IDs)

| REQ-ID | Gap | Observable truth (user perspective) |
|---|---|---|
| **TL-12** | Task List group-by-team | With >1 team checked in the filter, Task List rows band by **team**; with one team, they band by **project** (adaptive) — matching WPF. |
| **DR-11** | Daily Report picker active-team scope | The Daily Report backlog picker lists only backlogs of the **active team**, matching WPF (not "all your teams"). |
| **SET-05** | Default-tasks round-trip | An admin can **deactivate and reactivate** a default task in Settings — deactivation is no longer a one-way "remove". |

## Design decisions (with rationale)

1. **One milestone, one regen.** Rejected "3 independent fast-lane fixes" because each would trigger
   its own client regeneration (a global event) → needless churn and 3 chances to break the generated
   tree. Batch the C# contract changes, regen once, then wire the three screens in parallel.

2. **Team NAME is resolved client-side, never joined server-side.** The server sends only `teamId`.
   The WPF app resolves the team name from `AvailableTeams` (`TaskListViewModel.cs:206-217`); the web
   `team-filter.component.ts:177` already `forkJoin`s `/api/me` + `/api/teams` and holds the same
   id→name map. Rejected "JOIN Teams for the name in `SearchAsync`" because it adds a join that does
   not exist today, for a value the client already has.

3. **B2 hard-filters to the active team** (not "default active, allow widening"). WPF scopes the
   picker to the active team, full stop. Simpler and faithful. Not a security change — every entry is
   already server-authorized and stamped with the active team regardless.

4. **B3 is presented as a reversible toggle.** The DTO already carries `isActive`
   (`Dtos.cs:186`), the generated TS already has `isActive?`, and `GetAllAsync` is a one-line mirror of
   `GetActiveAsync`. The only real new surface is one admin-gated route + its contract-test rows.

## Technical detail

### TL-12 — Task List group-by-team

**Server (projection-only — no join, no N+1; `b.TeamId` is already loaded by `SearchAsync`):**
- `TimesheetApp.Core/Models/ReadModels.cs` — add `int? TeamId` to `TaskListRow` (`:127-133`).
- `TimesheetApp.Api/Contracts/Dtos.cs` — add `int? TeamId` to `TaskListRowDto` (`:145-151`) and its
  `ToDto(this TaskListRow r)` mapper (`:248-254`).
- `TimesheetApp.Core/Services/TaskListService.cs` — thread `b.TeamId` into the `TaskListRow`
  construction in `GetRowsAsync` (`:117-121`).

**Client (`src/timesheet-web/src/app/pages/task-list/`):**
- `task-list.model.ts` — make `groupRows` adaptive: accept the filter mode (`showTeamColumn`) + the
  id→name map; when banding by team, key on `teamId`→name with alpha order; else band by project with
  `ProjectOrder`. Remove the "DELIBERATELY ABSENT" comment (`:90-102`).
- `task-list.component.ts` — feed the `bands` computed (`:126`) with the team mode + id→name map. The
  component already embeds `<app-team-filter>` and tracks `teamIds`/`noTeams`; source the id→name map
  from teams (via the filter or a `getTeamsActive()` call, mirroring WPF's `AvailableTeams`).

**Contract tests:** none — this is a DTO-field-only change; route/tag/status/params unchanged.

### DR-11 — Daily Report picker active-team scope

**Server (projection-only):**
- `TimesheetApp.Api/Contracts/Dtos.cs` — add `int? TeamId` to `BacklogListItemDto` (`:83-85`) and its

**Client (`src/timesheet-web/src/app/pages/daily-report/`):**

**Contract tests:** none — DTO-field-only.

## Waves (goal-backward; same-wave = zero file overlap)

```
Wave A — C# contract (SEQUENTIAL: A1/A2 both edit Dtos.cs → file overlap):
  A1  TL-12 server    (teamId → TaskListRow / TaskListRowDto / projection)
  A2  DR-11 server    (teamId → BacklogListItemDto)
  A3  SET-05 server   (GetAllAsync + GET /api/default-tasks/all + contract tests)

REGEN (serial) — API up on safe config → npm run gen:api ONCE

Wave B — Angular wiring (PARALLEL: 3 different page dirs, zero overlap):
  B1  task-list  (adaptive team/project band)
  B2  daily-report (active-team picker filter)
  B3  settings   (default-task toggle)
```

## Verification / safety

- **Pre-execute baseline:** run a clean `dotnet build` + full test suite with `ConfigPath` pointed at a
  fresh `mktemp -d` (which does NOT exist) and `DbPath`/`KeyRingPath` pinned into that same dir — the
  true invariant that avoids opening the live company DB. Confirm the "no users yet, nothing to
  bootstrap" proof line and both `Passed!` lines (Core/WPF + API). Baseline the untouched tree first
  (the ~15% ApiTests startup race is a known false-RED — re-run a lone `no such table` failure).
- **Regen** requires the API listening on :5080 — use the same safe config; nothing else may hold
  `TimesheetApp.Core.dll` open (close any running app) or the API build silently no-ops the gate.
- **Per-REQ verify:** TL-12 — a filter with 2 teams bands by team, 1 team bands by project; DR-11 — the
  picker shows only active-team backlogs; SET-05 — deactivate then reactivate a default task round-trips
  and Log Work reflects it.

## Out of scope

Admin demotion window (token-version claim), backup-list route / `RestoreAsync` exposure, remote
multi-user hosting. Team NAME is not added to the wire (resolved client-side by decision #2).

## Testing strategy

- Server: unit/integration tests for the new projection fields (teamId present on `TaskListRowDto` /
  `BacklogListItemDto`) and `GetAllAsync` (returns inactive rows); the 3 contract-test rows + 403 test
  for `/api/default-tasks/all`.
- Client: spec tests for adaptive `groupRows` (team vs project keying), the DR active-team filter, and
  the Settings toggle (reactivation path).
- Regression: full suite green after regen; 0 warnings.
