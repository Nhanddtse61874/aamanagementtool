# M9.2 — UI write honesty

**Date:** 2026-07-19
**Status:** Approved (STEP 2 brainstorm, user-approved 2026-07-19)
**Origin:** `.planning/M10-COVERAGE-AUDIT.md` — two `PARTIAL` findings the refuters rated more dangerous than most `MISSING` items. Both confirmed at the source by the controller before this spec was written.
**Fast Lane assessment:** `.planning/fast-lane-m9.2-ui-write-honesty.json` — combined milestone runs the full workflow.

---

## 1. The defect class

Both bugs are one thing: **the UI asserts a write that did not happen.**

- **BUG-1** asserts a *delete the user never asked for*. Typing unparseable text into a Log Work cell destroys the hours already stored there.
- **BUG-2** asserts a *backup that was never made*. "Backup now" toasts success when nothing was written.

BUG-1 is the more urgent of the two: it destroys real user data during ordinary daily use, and it does so silently. BUG-2 lies about a safety net — serious, but it does not itself lose hours.

## 2. Neither is new work — both violate existing v1 requirements

| Bug | Requirement violated | Requirement text |
|---|---|---|
| BUG-1 | **TS-04**, **XC-02**, **XC-04** | *"red border on `>8`, negative, or `>1`-decimal entry"* · *"shows a red error and the value is not committed"* · hours must be `>0` with at most 1 decimal |
| BUG-2 | **BK-02** | *"reports success/failure with the resulting path; **a no-op with a message if the DB file is missing or no folder is set**"* |

**Consequence: M9.2 introduces NO new REQ-IDs and `REQUIREMENTS.md` is not edited.** This milestone closes debt the web app carried from the start — the requirements were written for the WPF app, the WPF app satisfies them, and the web port never did.

## 3. Evidence

### BUG-1 — unparseable text silently deletes stored hours

```
grid-state.ts:197-201          parseHours('abc') -> null
log-work.component.ts:296-301  wanted === null   -> clearCell()   // a DELETE
```

`parseHours` maps **two different intents** onto one output:

| input | returns | means | correct? |
|---|---|---|---|
| `''`, `'   '` | `null` | clear the cell | ✅ |
| `'abc'` | `null` | clear the cell | ❌ should be *invalid* |

A green test pins the parse half — `grid-state.spec.ts:190` *"reads gibberish as null rather than sending it"*. The test is correct about what it asserts. **Nobody traced the `null` two files onward to the delete.** This is the failure mode already named in `STATE.md` as *"a gate that cannot move is a gate that cannot notice"*.

### BUG-2 — "Backup now" reports success when nothing was written

```
BackupService.cs:36            BackupNowAsync() -> BackupToFolderAsync(_config.BackupFolderPath, ...)
BackupService.cs:43            null when folder blank | dbPath blank | db file missing
JsonAppConfigTests.cs:54       asserts the DEFAULT BackupFolderPath is ""
SettingsEndpoints.cs:1051-1054 wraps the string? in SettingsOpsResult, unexamined
settings.component.ts:550-551  renders `r.value ?? 'the configured folder'`, toasts "Backup complete"
```

Note the counterexample in the same class: `BackupService.cs:62` — `AutoBackupIfDueAsync` **does** guard on a blank folder. Only the manual path is unguarded.

`null` therefore means **"a precondition failed, nothing was attempted."** A failure *during* the copy (disk full, permission denied, share offline) throws out of `Task.Run`, surfaces as a 500, and is already handled by the component's existing `error:` branch.

## 4. The oracle — what WPF does, while it still exists

M10 deletes the WPF app. This section is written down now because after M10 there is nothing left to compare against.

```csharp
// TimesheetViewModel.cs:367-376
/// Persist a single cell as soon as it commits. A red (invalid) cell is left on screen but never
/// written; the service also rejects a cell that would push the day over 8h (revert on reload).
if (row.HasErrorFor(col))
{
    SetStatus("Not saved — fix the highlighted cell", isError: true);
    return;
}
```

```csharp
// TimesheetRowVm.cs:42-44
/// True when the given day cell currently holds a validation error (e.g. >8h, ≤0, >1 decimal) —
/// the owner VM checks this before auto-persisting so a red cell is never written to storage.
```

**The WPF contract:** an invalid cell is *left on screen, marked red, and never written* — neither saved nor deleted. The web app does the opposite: it deletes.

WPF validates **four** states per cell. The web validates **none** — `log-work.component.html` carries only a page-level `loadError`, no per-cell affordance whatsoever.

## 5. Design — Part 1: Log Work cell input

### 5.1 Three-way result (replaces `parseHours`)

```ts
export type CellInput =
  | { kind: 'clear' }
  | { kind: 'value';   hours: number }
  | { kind: 'invalid'; reason: InvalidReason };

export type InvalidReason = 'not-a-number' | 'not-positive' | 'over-cap' | 'too-precise';

export function readCell(text: string): CellInput;
```

**Rejected alternative:** keep `parseHours` and add separate `isBlank()` / `isValid()` helpers. Less churn, but it spreads a three-way truth across three functions and leaves callers to recombine them correctly — which is precisely how this bug was created. One function, one total answer.

### 5.2 Rules, and where each comes from

| Input | `kind` | `reason` | Source |
|---|---|---|---|
| `''`, whitespace | `clear` | — | existing behavior, correct |
| not a finite number | `invalid` | `not-a-number` | TS-04 *"invalid … entry"* |
| `<= 0` (incl. `'0'`, `'-1'`) | `invalid` | `not-positive` | XC-04; WPF `TimesheetRowVm.Validate` |
| `> 8` | `invalid` | `over-cap` | XC-02 per-cell 8h cap |
| more than 1 decimal | `invalid` | `too-precise` | XC-04 |
| otherwise | `value` | — | — |

### 5.3 ⚠️ A recorded decision is being reversed

`grid-state.spec.ts:194-198` currently asserts that `'0'` is **sent to the server** so the API can answer, with a comment arguing that is more honest than guessing the user meant to delete.

M9.2 reverses that: `'0'` is rejected **client-side** with the same meaning, matching WPF (`TimesheetRowVm.Validate` rejects `<= 0`). The reversal is deliberate and is recorded here so it is not later mistaken for an accident. The user-visible result is strictly better — the message appears immediately instead of after a round-trip — but the previously written rationale no longer holds and that comment must be replaced, not left contradicting the code.

### 5.4 `commitCell` branching

```
clear   -> clearCell(taskId, iso)                    (unchanged)
value   -> saveCell(taskId, iso, hours)              (unchanged)
invalid -> keep the draft, mark the cell invalid, MAKE NO REQUEST,
           show "Not saved — fix the highlighted cell"
```

The `wanted === current` short-circuit at `log-work.component.ts:299` must be evaluated **only** for `clear` and `value`. An `invalid` input is never "equal to current".

### 5.5 🔴 The totals must not lie either

`log-work.component.ts:252` currently reads:

```ts
private cellHours(taskId: number, iso: string): number {
  return parseHours(this.cellText(taskId, iso)) ?? 0;
}
```

So typing gibberish drops the row/day/week totals to **0 on screen while the database still holds 4**. That is the same defect class as BUG-1 and is fixed in the same change: an `invalid` cell contributes its **last committed value**, not `0`.

This feeds `rowTotal`, `dayTotal`, `weekTotal` (`:255-269`). Those are display-only here — the `>8h` day-total gate (XC-03/TS-06) is **out of scope** (see §8).

### 5.6 Affordance

- red border on the offending input, driven by component state keyed by `cellKey(taskId, iso)`
- the reason exposed to assistive tech (`aria-invalid`, and the message as the input's accessible description)
- a status line reading exactly **`Not saved — fix the highlighted cell`** — WPF's own string (`TimesheetViewModel.cs:374`)
- touches `log-work.component.html` and `log-work.component.scss`

## 6. Design — Part 2: Backup honesty

**Client-only.** In `settings.component.ts` `runBackup()`:

```
value non-empty -> success, report the path (unchanged)
value null/empty -> FAILURE state, message: "Backup did not run — no file was written."
```

The message **must not** name a cause. `null` collapses three preconditions (§3) and the client cannot distinguish them; asserting "no backup folder configured" would replace one false claim with another.

**Rejected alternative:** have the server return a reason. That changes `IBackupService` in `TimesheetApp.Core` — still shared with the WPF host until M10 — plus the API contract plus a client regen. Out of proportion to one sentence of UI copy, and M11 is about to move this configuration anyway.

**Out of scope, stated so it does not ride along:** the web Settings screen has no backup-folder picker at all. That is **BK-01**, already on the audit's MISSING list. Adding it, or disabling the button until a folder is set, is a separate change touching the template and arguably a small feature.

## 10. Verification

Both defects are invisible to unit tests in their current form — that is how they shipped. UAT must include:

- **G-A:** type `abc` over a cell holding `4`, tab away. The `4` is still there, the cell is red, the status line shows. Reload — still `4`.
- **G-B:** press "Backup now" on a server with no backup folder configured. The UI reports failure, not success.

Per the standing user decision of 2026-07-19, these join the batched UAT session after M10 and M11 rather than gating the merge.
